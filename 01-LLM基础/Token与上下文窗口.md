# Token 与上下文窗口

## 学习目标

这篇文档帮助你理解一个非常工程化、但又非常基础的问题：

**模型不是按“字数”理解输入，而是按 token 和上下文窗口工作。**

学完后你应该能够：

1. 解释 token 是什么，以及它为什么不是“一个字”或“一个单词”
2. 理解上下文窗口会限制什么
3. 判断长输入为什么会影响成本、效果和截断风险
4. 在实际写 Prompt 或做应用时具备长度意识

## 1. 什么是 Token

Token 可以理解成：

**模型内部处理文本时使用的基本切分单位。**

它不是严格等于：

- 一个汉字
- 一个英文单词
- 一个标点

实际切分取决于 tokenizer 规则。

例如一段文本在不同模型里，token 数可能不同。也就是说：

**同样一句话，不同模型看到的“颗粒度”可能不一样。**

---

## 2. 为什么 Token 很重要

因为模型很多关键限制和成本，都是按 token 计算的。

包括：

- 输入长度限制
- 输出长度限制
- API 计费
- 推理延迟
- 上下文截断风险

所以在工程里，真正要关心的不是“这段话 500 字”，而是：

**这段话会占多少 token。**

---

## 3. 什么是上下文窗口

上下文窗口可以理解成：

**模型在当前一次推理中，最多能同时看到多少 token。**

这个“看到”包括：

- 系统提示词
- 用户输入
- 历史对话
- 检索到的外部资料
- 工具返回结果
- 模型已经生成的部分内容

这点非常关键，因为很多人只算用户问题，却忘了其他部分也会占窗口。

---

## 4. 上下文窗口会影响什么

### 4.1 影响能塞进去多少信息

窗口小，很多背景资料就放不进去。

### 4.2 影响成本

上下文越长，通常推理成本越高。

### 4.3 影响速度

处理 token 越多，通常响应越慢。

### 4.4 影响效果

如果窗口快满了，系统可能：

- 截断前文
- 压缩历史
- 丢掉一部分检索结果

这样就可能导致回答质量下降。

---

## 5. 为什么长上下文不等于一定更好

很多人会觉得：

“窗口越大，能塞的资料越多，效果当然越好。”

这只说对了一半。

### 5.1 信息多，不等于相关信息突出

如果给模型塞了大量无关内容，真正重要的信息反而会被淹没。

### 5.2 长上下文会增加噪声

当背景材料太多时，模型可能抓不住重点，甚至把边缘信息当成核心信息。

### 5.3 成本和延迟会上升

长上下文通常意味着更高成本和更慢响应，这在真实产品里很敏感。

所以更好的思路通常不是“无脑塞满”，而是：

**给足够相关的信息，而不是给尽可能多的信息。**

---

## 6. 实践中的常见问题

### 6.1 对话越聊越长，模型开始遗忘重点

这是典型的上下文管理问题。历史越长，越可能：

- 关键信息被稀释
- 早期约束被忘记
- 新旧指令互相冲突

### 6.2 RAG 塞入太多检索片段

很多初学者觉得召回越多越保险，但实际可能造成：

- 上下文拥挤
- 资料互相矛盾
- 回答引用不聚焦

### 6.3 输出被截断

如果输入已经占掉大量窗口，模型可用于输出的空间就会变少，回答可能中途停掉。

---

## 7. 实战建议

### 7.1 先建立 token 意识

写 Prompt、设计 RAG、做多轮对话时，都要意识到每一段文本都在抢上下文预算。

### 7.2 给模型“精选资料”，不要给“所有资料”

尤其在 RAG 里，要优先追求相关性，而不是堆数量。

### 7.3 控制历史对话

可以考虑：

- 定期摘要历史
- 保留关键状态
- 删除无关闲聊

### 7.4 为输出预留空间

不要把输入塞到接近窗口极限，否则模型就没有足够空间完整回答。

---

## 8. 练习题

### 8.1 判断题

“上下文窗口是只计算用户这次提问长度的。”

参考答案：

错误。系统提示词、历史消息、检索资料、工具结果和已生成内容都会占用窗口。

### 8.2 思考题

为什么说长上下文能力很重要，但不能把它理解成“什么都往里塞”？

参考思路：

因为上下文质量比上下文数量更关键，过多无关信息会提高噪声、成本和干扰。

---

## 9. 总结与下一步

这篇文档最值得记住的一句话是：

**上下文窗口不是一个抽象参数，它直接决定模型这次能看到什么、忘掉什么、花多少钱、回答能不能完整。**
