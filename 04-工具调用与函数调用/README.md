# 04 工具调用与函数调用：让模型真正“能做事”

工具调用（Tool Calling）与函数调用（Function Calling）是从“会聊天”走向“会执行任务”的关键能力。  
这一章面向初学者，从 0 带你建立完整认知：什么时候调用工具、如何设计参数、怎么处理失败、如何把结果回注给模型。

## 适用人群

- 从 0 学习 LLM/Agent 的同学
- 已经会写 Prompt，但应用还停留在“纯文本回答”
- 想把模型接入天气、搜索、数据库、业务 API 等外部能力的开发者

## 学习目标

完成本章后，你应该能够：

1. 解释“模型直接回答”和“调用工具后回答”的区别。
2. 设计结构化函数参数（可校验、可扩展、可维护）。
3. 处理工具失败场景（超时、空结果、格式异常、脏数据）。
4. 让模型基于工具结果做二次推理并输出最终答案。
5. 在多工具场景下做出合理的工具选择策略。

## 为什么这一章很关键

如果没有工具调用，模型只能基于训练数据和上下文“猜”。  
有了工具调用，模型可以：

- 查实时数据（天气、汇率、新闻、库存）
- 读外部系统（数据库、日志、业务服务）
- 执行动作（发消息、下工单、创建任务）

简单说：**工具调用决定了你的 Agent 是否真的具备“行动能力”。**

## 最小实践项目

你可以做一个“城市天气问答助手”作为本章实战：

- 输入：`明天上海会下雨吗？要带伞吗？`
- 过程：
  1) 模型判断需要天气工具；  
  2) 生成结构化参数（城市、日期）；  
  3) 调用天气 API；  
  4) 回注天气结果；  
  5) 生成可读建议。
- 输出：既有数据依据，也有自然语言解释。

## 常见误区

- 误区 1：把工具调用当“黑箱自动能力”，不做参数校验。  
  结果：线上充满脏参数和不稳定行为。

- 误区 2：工具返回后直接展示原始 JSON。  
  结果：用户看不懂，且容易暴露内部字段。

- 误区 3：失败就报错，不做重试和降级。  
  结果：用户体验差，系统不可用时间变长。

- 误区 4：见工具就调。  
  结果：成本增加、延迟变高，且并不总能提升质量。
