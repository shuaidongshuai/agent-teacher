## 1. 适用人群

- 已经学过函数调用、Tool Calling 基础的同学
- 想自己写一个可以被 VS Code 直接使用的 MCP 服务的人
- 想分清“教学 Demo”和“真实可接入 MCP 服务”差别的开发者

## 2. 学习目标

学完这篇文章后，你应该能：

1. 理解 MCP 服务为什么更像“按协议运行的插件进程”。
2. 使用官方 Python MCP SDK 写出一个最小可运行服务。
3. 实现一个今日天气查询工具，并理解参数、返回值、错误处理怎么设计。
4. 配置 VS Code 的 `.vscode/mcp.json`，把本地 MCP 服务接进去。
5. 知道如何排查“VS Code 发现不了工具”或“服务启动失败”这类常见问题。

## 3. 这篇文章解决什么问题

很多资料在讲 MCP 时，容易停留在概念层：

- 什么是 Tool
- 什么是 Resource
- 什么是 Prompt

但学生真正卡住的，通常不是概念，而是下面两个更具体的问题：

1. 我怎么用正式 SDK 自己写一个 MCP 服务？
2. 写完以后，VS Code 到底怎么接入和调用？

这篇文章就是围绕这两个问题展开，而且会刻意避开“手写一堆 JSON-RPC 消息”的教学方式，直接采用官方 SDK 的主流写法。

## 4. 先纠正一个常见误解：MCP 插件本质上是什么

很多同学会说“我要写一个 MCP 插件”。

这个说法不算错，但更准确的理解是：

**你写的是一个 MCP 服务端进程，VS Code 这类宿主把它当成插件能力来使用。**

也就是说：

- 你负责写一个符合 MCP 协议的服务
- VS Code 负责启动它、发现它的工具、在聊天过程中调用它

所以从工程角度看，真正要写的核心不是“VS Code 扩展”，而是：

**一个能被 VS Code 作为 MCP Server 启动和调用的服务。**

## 5. 为什么这次要用正式 MCP SDK，而不是手写协议

前一版文章里用“手写最小 JSON-RPC 服务”的方式，适合理解协议本身，但不适合让学生以为那就是生产写法。

这次改成官方 SDK，有三个原因。

### 5.1 这是更接近真实开发的路径

官方 Python SDK 已经帮你处理了很多底层细节，比如：

- 工具注册
- 参数模式生成
- 标准 MCP 生命周期
- `stdio` 传输层运行

你不需要自己再去拼装所有消息路由。

### 5.2 更容易被真实宿主接入

VS Code 这类宿主关心的是：

- 你的服务能不能按标准方式启动
- 能不能暴露出它能发现的工具

用正式 SDK，比手写一个“只跟自己 Demo Client 对得上”的版本可靠得多。

### 5.3 更适合教学迁移

学生学完后，可以直接把天气工具换成：

- 数据库查询
- 内部知识库搜索
- Jira 创建工单
- 企业 API 调用

也就是说，真正可迁移的是“SDK 写法 + 宿主接入方式”，而不只是某个特定 Demo。

## 6. 我们要做的样例：今日天气查询

这篇文章实现一个最小但真实的工具：

- 工具名：`get_today_weather`
- 输入：城市名
- 输出：今天的天气摘要，包括温度、体感、风速和天气描述

为了让示例更聚焦 MCP 本身，我们做两个取舍：

1. 只支持少量内置城市坐标
2. 通过 Open-Meteo API 获取天气，不引入额外鉴权

这样做的好处是：

- 代码足够短，学生能完整看懂
- 仍然是真实外部 API 调用
- 可以直接被 VS Code 调起和使用

## 7. 项目结构与环境准备

官方 MCP 文档当前推荐 Python 3.10+，并建议使用 `uv` 管理项目；官方天气服务示例也使用了 `FastMCP` 和 `httpx`。VS Code 当前支持通过 `.vscode/mcp.json` 配置本地 MCP 服务。

建议你新建一个最小目录：

```text
mcp-weather-vscode/
├── pyproject.toml
├── weather_server.py
└── .vscode/
    └── mcp.json
```

### 7.1 初始化项目

```bash
uv init mcp-weather-vscode
cd mcp-weather-vscode
uv venv
source .venv/bin/activate
uv add "mcp[cli]" httpx
```

如果你使用 `pip` 也可以，但本篇按官方文档推荐的 `uv` 路线来讲。

## 8. 用官方 Python MCP SDK 实现服务端

官方 Python SDK 的核心写法很简单：

1. `from mcp.server.fastmcp import FastMCP`
2. 创建一个 `FastMCP(...)` 实例
3. 用 `@mcp.tool()` 声明工具
4. 最后 `mcp.run(transport="stdio")`

对 VS Code 这种本地宿主来说，`stdio` 是最适合入门的方式：

- 宿主直接拉起你的 Python 进程
- 通过标准输入输出和服务通信
- 不需要你自己额外开 HTTP 端口

## 9. 完整代码讲解

下面是一份可以直接作为教学样例的服务端代码。

```python
from __future__ import annotations

import logging
from typing import Any

import httpx
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("today-weather")

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("today-weather")

CITY_COORDS: dict[str, dict[str, float]] = {
    "beijing": {"lat": 39.9042, "lon": 116.4074, "label": "北京"},
    "shanghai": {"lat": 31.2304, "lon": 121.4737, "label": "上海"},
    "hangzhou": {"lat": 30.2741, "lon": 120.1551, "label": "杭州"},
    "shenzhen": {"lat": 22.5431, "lon": 114.0579, "label": "深圳"},
    "guangzhou": {"lat": 23.1291, "lon": 113.2644, "label": "广州"},
    "chengdu": {"lat": 30.5728, "lon": 104.0668, "label": "成都"},
}

WEATHER_CODE_MAP = {
    0: "晴",
    1: "大部晴朗",
    2: "局部多云",
    3: "阴",
    45: "有雾",
    48: "冻雾",
    51: "毛毛雨",
    53: "中等毛毛雨",
    55: "强毛毛雨",
    61: "小雨",
    63: "中雨",
    65: "大雨",
    71: "小雪",
    73: "中雪",
    75: "大雪",
    80: "阵雨",
    81: "较强阵雨",
    82: "强阵雨",
    95: "雷暴",
}


def normalize_city(city: str) -> str:
    return city.strip().lower().replace(" ", "")


async def fetch_today_weather(lat: float, lon: float) -> dict[str, Any]:
    url = "https://api.open-meteo.com/v1/forecast"
    params = {
        "latitude": lat,
        "longitude": lon,
        "current": "temperature_2m,apparent_temperature,weather_code,wind_speed_10m",
        "timezone": "Asia/Shanghai",
        "forecast_days": 1,
    }

    async with httpx.AsyncClient(timeout=10.0) as client:
        response = await client.get(url, params=params)
        response.raise_for_status()
        return response.json()


@mcp.tool()
async def get_today_weather(city: str) -> dict[str, Any]:
    """
    查询指定城市今天的天气。

    参数:
        city: 城市英文名，例如 beijing、shanghai、hangzhou
    """
    key = normalize_city(city)
    if key not in CITY_COORDS:
        supported = ", ".join(sorted(CITY_COORDS.keys()))
        return {
            "ok": False,
            "message": f"暂不支持城市: {city}。当前支持: {supported}",
        }

    city_info = CITY_COORDS[key]
    weather = await fetch_today_weather(city_info["lat"], city_info["lon"])
    current = weather.get("current", {})
    weather_code = current.get("weather_code")
    weather_desc = WEATHER_CODE_MAP.get(weather_code, f"未知天气代码 {weather_code}")

    result = {
        "ok": True,
        "city": city_info["label"],
        "query_city": city,
        "time": current.get("time"),
        "temperature_c": current.get("temperature_2m"),
        "apparent_temperature_c": current.get("apparent_temperature"),
        "wind_speed_kmh": current.get("wind_speed_10m"),
        "weather_code": weather_code,
        "weather_desc": weather_desc,
        "summary": (
            f"{city_info['label']}今天{weather_desc}，"
            f"当前温度 {current.get('temperature_2m')}°C，"
            f"体感 {current.get('apparent_temperature')}°C，"
            f"风速 {current.get('wind_speed_10m')} km/h。"
        ),
    }
    logger.info("weather result: %s", result)
    return result


def main() -> None:
    mcp.run(transport="stdio")


if __name__ == "__main__":
    main()
```

### 9.1 这段代码的关键点

#### 第一，`FastMCP("today-weather")`

这一行是在创建服务实例。

你可以把它理解成：

- “我要启动一个名字叫 `today-weather` 的 MCP Server”

#### 第二，`@mcp.tool()`

这就是把一个 Python 函数暴露成 MCP Tool。

这里很关键的一点是：

- 函数签名决定参数结构
- 函数 docstring 会帮助宿主理解工具用途

也就是说，SDK 会帮你自动生成一部分工具描述，而不是让你自己手写一大段 schema。

#### 第三，返回值要尽量结构化

不要只返回一句自然语言。

更好的做法是像上面一样同时返回：

- `temperature_c`
- `apparent_temperature_c`
- `weather_desc`
- `summary`

这样模型既能读自然语言摘要，也能拿结构化字段继续推理。

#### 第四，日志写到 stderr，不要写 stdout

这点非常重要。

对于 `stdio` MCP 服务，`stdout` 是协议通道。如果你在服务里乱 `print()`，就会污染 JSON-RPC 消息，导致 VS Code 无法正常识别服务。

所以：

- 正常业务输出交给 MCP SDK
- 调试日志用 `logging` 或写到 `stderr`

## 10. 把这个 MCP 服务接入 VS Code

VS Code 官方文档当前支持通过 `.vscode/mcp.json` 或用户级 `mcp.json` 配置 MCP Server。对教学场景来说，最推荐工作区级配置，因为学生打开项目就能看到完整接法。

在项目根目录创建：

```text
.vscode/mcp.json
```

写入：

```json
{
  "servers": {
    "today-weather": {
      "type": "stdio",
      "command": "uv",
      "args": [
        "run",
        "weather_server.py"
      ]
    }
  }
}
```

如果你没有用 `uv`，也可以改成：

```json
{
  "servers": {
    "today-weather": {
      "type": "stdio",
      "command": ".venv/bin/python",
      "args": [
        "weather_server.py"
      ]
    }
  }
}
```

在 Windows 上，则需要改成对应的 Python 路径，比如：

```json
{
  "servers": {
    "today-weather": {
      "type": "stdio",
      "command": ".venv\\Scripts\\python.exe",
      "args": [
        "weather_server.py"
      ]
    }
  }
}
```

### 10.1 为什么这里推荐工作区级 `.vscode/mcp.json`

因为它有两个教学优势：

1. 配置和项目放在一起，学生更容易理解
2. 提交到仓库后，团队成员打开项目就能复用

### 10.2 VS Code 接入后的大致流程

当你把配置写进 `mcp.json` 后，VS Code 会：

1. 按配置启动你的服务进程
2. 发现这个服务暴露的工具
3. 在聊天或 Agent 场景里按需调用这些工具

所以你真正需要保证的是：

- 命令能启动
- 服务不往 stdout 打乱七八糟的日志
- 工具描述和参数足够清晰

## 11. 在 VS Code 里实际怎么用

配置完成后，通常按下面顺序验证。

### 11.1 先确认服务有没有被发现

你可以在 VS Code 里：

- 打开命令面板
- 运行 `MCP: List Servers`

如果配置无误，你应该能看到 `today-weather`。

### 11.2 再确认工具有没有暴露出来

成功启动后，VS Code 会发现这个服务提供的工具。

此时你就可以在聊天里尝试类似提示：

```text
请使用 today-weather 里的工具，查询今天上海的天气，并给我一句穿衣建议。
```

如果模型决定使用工具，调用链路大致会是：

1. 模型判断需要天气信息
2. VS Code 通过 MCP 调用 `get_today_weather`
3. 服务返回结构化天气结果
4. 模型基于结果生成自然语言答复

### 11.3 一个很重要的教学观察

学生看到这里时，要明确一件事：

**模型本身并不会“执行 Python 函数”，真正执行的是 VS Code 作为 MCP Host 去调用你的服务。**

这正是 MCP 的工程价值所在。

## 12. 常见报错与排查方法

### 12.1 VS Code 看不到你的服务

优先检查：

- `.vscode/mcp.json` 路径对不对
- JSON 格式是否有效
- `command` 和 `args` 是否能在当前项目下实际运行

### 12.2 服务启动了，但工具没有出现

优先检查：

- 是否真的用了 `@mcp.tool()`
- 文件启动后有没有在 `main()` 中执行 `mcp.run(transport="stdio")`
- 是否有异常直接导致进程退出

### 12.3 服务一启动就报协议错误

最常见原因是：

- 你在 `stdout` 打了调试信息

例如下面这种写法就很危险：

```python
print("server started")
```

因为 `stdio` 模式下，`stdout` 是协议消息通道。

### 12.4 天气接口请求失败

优先检查：

- 网络是否可用
- API 是否返回 200
- 你的工具超时时间是否太短

教学上可以顺手提醒学生：

- MCP 能帮你接入工具
- 但它不会替你解决外部 API 的稳定性问题

## 13. 常见误区

### 13.1 误区一：把 MCP 服务理解成 VS Code 扩展开发

不是一回事。

这里的关键是写一个 MCP Server，让 VS Code 作为宿主来接入它。

### 13.2 误区二：以为“手写能跑”就等于“外部可接”

如果只和你自己的 Demo Client 能通，不代表 VS Code 一定能正常使用。

### 13.3 误区三：工具只返回一句自然语言

这样不利于模型继续利用结果。

更好的方式是：

- 结构化字段 + 可读摘要

### 13.4 误区四：所有问题都怪 MCP

如果天气 API 超时、城市映射没配、Python 环境没装好，这些都不是 MCP 协议本身的问题。

## 14. 练习题

### 练习 1

把 `get_today_weather(city: str)` 改成支持中文城市名，比如：

- 北京
- 上海
- 杭州

参考思路：

- 增加一层别名映射
- 在进入坐标表前先做标准化

### 练习 2

让工具除了天气摘要，还返回一个“穿衣建议”字段。

参考思路：

- 温度低于 10°C，建议加外套
- 温度高于 28°C，建议轻薄衣物
- 有雨时建议带伞

### 练习 3

把这个天气工具扩展成两个工具：

1. `get_today_weather`
2. `get_weather_for_city_list`

参考思路：

- 第二个工具接受城市列表
- 返回多城市天气结果
- 让学生体会单工具和批量工具的参数设计差别

## 15. 总结与下一步建议

如果你希望学生真正学会“自己写一个能被 VS Code 使用的 MCP 插件”，最关键的不是背概念，而是跑通下面这条链路：

1. 用官方 MCP SDK 定义服务
2. 用 `@mcp.tool()` 暴露工具
3. 用 `stdio` 跑起来
4. 在 VS Code 的 `mcp.json` 里配置这个服务
5. 在聊天中让宿主真正调用它

这篇文章最重要的一句话可以记成：

**MCP 插件开发的本质，是写一个标准 MCP 服务，让宿主应用替你完成“发现工具”和“调用工具”。**

下一步建议：

1. 先把天气工具亲手跑通
2. 再把“内置城市坐标”改成“先调用地理编码 API，再查天气”
3. 然后补一个 `Resource`，比如城市天气说明文档
4. 最后再扩展到企业内部 API 或知识库检索
