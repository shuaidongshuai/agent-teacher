## 学习目标

- 识别函数调用链路中的主要失败点
- 设计合理重试策略（次数、间隔、条件）
- 建立降级与监控思维

## 为什么这部分不可省略

教学 Demo 常常只关注“成功路径”，但真实系统里失败是常态。  
如果没有错误处理，再聪明的 Agent 也会频繁中断。

## 常见失败类型

1. **参数错误**：字段缺失、类型不对、值越界。  
2. **网络问题**：超时、连接失败、DNS 异常。  
3. **服务端错误**：5xx、限流、临时不可用。  
4. **鉴权失败**：token 过期、权限不足。  
5. **脏数据**：返回结构变化、字段为空、语义冲突。

## 重试策略设计

重试不是“无脑多试几次”，要有规则：

- 仅对可恢复错误重试（如超时、临时 5xx）
- 使用指数退避（例如 1s、2s、4s）
- 设置最大重试次数（避免雪崩）
- 增加随机抖动（降低并发冲突）

不建议重试的场景：

- 参数校验失败（应先纠错参数）
- 401/403 鉴权错误（应先修复认证）

## 降级与兜底策略

当重试失败，系统仍要“可用”：

- 返回缓存数据（并标注更新时间）
- 切换备选工具（主 API 挂了用备用）
- 提供保守回答（说明当前不确定性）
- 引导用户稍后重试（避免假装确定）

## 可观测性建议

建议每次调用记录：

- `trace_id`（贯穿一次请求）
- 工具名、参数、耗时、重试次数
- 失败类型、错误码、最终状态

这些日志能帮助你快速定位是模型问题、参数问题还是服务问题。

## 练习题

1. 为天气工具设计一份错误处理策略表（错误类型 -> 处理动作）。  
2. 写出“最多重试 3 次 + 指数退避”的伪代码。  
3. 设计一个降级提示文案，避免让用户误以为结果完全准确。

参考方向：

- 文案包含：失败原因、已采取动作、下一步建议

## 总结

稳定性不是锦上添花，而是工具调用系统能否上线的前提。  
把错误处理做扎实，你的 Agent 才真正可用。
