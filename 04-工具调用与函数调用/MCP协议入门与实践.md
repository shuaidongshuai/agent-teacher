## 1. 适用人群

- 已经学过函数调用、Tool Calling、基础 Agent 概念的学习者
- 想搞清楚 MCP 为什么重要的人
- 想把“单个应用内的工具调用”升级成“可复用协议接入”的开发者

## 2. 学习目标

学完这份文档后，你应该能：

1. 用自己的话解释 MCP 是什么，以及它和普通函数调用的关系。
2. 区分 MCP 里的 `Tools`、`Resources`、`Prompts` 分别负责什么。
3. 说明为什么 MCP 更像“协议层”，而不是某一个具体 Agent 框架。
4. 看懂一个最小 MCP 客户端和服务端的交互链路。
5. 判断一个项目什么时候值得接入 MCP，什么时候没必要上 MCP。

## 3. 什么是 MCP

MCP，全称是 `Model Context Protocol`。

你可以先把它理解成：

**让模型、Agent、宿主应用和外部能力之间，用更统一的方法交换工具、资源和提示模板的协议。**

如果你之前学过函数调用，会发现 MCP 并不是在否定函数调用，而是在更高一层做标准化：

- 不只是定义“某个函数怎么调”
- 而是定义“一个能力如何被发现、描述、读取、调用和复用”

所以更准确地说：

**函数调用解决的是“调用一个工具”，MCP 解决的是“怎样把一组能力标准化暴露给不同客户端”。**

## 4. 它解决什么问题

如果没有 MCP，很多系统最后会变成这样：

- 每接一个新工具，就写一套私有接法
- 每个客户端都要自己理解不同工具的参数格式
- 工具之外的上下文能力没有统一暴露方式
- 一个地方接好的工具生态，换个平台就得重接一遍

MCP 想解决的核心问题是：

1. 工具接入标准不统一
2. 上下文提供方式不统一
3. 宿主应用和外部能力耦合太重
4. 工具生态难以跨客户端复用

一句话总结：

**MCP 解决的不是“模型能不能调用工具”，而是“工具和上下文能力能不能标准化接入”。**

## 5. 为什么放在“工具调用与函数调用”这一章

这个仓库里，MCP 最适合把主讲解放在 `04-工具调用与函数调用`，而不是直接塞进 Agent 章节。

原因有三点。

### 5.1 它的前置知识是工具调用

如果下面这些问题还没搞清楚：

- 什么时候要调工具
- 参数怎么设计
- 失败怎么处理
- 工具结果怎么回注给模型

那你学 MCP 时只会觉得它是一堆新接口名。

### 5.2 它本质上是协议层，而不是规划层

MCP 解决的是“能力怎样标准化暴露”，不是：

- 任务如何拆解
- 记忆如何组织
- 多 Agent 如何协作

这些是 Agent 章节更关心的事情。

### 5.3 它最常见的工程价值在“标准化接入”

MCP 真正有价值的地方通常不是让 Agent 看起来更复杂，而是让：

- 客户端统一接工具
- 服务端独立演化
- 资源和 Prompt 也能复用

所以合理顺序是：

1. 先学函数调用
2. 再学 MCP 这种协议化接入
3. 最后在 Agent 项目里使用它

## 6. MCP 的三类核心能力

### 6.1 Tools

Tools 是“可执行动作”。

例如：

- 搜索资料
- 查询数据库
- 创建任务
- 调用天气 API

它解决的是“能做什么动作”。

### 6.2 Resources

Resources 是“可读取上下文”。

例如：

- 一篇文档
- 一组课程笔记
- 某个数据表结构
- 某个配置文件

它解决的是“能提供什么上下文”。

很多初学者容易把所有东西都做成工具，这是不对的。

如果一个能力主要是给模型读，而不是执行动作，那它通常更像 Resource。

### 6.3 Prompts

Prompts 是“可复用的提示模板”。

例如：

- “给初学者解释这个主题”
- “对这份资料做知识点梳理”
- “根据一个主题生成练习题”

它解决的是“如何把一类交互模板标准化复用”。

记忆方式很简单：

- `Tools`：做事
- `Resources`：给上下文
- `Prompts`：给交互模板

## 7. 一个最小 MCP 交互流程

一个最小 MCP Demo，先记住下面这条链路就够了：

1. 客户端启动 MCP 服务端
2. 客户端发送 `initialize`
3. 服务端返回支持的能力
4. 客户端发送 `initialized` 通知
5. 客户端请求 `tools/list`、`resources/list`、`prompts/list`
6. 客户端按需执行 `tools/call` 或 `resources/read`
7. 客户端把结果交给宿主应用或模型继续处理

这里最重要的工程直觉是：

**客户端不需要提前写死服务端所有能力，可以先发现，再使用。**

## 8. MCP 和普通函数调用的区别

可以这样粗略理解：

| 维度 | 普通函数调用 | MCP |
| --- | --- | --- |
| 关注点 | 调一个函数 | 暴露一组标准化能力 |
| 能力范围 | 多数只管工具 | 同时支持工具、资源、Prompt |
| 发现机制 | 常常写死在代码里 | 先 list，再决定怎么用 |
| 可移植性 | 偏平台私有 | 更适合跨客户端复用 |
| 适合场景 | 单应用内快速接入 | 工具生态化、标准化接入 |

所以不要把两者看成互斥关系。

更准确的理解是：

- 普通函数调用是能力本身
- MCP 是暴露能力的协议方式之一

## 9. 常见误区与失败原因

### 9.1 误区一：把 MCP 当成 Agent 框架

MCP 不是 LangGraph，也不是任务规划框架。

它主要定义的是客户端和服务端怎么交换能力。

### 9.2 误区二：所有内容都做成 Tool

如果只是读文档、读配置、拿表结构，很多时候应该建成 Resource，而不是 Tool。

### 9.3 误区三：接了 MCP 就等于系统更智能

MCP 解决的是“接入标准化”，不是“推理质量自动提升”。

模型会不会正确选择工具、能不能读懂结果、会不会失败重试，这些问题仍然存在。

### 9.4 误区四：一上来就做很重的远程服务

最好的入门方式往往不是先做认证、远程部署、复杂网关，而是先做本地最小项目。

先跑通：

- 初始化
- 能力发现
- 工具调用
- 资源读取

再考虑远程化和权限控制。

## 10. 实战建议

### 10.1 第一个项目尽量本地可跑

不要让第一个 MCP 项目依赖太多外部 API，不然你会把大量时间花在：

- 配 Key
- 配网络
- 处理权限
- 对接第三方异常

### 10.2 第一个项目最好同时包含 Tool 和 Resource

这样你才能真正理解：

- 什么是动作
- 什么是上下文

### 10.3 Prompt 也建议加上

很多人学 MCP 只看 Tools，但这样理解是不完整的。

只要项目里加一个最小 Prompt 模板，你就能更直观地理解 MCP 不只是“工具协议”。

### 10.4 项目主题尽量贴近仓库本身

比如这个学习仓库，最适合的 MCP 入门项目不是“天气助手”，而是：

- 学习资料搜索助手
- 课程讲义读取服务
- 教学 Prompt 模板服务

因为这样更容易理解它的工程意义。

## 11. 练习题

### 练习 1

为什么说 MCP 更适合放在“工具调用与函数调用”之后学习，而不是一开始就放进 Agent 章节？

参考思路：

- 先说明 MCP 的前置知识是什么
- 再说明它解决的是协议接入问题而不是规划问题
- 最后说明 Agent 只是 MCP 的使用方之一

### 练习 2

如果一个能力只是“把课程讲义内容读出来”，它更像 Tool 还是 Resource？为什么？

参考思路：

- 判断它是不是执行动作
- 判断它是不是主要提供上下文
- 说明为什么不应该把所有能力都做成 Tool

### 练习 3

如果你要给这个仓库做一个最小 MCP 项目，你会至少暴露哪三种能力？

参考思路：

- 一个搜索类 Tool
- 一个文档类 Resource
- 一个教学类 Prompt

## 12. 总结与下一步建议

MCP 不是替代函数调用，而是把函数、资源和 Prompt 这些能力标准化暴露出来。

在这个仓库里，最合理的放置方式是：

- `04-工具调用与函数调用`：讲清 MCP 的定位、核心对象和协议直觉
- `07-项目实战`：通过一个本地可运行的 Demo，把 `initialize -> list -> call/read` 跑通

下一步建议：

1. 先读完本目录已有的函数调用文档。
2. 再阅读 `07-项目实战/MCP学习资料助手实战.md`。
3. 最后亲自运行一次最小 MCP 客户端和服务端，观察协议交互日志。
