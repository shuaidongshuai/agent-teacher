## 1. 这个项目是什么

这是一个基于 LangGraph 构建的 **Agentic RAG 金融投研助手**。与前面的"智能金融投研助手-RAG"（固定管道）不同，这个项目中 Agent 自主控制检索过程——分析问题、执行检索、评估结果、改写查询、多轮补充，最终生成带引用的答案。

## 2. 它解决什么问题

"分析公司的盈利能力变化趋势，并解释原因，同时对比研发投入是否在增长"

这类问题涉及多个章节、多个维度，一次检索很难覆盖全。传统 RAG 只会检索一次，返回的结果可能只覆盖"盈利能力"而遗漏"研发投入"。

Agentic RAG 的解法：Agent 检索一轮后评估"我还缺什么信息"，然后针对性地改写 query 补充检索，直到信息充分才生成答案。

## 3. 为什么要用 LangGraph

这个项目的执行流程包含：

- **条件分支**：evaluate_results 节点根据评估结果走不同路径
- **循环**：retrieve → evaluate → refine → retrieve 形成闭环
- **状态累积**：每轮检索的结果需要去重累积

这些都是 LangGraph 的 StateGraph 天然擅长的模式。如果用普通代码写，需要手动管理循环、状态和路由；LangGraph 把这些变成了声明式的图拓扑。

## 4. 项目目标与最终效果

运行 `python scripts/run_agent.py`，你会看到：

1. Agent 分析问题的复杂度和信息需求
2. 执行第一轮检索，返回最相关的文档片段
3. Agent 评估"信息够不够"
4. 如果不够，生成新的检索 query，执行第二轮检索
5. 直到信息充分（或达到最大轮数），生成带引用的答案
6. 完整的推理轨迹（execution_log）和统计信息

## 5. 什么是 Agentic RAG

对比三个层次：

| | Naive RAG | Advanced RAG | Agentic RAG |
|--|-----------|-------------|-------------|
| 检索次数 | 1次 | 1次 | 1-N次（Agent决定） |
| Query处理 | 原文 | 改写/扩展 | Agent动态制定 |
| 结果评估 | 无 | 无 | Agent评估充分性 |
| 检索策略 | 固定 | 增强但固定 | 自适应 |

**关键区别：传统 RAG 的流程是预设的，Agentic RAG 的流程是 Agent 运行时动态决定的。**

## 6. 核心流程拆解

### 图拓扑

```text
START → analyze_query → retrieve → evaluate_results ─┬─→ refine_query → retrieve
                                                      └─→ generate_answer → format_output → END
```

### 节点详解

**analyze_query（查询分析）**

输入：用户原始问题

LLM 做什么：
- 判断问题复杂度（simple / moderate / complex）
- 列出需要获取的信息点（如：["盈利能力变化", "毛利率下降原因", "研发投入趋势"]）
- 生成初始检索 query

为什么需要这一步：直接用用户原文检索，效果往往不好。先分析再检索，可以制定更精准的策略。

**retrieve（检索执行）**

输入：current_query

不调用 LLM，直接调用 KnowledgeBase：
- 混合检索（BM25 + FAISS 向量）
- RRF 融合
- 交叉编码器重排序

结果去重后累积到 accumulated_contexts。

**evaluate_results（结果评估）**

输入：用户问题 + 所有累积上下文 + 已检索轮数

LLM 做什么：
- 评估当前信息是否足以回答
- 输出覆盖摘要和缺失信息
- 做出路由决策：answer / refine_query / retrieve

这是图中唯一的条件路由节点。

**refine_query（查询改写）**

输入：原始问题 + 历史 query + 覆盖摘要 + 缺失信息

LLM 做什么：
- 根据"已有什么、缺什么"生成新 query
- 确保不重复已搜过的 query
- 针对缺失信息点设计

**generate_answer（答案生成）**

输入：原始问题 + 所有累积上下文

LLM 做什么：
- 基于上下文生成结构化回答
- 引用出处（页码、章节）
- 如果信息不足，明确指出

## 7. Agentic RAG 的核心能力

1. **自主检索判断**：不是每轮都检索，而是 Agent 评估后决定
2. **多轮信息累积**：每轮检索的新结果去重后累积
3. **针对性查询改写**：根据"缺什么"生成新 query，而不是随机改写
4. **可控的执行边界**：最大轮数 + LLM 调用上限，防止 Agent 失控
5. **完整的推理轨迹**：execution_log 记录每个节点的决策和理由

## 8. Agentic RAG 的核心难点

1. **evaluate_results 的准确性**：LLM 判断"信息是否充分"并不总是准确
2. **refine_query 的有效性**：改写后的 query 如果和之前太像，会检索到重复内容
3. **成本控制**：每多一轮检索就多消耗 2 次 LLM 调用
4. **上下文膨胀**：累积太多上下文可能超出 context window 或降低生成质量
5. **延迟**：多轮检索的延迟是固定管道的数倍

## 9. 项目代码结构

```text
agent-agentic-rag/
├── app/
│   ├── config.py           # 配置中心：路径、模型名、最大轮数、LLM 参数
│   ├── llm_client.py       # OpenAI 兼容 API 客户端
│   ├── prompts.py          # 所有 LLM prompt 集中管理
│   ├── state.py            # LangGraph 状态定义（TypedDict）
│   ├── nodes.py            # 6 个图节点的实现（核心逻辑）
│   ├── graph.py            # LangGraph 图构建和编译
│   ├── knowledge_base.py   # 知识库封装（整合检索模块）
│   └── rag/                # RAG 能力模块
│       ├── bge_embedding.py    # BGE 中文 embedding
│       ├── vector_store.py     # FAISS 向量存储
│       ├── bm25_retriever.py   # BM25 关键词检索
│       ├── hybrid_retriever.py # RRF 混合检索
│       └── reranker.py         # 交叉编码器重排序
├── scripts/
│   ├── build_index.py      # 构建知识库索引
│   └── run_agent.py        # 主入口
└── data/
    ├── sample_blocks.json  # 金融年报模拟数据
    └── sample_queries.json # 测试查询
```

**重点阅读文件：**
- `app/nodes.py` — 所有 Agent 决策逻辑
- `app/prompts.py` — 所有 prompt 设计
- `app/graph.py` — 图拓扑和路由

## 11. 你需要重点观察什么

1. **推理轨迹**：看 execution_log 中每一步的决策和理由
2. **检索轮数**：简单问题是否 1 轮就结束？复杂问题是否触发了多轮？
3. **query 演变**：query_history 中的 query 是否在不断聚焦？
4. **评估决策**：evaluate_results 说"够了"或"不够"的理由是否合理？
5. **最终答案**：引用是否准确？是否有遗漏？

## 12. 常见误区与失败原因

1. **所有问题都触发 3 轮检索**：说明 evaluate prompt 太严格
2. **改写 query 和原 query 太像**：说明 refine prompt 没有利用好"缺失信息"
3. **答案引用了不存在的数据**：检索结果不够好 + 生成 prompt 约束不够强
4. **Agent 判断"信息充分"但答案不完整**：evaluate 的信息点检查不到位
5. **运行很慢**：模型下载（首次）或 LLM API 延迟

## 13. 实战练习题

### 练习1

用 `data/sample_queries.json` 中的 4 个查询分别运行 Agent，记录：
- 每个查询的检索轮数
- query_history 的演变过程
- 最终答案的质量评分（满分 5 分）

### 练习2

修改 `app/config.py` 中的 `max_retrieval_rounds` 为 1，强制 Agent 只能检索一次。对比与默认（3 轮）的答案质量差异。

### 练习3

打开 `app/prompts.py`，修改 EVALUATE_RESULTS_PROMPT，让它包含一个"信息点清单逐项检查"的机制。观察对 Agent 决策质量的影响。

### 练习4

在 `app/nodes.py` 中增加一个 `self_check` 节点，在 generate_answer 之后、format_output 之前，让 LLM 检查答案是否有幻觉或遗漏。

## 14. 参考答案与思考方向

### 练习1 思考方向

- 简单问题（如"毛利率最高的板块"）通常 1 轮就够
- 复杂问题（如"从收入和成本两个维度评估风险"）需要 2-3 轮
- 如果所有问题都走满 3 轮，说明评估 prompt 需要调优

### 练习2 思考方向

- 限制为 1 轮后，简单问题影响不大，复杂问题会明显缺失信息
- 这直观证明了 Agentic RAG 相比固定管道 RAG 的价值

### 练习3 思考方向

- 可以在 analyze_query 阶段输出 info_points 列表
- 在 evaluate_results 中逐一检查每个 info_point 是否被覆盖
- 这类似于"检查清单"模式，可以显著提升评估准确性

### 练习4 思考方向

- self_check 节点需要新的 prompt，让 LLM 对比答案和上下文
- 如果发现幻觉，可以标注出来但不修改（修改可能引入新问题）
- 这是 Self-RAG 的思路雏形

## 15. 总结与下一步建议

这个项目的核心学习点：

1. **Agentic RAG 的本质是 Agent 控制检索决策**，不是简单的多轮检索
2. **LangGraph 的条件边和循环**是实现自适应流程的关键工具
3. **prompt 设计决定 Agent 质量**——尤其是 evaluate_results 和 refine_query 的 prompt
4. **安全控制不可或缺**——没有最大轮数和调用上限，Agent 会失控

下一步建议：

1. 回到 `05-Agent/6.Agentic RAG：当Agent遇见检索.md` 加深理论理解
2. 学习 `06-评测与优化/5.Agent评测.md`，思考如何评测 Agent 的检索决策质量
3. 尝试扩展项目：添加多知识库支持、Web 搜索工具、Self-RAG 自检机制

一句话总结：

**Agentic RAG 的核心竞争力是"智能决策"而不是"更多检索"——Agent 知道什么时候该检索、检索什么、什么时候该停。**
