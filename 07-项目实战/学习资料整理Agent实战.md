# 学习资料整理 ReAct Agent 实战

## 适用人群

这份实战适合已经理解 Agent 基本概念，并希望进一步练习真正的 ReAct 闭环、工具调用和多轮任务推进的人。

## 学习目标

完成这个项目后，你应该能够：

1. 理解为什么“让 LLM 自己决定是否用工具”比“提前写死流程”更像 Agent
2. 理解 ReAct 模式中 `Thought -> Action -> Observation` 的代码落地方式
3. 看懂一个可扩展工具注册表如何和 LLM 决策循环结合
4. 自己新增一个工具，让 LLM 在后续轮次里使用它

## 1. 这个项目是什么

这是一个真正使用 ReAct 模式的学习资料整理 Agent。

你给它一个总任务，例如：

- 为 RAG 入门生成一套教学资料
- 为 Prompt 工程基础生成一组讲义
- 为 Agent 工作流设计生成学习文档

它不会一开始就把所有内容一次性生成完，而是会这样工作：

1. 先思考当前最该做什么
2. 再决定是否需要工具
3. 如果需要工具，就指定工具和参数
4. Agent 执行工具
5. 把工具结果回传给 LLM
6. 让 LLM 再决定下一步
7. 直到 LLM 自己判断任务完成

## 2. 为什么它比前一个版本更像 Agent

前一个版本虽然已经有了 `plan -> act -> observe -> reflect` 的结构，但还是偏“框架式 Agent”：

- 很多动作逻辑是规则提前写好的
- LLM 主要参与内容生成
- 工具选择还不够自由

这次的版本更像真正的 Agent，因为：

### 2.1 LLM 自己决定是否使用工具

它不只是负责写正文，还负责判断下一步到底要不要工具。

### 2.2 工具结果会回流到下一轮决策

工具不是“调用完就算了”，而是会变成下一轮推理的输入。

### 2.3 任务不是一步完成，而是多轮推进

这意味着 Agent 真正开始表现出“边做边想”的特征。

## 3. 这个项目解决什么问题

很多人做 Agent 项目时，其实是在做下面这种东西：

- 模型先一次性生成计划
- 系统按固定流程执行
- 过程里几乎没有真正的自主决策

这更像 workflow，而不是真正的 Agent。

本项目要解决的是：

**让 LLM 围绕一个总任务，多轮决定是否调用工具、调用什么工具、什么时候结束。**

## 4. ReAct 核心流程

### 4.1 Thought

LLM 先读取：

- 当前总任务
- 输出目录
- 可用工具
- 历史轨迹
- 上一轮的 Observation

然后思考现在最合理的一步是什么。

### 4.2 Action

LLM 必须在两种动作中二选一：

1. `tool`
2. `finish`

如果是 `tool`，它还要给出：

- 工具名
- 工具参数

### 4.3 Observation

Agent 执行工具，把结果整理成结构化 Observation。

### 4.4 Next Thought

下一轮，LLM 会看到上一轮的 Observation，并继续思考。

所以这才是真正的 ReAct：

**Thought -> Action -> Observation -> Thought -> Action -> Observation**

## 5. 工具系统设计

这个项目的工具不是写死在 Prompt 里的字符串，而是一个可扩展的注册系统。

当前默认工具包括：

- `list_dir`
- `read_file`
- `write_file`
- `append_file`
- `make_dir`
- `file_exists`

你后面还可以继续加：

- `delete_file`
- `search_text`
- `replace_text`
- `save_summary`

重点不是工具越多越好，而是：

**LLM 是否真的在根据任务需要选择工具。**

## 7. 如何运行

### 使用 LLM 模式

```bash
export OPENAI_API_KEY=你的Key
export OPENAI_BASE_URL=https://api.chatanywhere.tech
export OPENAI_MODEL=gpt-4o
export OPENAI_API_STYLE=chat_completions
export OPENAI_SSL_VERIFY=false

python3 07-项目实战/agent-study-builder/main.py "RAG 入门" --audience "初学者"
```

### 不配置 Key 时

项目会回退到最小本地模式，只生成一个入口文档，主要用于保证项目可运行。

## 8. 重点观察什么

### 8.1 LLM 的 Thought 是否真的在变化

如果每一轮的 Thought 都只是重复模板，那说明它还没有真正利用 Observation。

### 8.2 工具是否是“按需选择”的

真正的 Agent 不是固定顺序调用工具，而是根据当前情况选择。

### 8.3 Observation 是否真的影响下一轮

如果上一轮工具结果没有进入下一轮推理，那不算真正的 ReAct。

### 8.4 最终是否真的产出了文件

一个好的 Agent Demo，不是只打印出推理链，而是要对外部环境产生真实结果。

## 9. 实战作业

### 作业 1：跑 3 个不同主题

建议至少试：

1. `RAG 入门`
2. `Prompt 工程基础`
3. `Agent 工作流设计`

### 作业 2：观察 ReAct 轨迹

记录下面几个问题：

1. 第一轮它为什么先选那个工具？
2. 它有没有主动读回自己写的文件？
3. 它什么时候判断任务完成？

### 作业 3：加一个新工具

建议从下面任选一个：

1. `delete_file`
2. `search_text`
3. `replace_text`
4. `save_report`

### 作业 4：让 Agent 更稳

你可以尝试增加这些约束：

1. 限制单轮最大写入长度
2. 要求写完文件后必须读回验证
3. 要求 finish 前必须至少检查一次目录结构

## 10. 总结与下一步建议

这个版本比之前更像真正的 Agent，不是因为它“写了更多代码”，而是因为它已经具备了更关键的特征：

1. 总任务驱动
2. 多轮推理
3. 自主工具选择
4. 工具结果反馈
5. 直到完成才结束

一句话总结：

**真正像 Agent 的系统，不是一次把答案说完，而是在环境里边想边做，边看结果边继续推进。**
