## 1. 这个项目是什么

这是一个使用 Planner-Executor 模式的教学型 Agent 项目。

它不让一个模型一边想一边乱跳步骤，而是强制拆成两个角色：

1. 架构师先产出一份结构化计划
2. 工人只负责按计划逐步执行
3. 每一步执行后记录状态
4. 如果某一步失败，只重试那一步
5. 全部完成后统一整理最终结果

这个模式的核心逻辑就是：

```text
Plan (Step 1, 2, 3...) -> Execute All -> Final Answer
```

## 2. 它解决什么问题

很多 Agent Demo 的常见问题是：

- 规划和执行混在一起
- 模型每一轮都可能临时改主意
- 中间失败后只能整段重来
- 轨迹很难复盘，也很难定位是哪一步出了错

Planner-Executor 想解决的是：

**先把路线想清楚，再像流水线一样稳定执行。**

对于教学场景，这种模式尤其好，因为你能清楚看到：

- 计划长什么样
- 每一步为什么存在
- 执行阶段到底做了什么
- 失败时是如何被隔离和重试的

## 3. 为什么这次要用 Planner-Executor

前面的最小 Agent 和 ReAct Agent 更强调灵活性，而这次更强调稳定性。

Planner-Executor 的优势主要有 3 点。

### 3.1 规划和执行彻底解耦

架构师的任务是“想清楚”，工人的任务是“做出来”。

这样做的好处是：

- 规划逻辑更容易单独优化
- 执行逻辑更容易单独测试
- 你不会把“计划不合理”和“执行失败”混成一类问题

### 3.2 更适合做可复盘的流水线

当计划已经固定后，执行阶段更像一个工程管线：

1. 读计划
2. 执行第 1 步
3. 执行第 2 步
4. 执行第 3 步
5. 汇总结果

这比“边想边做”更可控。

### 3.3 更容易支持单步重试

如果第 3 步失败，不需要把 1、2 步也推倒重来。

你只需要：

1. 记录失败原因
2. 对当前步骤重试
3. 超过上限后中止并输出日志

这就是它稳定性强的关键原因。

## 5. 核心流程拆解

### 5.1 第一步：架构师生成计划

架构师模型负责输出结构化步骤清单。

每个步骤至少包括：

- 步骤编号
- 步骤名称
- 目标
- 要写入的文件
- 完成标准

重点是：

**计划必须足够细，工人才能稳定执行。**

### 5.2 第二步：工人按步骤执行

工人不会重新规划，也不会擅自跳步。

它只会：

1. 读取当前步骤
2. 生成对应内容
3. 写入目标文件
4. 返回执行结果

### 5.3 第三步：失败时只重试当前步骤

如果某一步因为内容为空、文件写入失败、输出不合格而失败，系统会只针对这一小步重试。

这就是 Planner-Executor 和很多“整段对话式 Agent”的区别：

**失败是局部处理的，不是全局重来。**

### 5.4 第四步：统一输出最终总结

全部步骤执行完成后，系统会根据：

- 原始任务
- 计划
- 每一步产物
- 执行日志

生成最终总结文件，方便复盘整个流水线。

## 6. 架构师与工人的职责边界

这是这个项目最值得你学习的地方。

### 6.1 架构师负责什么

架构师负责：

- 理解总任务
- 定义执行顺序
- 约束每一步产出
- 让计划足够可执行

它不直接写最终文件。

### 6.2 工人负责什么

工人负责：

- 接受单步任务
- 按要求产出内容
- 写文件
- 汇报成功或失败

它不负责改大方向。

### 6.3 为什么这种分工重要

如果让同一个执行角色一边做一边改计划，经常会出现：

- 中间插入多余步骤
- 忘掉原始目标
- 每轮输出风格都变化
- 出错时没有明确责任边界

所以在工程上，这种模式很像：

- 架构师画蓝图
- 工人按蓝图施工

## 9. 你需要重点观察什么

### 9.1 计划是否足够细

如果计划只是“先学习，再练习，再总结”，那工人其实没法稳定执行。

好的计划应该能直接转成执行任务。

### 9.2 执行器是否真的不改计划

这是最关键的观察点。

如果执行器又开始自己决定顺序、自己新增步骤，那就不是纯粹的 Planner-Executor 了。

### 9.3 失败是否被限制在单步内

重点看：

- 哪一步失败了
- 重试了几次
- 其他已完成步骤有没有被重复执行

### 9.4 最终产物是否真实落地

一个好的执行型 Agent，不应该只输出“我完成了”，而应该真的生成文件、日志或外部结果。

## 10. 常见误区与失败原因

### 10.1 计划太粗

这是最常见的问题。

计划过粗时，工人要自己补脑，最后又变回“边想边做”。

### 10.2 步骤产出标准不明确

如果没有说明每一步要写到哪个文件、完成标准是什么，执行阶段就很难自动化。

### 10.3 执行器权限过大

如果你允许执行器任意删改计划，它就会慢慢退化成一个弱化版 ReAct，而不是稳定流水线。

### 10.4 没有状态记录

如果不记录每一步状态，你就没法知道：

- 哪一步完成了
- 哪一步失败了
- 哪一步需要重试

### 练习 1：运行 3 个不同主题

建议至少试：

1. `Agent 稳定性设计`
2. `RAG 检索流程入门`
3. `Prompt 工程基础`

### 练习 2：故意让某一步失败

你可以手动修改代码，让某一步在第一次执行时抛错，然后观察：

1. 是否只重试当前步骤
2. 已完成步骤是否被保留
3. 最终日志是否能定位问题

### 练习 3：新增一个执行步骤

例如新增：

1. `05_术语表.md`
2. `06_常见误区.md`

练习把新步骤加入架构师计划模板和工人执行逻辑中。

## 12. 参考答案与思考方向

练习题没有唯一标准答案，但建议你重点往下面几个方向思考。

### 12.1 判断一个 Planner 是否合格

一个合格的 Planner 至少要做到：

- 步骤顺序合理
- 每一步边界清晰
- 每一步有明确产出
- 工人拿到步骤后不用重新猜意思

### 12.2 判断一个 Executor 是否稳定

一个稳定的 Executor 至少要做到：

- 严格按计划执行
- 对失败步骤做有限次重试
- 产出可落地文件
- 记录状态和错误信息

### 12.3 什么时候不适合 Planner-Executor

如果任务高度开放、需要频繁根据环境反馈调整路线，比如：

- 自主探索式网页操作
- 多轮外部搜索与动态决策
- 高不确定性的长链工具调用

这时 ReAct 或图调度模式可能更合适。

## 13. 总结与下一步建议

Planner-Executor 模式非常适合你学习下面这些能力：

- 如何把模糊任务变成可执行步骤
- 如何把执行过程做成稳定流水线
- 如何让失败可定位、可重试、可复盘

如果你已经跑通这个项目，下一步建议继续做两件事：

1. 给执行器加入文件校验和质量检查
2. 把单个工人升级成多个专职工人，例如“写作者、审稿者、整理者”

这样你就会自然走到更复杂的多角色 Agent 系统设计。
