## 1. 这个项目是什么

这是一个基于 LangGraph 构建的**记忆增强对话 Agent**。与 `agent-chat-langgraph`（使用 regex 提取 + JSON 存储）不同，这个项目用 LLM 提取记忆、FAISS 语义检索、LLM 压缩三个核心技术，实现了更智能的三层记忆。

## 2. 它解决什么问题

"我上周跟你说过我住在北京，你还记得吗？"

这类跨会话记忆在传统对话系统中几乎不可能实现。更难的是：

"我之前提到的那个 RAG 项目进展怎样了？"

这需要 Agent 不仅记住"用户在做 RAG 项目"，还要能从语义层面把"那个项目"和之前的对话关联起来。

## 3. 为什么要用三层记忆

单层记忆的局限：

| 方案 | 问题 |
|------|------|
| 全量保存消息 | Context window 爆炸，20 轮后回答质量下降 |
| 只保留最近 N 条 | 早期信息丢失 |
| 只用长期存储 | 缺少当前对话的连贯性 |
| 只用关键词匹配 | 无法理解语义相似性 |

三层协作：短期保连贯性，长期保跨会话，工作记忆保任务聚焦。

## 4. 项目目标与最终效果

运行 `python scripts/run_chat.py`，你会看到：

1. 告诉 Agent 你的信息（姓名、城市、偏好）
2. 几轮对话后询问"你还记得我叫什么吗"，Agent 能准确回答
3. 退出并重新启动，Agent 仍然记得（长期记忆持久化）
4. 输入 `/memory` 查看三层记忆的实时状态

## 5. 三层记忆架构详解

### 5.1 短期记忆（ShortTermMemory）

**机制**：维护消息列表 + LLM 压缩

**触发条件**：消息数 > 20 条时自动压缩

**压缩过程**：
```text
[旧消息 1..14] → LLM 压缩 → "用户是数据工程师，在北京工作，正在学习 RAG..."
[新消息 15..20] → 保留原文
```

**结果**：context = 摘要（200字） + 最近 6 条消息

### 5.2 长期记忆（LongTermMemory）

**机制**：LLM 提取 + BGE 编码 + FAISS 索引 + JSON 持久化

**存储流程**：
```text
每轮对话 → LLM 判断"有什么值得记住的" → 提取条目 → BGE 编码 → FAISS 索引 + JSON 文件
```

**检索流程**：
```text
新输入 → BGE 编码（带检索前缀）→ FAISS 语义搜索 → top-5 相关记忆
```

### 5.3 工作记忆（WorkingMemory）

**机制**：结构化字典，每轮由 LLM 更新

```python
{
    "current_goal": "帮用户优化 RAG 召回率",
    "key_facts": ["召回率约60%", "已尝试调chunk size"],
    "pending_questions": ["embedding 模型是什么？"],
    "reasoning_steps": ["先检查embedding质量"]
}
```

注入到 system prompt，帮助 LLM 保持任务聚焦。

## 6. 核心流程拆解

### 图拓扑

```text
START → load_memory → chat → extract_and_store → compress_if_needed → END
```

### 节点详解

**load_memory**：新一轮开始时，用当前 query 从长期记忆语义检索。不调用 LLM，只用 FAISS。

**chat**：构建完整的 system prompt（含三层记忆），调用 LLM 生成回复。这是唯一面向用户的节点。

**extract_and_store**：LLM 从本轮对话中提取值得记忆的信息，存入长期记忆。同时记录到短期记忆。

**compress_if_needed**：检查短期记忆是否超过阈值，超过则触发 LLM 压缩。同时更新工作记忆。

## 7. 记忆系统的核心能力

1. **跨会话记忆**：用户关闭对话再重新打开，Agent 仍然记得
2. **语义检索**：不靠关键词精确匹配，而是理解语义相似性
3. **自动压缩**：长对话不会撑爆 context，自动压缩旧消息
4. **选择性记忆**：LLM 判断什么信息值得记住，不记垃圾信息
5. **任务聚焦**：工作记忆帮助 Agent 不偏题

## 8. 记忆系统的核心难点

1. **提取质量**：LLM 可能遗漏重要信息，或提取不重要的信息
2. **压缩损失**：摘要必然丢失细节，可能丢掉关键信息
3. **检索噪音**：语义相似但实际不相关的记忆可能被召回
4. **成本增长**：每轮多 2-3 次 LLM 调用
5. **记忆冲突**：用户修正信息时（"我搬到上海了"），旧记忆和新记忆冲突

## 9. 项目代码结构

```text
agent-memory-enhanced/
├── app/
│   ├── config.py           # 配置（短期阈值、长期top-k等）
│   ├── llm_client.py       # OpenAI 兼容客户端
│   ├── prompts.py          # 4 个核心 prompt
│   ├── state.py            # LangGraph 状态定义
│   ├── memory/
│   │   ├── short_term.py   # 短期记忆（LLM 压缩）
│   │   ├── long_term.py    # 长期记忆（FAISS + JSON）
│   │   └── working.py      # 工作记忆（scratch pad）
│   ├── nodes.py            # 4 个节点（make_nodes 闭包）
│   └── graph.py            # 图构建 + run_turn 函数
├── data/
│   ├── sample_conversations.json  # 测试对话
│   └── memory_store/              # 持久化目录
└── scripts/
    ├── run_chat.py          # 交互式入口
    └── run_memory_demo.py   # 自动演示
```

**重点阅读文件：**
- `app/memory/long_term.py` — FAISS 语义记忆的核心实现
- `app/memory/short_term.py` — LLM 压缩逻辑
- `app/nodes.py` — 记忆与 LangGraph 的集成方式

## 11. 你需要重点观察什么

1. **记忆提取**：每轮对话后，Agent 提取了哪些记忆？合理吗？
2. **记忆召回**：问"我之前说什么"时，Agent 召回的记忆是否准确？
3. **压缩质量**：20 轮后触发压缩，摘要是否保留了关键信息？
4. **工作记忆变化**：`current_goal` 和 `key_facts` 是否随对话演变？
5. **持久化验证**：退出再进入，长期记忆是否保留？

## 12. 常见误区与失败原因

1. **记忆太多**：每轮都提取 5+ 条记忆，检索噪音大 → 调整提取 prompt 标准
2. **记忆太少**：重要信息没被提取 → 提取 prompt 太严格
3. **压缩丢失关键信息**：用户姓名在压缩后消失 → 调整压缩 prompt，强调保留身份信息
4. **语义检索不准**：检索到不相关的记忆 → 检查 embedding 质量，考虑调高相似度阈值
5. **工作记忆不更新**：goal 始终是"自由对话" → 检查 UPDATE_WORKING_MEMORY_PROMPT

## 13. 实战练习题

### 练习 1

运行 `run_memory_demo.py`，记录每组测试对话的：
- 提取的记忆条目数
- 最后一轮的召回准确率
- 总 LLM 调用次数

### 练习 2

修改 `app/memory/long_term.py`，为每条记忆添加"重要性分数"（1-5），在检索时用重要性加权。观察对召回质量的影响。

### 练习 3

修改 `app/memory/short_term.py`，将压缩阈值从 20 改为 8（更激进的压缩）。运行 `demo_004`（长对话测试），对比摘要质量。

### 练习 4

在 `app/nodes.py` 中增加一个 `conflict_check` 节点：当新记忆和已有记忆冲突时（如"住在上海" vs "住在北京"），用新记忆替换旧记忆。

## 14. 参考答案与思考方向

### 练习 1 思考方向

- `demo_001`（基础信息）：应该提取 3-4 条记忆，最后一轮准确率应接近 100%
- `demo_003`（任务跟踪）：工作记忆应正确追踪"RAG 项目"和"召回率 60%"
- 如果所有测试都失败，先检查 API key 是否有效

### 练习 2 思考方向

- 重要性分数可以在 EXTRACT_MEMORIES_PROMPT 中让 LLM 输出
- 检索时 score = cosine_similarity × importance_weight
- 个人信息应该比一般性讨论更重要

### 练习 3 思考方向

- 阈值 8 意味着每 4 轮就压缩一次
- 多次压缩是"摘要的摘要"，信息损失会累积
- 关键信息（姓名、城市）应该在压缩中特别保护

### 练习 4 思考方向

- 冲突检测可以用 LLM：给它新记忆和相同 category 的旧记忆，让它判断是否冲突
- 冲突时用新记忆替换旧记忆（因为用户信息可能更新）
- 但要小心误判：同一 category 不同 topic 的记忆不是冲突

## 15. 总结与下一步建议

这个项目的核心学习点：

1. **三层记忆是工程权衡的结果**——每层解决一个具体问题
2. **LLM 提取 + FAISS 检索**是当前 Agent 记忆的主流技术路线
3. **压缩是有损的**——设计 prompt 时要明确"什么信息必须保留"
4. **记忆系统的价值取决于检索质量**——存了 100 条但检索不到有用的，等于没存
