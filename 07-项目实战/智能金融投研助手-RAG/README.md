完成这个项目后，你应该能够：

1. 理解金融 RAG 系统从 PDF 解析到回答生成的完整链路
2. 理解为什么金融场景不能只靠“固定切片 + 向量检索”
3. 搭建一个可持续扩展的实战项目骨架
4. 在现有代码上继续实现混合搜索、重排序、查询转换、评估与 LangGraph 编排

## 项目目标

这个系统要面向金融 PDF 报告场景，例如：

- 上市公司财报
- 券商研报
- 行业深度报告

目标问题示例：

- 某公司去年的毛利率变化趋势及其原因？
- 过去三年研发投入变化与净利润趋势是否一致？
- 这份研报给出的核心投资逻辑是什么？

系统最终会逐步演进为：

1. 能解析大量金融 PDF
2. 能做高级切片
3. 能做混合检索
4. 能做重排序
5. 能做查询转换
6. 能做回答生成与评估
7. 能用 LangGraph 管理复杂状态流

## 项目路线图

### 里程碑 0：定义问题与评测样本

先明确系统要回答的问题类型，并准备一批黄金测试集。

建议覆盖：

1. 单文档事实问答
2. 跨页归纳问答
3. 多文档对比问答

### 里程碑 1：金融 PDF 解析、清洗与结构化

目标：

- 把 PDF 解析成结构化 block
- 去除页眉、页脚、免责声明等噪声
- 保留标题层级、表格、图表说明等结构信号

### 里程碑 2：高级切片

目标：

- 不做固定长度切片
- 结合文档结构与语义边界做 chunk
- 保留 metadata，便于后续检索和出处追踪

### 里程碑 3：多维向量化

目标：

- 对比不同 embedding 模型
- 处理金融术语、公司简称、年份和指标名的表达偏差
- 尝试标题向量、正文向量、摘要向量等多视角索引

### 里程碑 4：混合检索

目标：

- 结合 BM25 与向量检索
- 处理“强关键词 + 语义近似”两类问题
- 做简单分数融合

### 里程碑 5：查询转换

目标：

- 支持查询重写
- 支持 HyDE
- 让模糊问题变成更容易检索的问题

### 里程碑 6：重排序

目标：

- 使用 cross-encoder 或 reranker 对 top-k 结果重新排序
- 缓解召回结果顺序不准、证据埋没的问题

### 里程碑 7：生成层

目标：

- 基于检索证据生成答案
- 强制带出处
- 降低金融场景中的幻觉风险

### 里程碑 8：LangGraph 编排

目标：

- 把 RAG 各节点连接成状态图
- 当检索不足时自动触发后续补救流程

### 里程碑 9：评估体系

目标：

- 使用 RAGAS 评估 Context Precision、Faithfulness
- 使用 TruLens 做在线或开发时监控

## 如何运行

在仓库根目录执行：

```bash
python3 07-项目实战/智能金融投研助手-RAG/scripts/run_chunking_demo.py
```

这个 demo 会：

1. 加载一组模拟金融报告 block
2. 做清洗
3. 做结构化 + 语义切片
4. 把结果打印到终端

## 第一阶段重点：数据清洗与语义切片

这一阶段最重要的不是“把文本切开”，而是：

**保留金融文档里真正有检索价值的结构信号。**

你需要重点关注：

1. 标题是不是强边界
2. 表格是不是单独成块
3. 相邻段落是否真的属于同一语义单元
4. chunk 上是否保留了页码、标题层级、块类型等 metadata

## 下一步建议

建议你接下来按这个顺序继续推进：

1. 用真实 PDF 解析替换 `parser_stub.py`
2. 把 `SimpleEmbeddingModel` 替换为真实 embedding 模型
3. 落地 BM25 和向量检索双索引
4. 增加 reranker
5. 用 LangGraph 串联整个流程

一句话总结：

**金融 RAG 的上限，往往不是从模型开始，而是从文档解析、清洗和切片开始。**
