# 学习任务拆解 Agent 实战

## 适用人群

这份实战适合已经学过 LLM、Prompt、工具调用、Agent 基本概念，并希望做第一个可运行 Agent 小项目的学习者。

## 学习目标

完成这个实战后，你应该能够：

1. 理解一个最小 Agent 项目应该由哪些模块组成
2. 理解规划、执行、反思为什么是 3 个不同阶段
3. 自己跑通一个命令行 Agent，并观察它的轨迹输出
4. 在现有代码上继续扩展记忆、工具调用和真实模型接入

## 1. 这个项目是什么

这是一个最小可运行的命令行 Agent 项目。

你输入一个学习目标，比如：

- 3 天内入门 Python
- 一周内完成 RAG 基础学习
- 准备一次 Agent 技术分享

这个 Agent 会做 4 件事：

1. 分析目标
2. 生成任务计划
3. 给出执行建议
4. 对计划进行一次自检和修正

最后，它会输出一份更像“可执行学习计划”的结果，而不是只给你一段泛泛而谈的建议。

## 2. 它解决什么问题

很多初学者第一次做 Agent 项目时，容易一上来就做“联网搜索、多工具、多 Agent、长记忆”。

这样的问题是：

- 代码结构一下子变复杂
- 出错时很难定位问题
- 你不知道失败是 Prompt 问题、工具问题，还是流程设计问题

所以第一个 Agent 项目，最好先解决一个非常具体的问题：

**把一个模糊目标，变成一份阶段化、可执行、可复盘的任务计划。**

这个问题足够简单，但又能体现 Agent 的核心思想：

- 有目标
- 有规划
- 有执行建议
- 有反思修正

## 3. 为什么适合作为第一个 Agent 实战

这个项目有几个优点：

### 3.1 问题边界清楚

输入是学习目标，输出是学习计划，不需要太多外部系统依赖。

### 3.2 能体现 Agent 闭环

虽然它很小，但已经具备最关键的结构：

- Goal：目标
- Planner：规划器
- Executor：执行建议生成器
- Reflector：反思器

### 3.3 容易升级

你后面可以逐步把它升级成：

- 接入真实 OpenAI 模型
- 接入日历工具
- 接入本地知识库
- 加入长期记忆
- 支持多轮迭代规划

## 4. 项目目标与最终效果

### 输入示例

```text
一周内完成 Agent 基础学习，并做一个最小 Demo
```

### 输出示例

- 目标理解
- 按天拆解的任务列表
- 每一步的产出物
- 风险提示
- 自检后的改进建议

你会发现，这比普通“给我一个学习建议”的问答，更接近一个真正的执行型 Agent。

## 5. 核心流程拆解

整个项目采用非常适合教学的 4 步结构。

### 5.1 第一步：目标分析

先把用户输入拆成几个关键维度：

- 主题是什么
- 时间限制是什么
- 预期产出是什么
- 当前缺失的信息是什么

这一步的意义是让 Agent 不要直接跳到“给答案”，而是先理解任务。

### 5.2 第二步：任务规划

根据目标分析结果，生成阶段化任务。

例如：

1. 先补基础概念
2. 再看经典结构
3. 然后做最小 Demo
4. 最后复盘与改进

这一步体现的是 Agent 的规划能力。

### 5.3 第三步：执行建议

仅有任务标题还不够，Agent 需要把任务变成可执行动作，比如：

- 今天读哪些内容
- 做什么练习
- 产出什么结果
- 每步预计耗时多久

这一步体现的是 Agent 的执行导向，而不是停留在概念层面。

### 5.4 第四步：反思修正

最后，Agent 对自己的计划做一次简单检查，例如：

- 步骤是否过多
- 时间是否不现实
- 产出物是否不明确
- 是否缺少复盘环节

这一步虽然简单，但非常关键，因为它帮助你理解：

**Agent 不是只会规划，它还需要判断自己的规划是否靠谱。**

## 6. 你需要重点观察什么

第一次跑项目时，不要只看“结果像不像人写的”，而要重点看下面几点：

### 6.1 输入是如何被解析的

也就是 Agent 如何把一句模糊目标转换成结构化任务信息。

### 6.2 计划是如何分阶段的

规划不是随便列清单，而是要符合时间限制和目标结果。

### 6.3 反思器改了什么

如果反思阶段没有发挥作用，这个 Agent 其实还只是“单次生成器”。

### 8.4 轨迹输出是否清晰

一个好的学习型 Agent 项目，应该让你看得懂每一步发生了什么。

## 9. 常见误区与失败原因

### 9.1 一开始就接太多工具

结果是工程复杂度远大于学习价值。

### 9.2 只有最终答案，没有过程轨迹

这样你不知道 Agent 为什么做这个决策，也无法调试。

### 9.3 把规划和执行混在一起

如果没有清楚分层，后面会很难扩展。

### 9.4 反思只是重复原答案

这是假反思。真正的反思应该指出风险、遗漏或不合理之处。

## 10. 实战作业

下面这部分就是你这次实战的完整作业。

### 作业 1：跑通项目

请至少运行 3 次，分别输入不同类型目标，例如：

1. `三天内入门 Python`
2. `一周内完成 RAG 基础学习`
3. `准备一次 20 分钟的 Agent 分享`

### 作业 2：观察规划质量

记录下面 3 个问题：

1. 它有没有正确理解时间约束？
2. 它给出的产出物是否清晰？
3. 它的步骤是否可执行？

### 作业 3：自己扩展一个能力

从下面任选一个：

1. 增加优先级字段
2. 增加每日学习时长输入
3. 增加失败风险评分
4. 增加保存结果到 Markdown 文件的功能

### 作业 4：写一份复盘

请回答下面 4 个问题：

1. 这个 Agent 真正解决了什么问题？
2. 它最像 Agent 的地方是什么？
3. 它最不像 Agent 的地方是什么？
4. 如果继续升级，我下一步最想加什么？

## 11. 参考答案与思考方向

### 关于作业 2

如果一个计划没有体现时间限制，比如用户说“三天内”，结果它输出了 10 个大阶段，那说明规划器没有真正理解约束。

如果一个计划只有“学习、阅读、练习”这种泛词，没有写清楚输出物，比如笔记、Demo、总结，那么执行性会很差。

### 关于作业 3

推荐你优先做这两个扩展：

1. 增加每日学习时长输入
2. 增加保存结果到 Markdown

因为这两个改动最容易帮你理解：

- 输入结构变化如何影响规划
- Agent 输出如何转成真实产物

### 关于作业 4

一个参考思路是：

- 它像 Agent，因为它有阶段化流程和自检机制
- 它又不像完整 Agent，因为它还没有真实环境交互能力

这正是入门项目的意义：

**先学结构，再学复杂度。**

## 12. 总结与下一步建议

这个项目不是为了做一个“很强”的 Agent，而是为了让你第一次真正把 Agent 的核心闭环跑起来。

如果你已经跑通并看懂了这份代码，下一步建议按这个顺序升级：

1. 接入真实 LLM
2. 增加文件保存能力
3. 增加工具调用
4. 增加记忆模块
5. 增加多轮反思

一句话总结：

**第一个 Agent 实战，重点不是炫技，而是把“目标、规划、执行、反思”这条主线做清楚。**
