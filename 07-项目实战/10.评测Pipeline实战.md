## 1. 这个项目是什么

这是一个可运行的**评测工程化框架**，覆盖 Prompt、RAG、Agent 三类评测场景。它不是一个评测理论讲义，而是一个真正能加载数据、运行评估、输出报告的代码项目。

## 2. 它解决什么问题

大多数 AI 项目的评测现状是："跑几个 case 看看效果，凭感觉判断好不好"。

这带来三个问题：

1. **不可重复**：换个人、换个时间，结论可能完全不同
2. **不可对比**：改了 prompt 之后"好像好了一点"，但没有量化依据
3. **不可积累**：每次都从零开始，没有评测基线

评测 Pipeline 的核心价值：**让评测可重复、可量化、可积累**。

## 3. 为什么需要评测 Pipeline

### 3.1 手动评测 vs 自动评测

| 维度 | 手动评测 | 自动评测 Pipeline |
|------|---------|-------------------|
| 速度 | 每条 1-2 分钟 | 每条 < 1 秒 |
| 一致性 | 受心情/疲劳影响 | 完全一致 |
| 可重复 | 难以复现 | 完全可复现 |
| 覆盖率 | 5-10 条就累了 | 轻松跑上百条 |
| 成本 | 人力成本高 | 一次编写反复使用 |

### 3.2 评测 Pipeline 在开发流程中的位置

```text
改代码/改 Prompt → 运行评测 Pipeline → 查看报告 → 决定是否合并
```

这和软件工程中的 CI 测试是同一个思路：**每次修改后自动验证，防止回退**。

## 4. 项目目标与最终效果

运行 `python scripts/run_prompt_eval.py`，你会看到：

1. 加载 15 条 Prompt 评测数据
2. 逐条运行 exact_match 评估器（包含匹配）
3. 如果有 API key，同时运行 LLM Judge 评估器
4. 输出终端评分表格（通过率、平均分）
5. 保存 JSON 格式详细报告

## 5. 评测集设计

### 5.1 好的评测集有什么特征

1. **覆盖典型场景**：简单/中等/困难都要有
2. **标注清晰**：reference 必须明确，不能模棱两可
3. **可扩展**：添加新 case 不需要改代码
4. **带元数据**：category、difficulty 等标签方便后续分析

### 5.2 三种评测集的设计思路

**Prompt 评测集**：覆盖总结、改写、抽取、校验等典型 Prompt 任务。每条包含 input → reference → prediction 三元组。

**RAG 评测集**：重点在检索质量。每条包含 `ground_truth_contexts`（正确文档 ID）和 `retrieved_contexts`（实际检索结果），用于计算 Recall、MRR 等指标。

**Agent 评测集**：评估行为质量。每条包含 `expected_tool_calls` 和 `actual_tool_calls`，以及步数信息，评估工具选择准确性和执行效率。

## 6. 评估器架构

### 6.1 BaseEvaluator 协议

所有评估器实现统一接口：

```python
class BaseEvaluator(Protocol):
    name: str
    def evaluate(self, prediction: str, reference: str, **kwargs) -> EvalResult: ...
```

`EvalResult` 包含三个字段：
- `score`: 0.0-1.0 的归一化分数
- `passed`: 布尔值，是否通过
- `details`: 字典，评估细节

### 6.2 为什么用 Protocol 而不是抽象基类

Protocol 是鸭子类型的静态检查：只要实现了 `name` 和 `evaluate`，就是一个合法的评估器。不需要继承，不需要注册，添加新评估器零成本。

## 7. 各评估器详解

### 7.1 ExactMatchEvaluator

三种模式：
- **exact**：完全匹配（标准化后比较）
- **contains**：prediction 包含 reference
- **keywords**：prediction 包含所有指定关键词

适用场景：有明确标准答案的场景（抽取、事实校验）。

局限：对表述灵活的场景（总结、分析）过于严格。

### 7.2 LLMJudgeEvaluator

让 LLM 充当评委打分。Prompt 要求：
1. 输出 1-5 分
2. 附带评分理由
3. 严格 JSON 格式输出

分数归一化：`normalized_score = raw_score / 5`。通过标准：≥ 4 分。

**优势**：能处理语义等价、表述变体。
**局限**：受模型质量影响，成本和延迟较高，评分可能不稳定。

### 7.3 RAGMetricsEvaluator

三个指标：

**Recall@K**：ground truth 文档在 top-K 检索结果中的命中率。
```
Recall@5 = 命中的 ground truth 数 / 总 ground truth 数
```

**MRR**：第一个正确结果的排名倒数。排名越靠前，MRR 越高。
```
MRR = 1 / (第一个正确结果的排名)
```

**Context Precision**：top-K 中相关文档的比例。衡量检索的"噪音"程度。

综合得分 = 0.4 × Recall + 0.3 × MRR + 0.3 × Precision

### 7.4 AgentMetricsEvaluator

三个维度：

**任务完成率**（权重 50%）：LLM 判断 Agent 是否完成了任务。无 API key 时回退为包含匹配。

**工具调用匹配度**（权重 30%）：比较 actual_tool_calls 和 expected_tool_calls 的 F1 分数。

**步数效率**（权重 20%）：`efficiency = min(expected_steps / actual_steps, 1.0)`。步数越少越好，但不会超过 1.0。

## 8. EvalRunner 流程

```text
评测集 (List[Dict])
    │
    ▼
EvalRunner.run(eval_set, evaluators)
    │
    ├─ 遍历每条数据
    │   ├─ 获取 prediction（使用 predict_fn 或数据中的字段）
    │   ├─ 获取 reference
    │   ├─ 调用 evaluator.evaluate()
    │   └─ 收集 EvalResult
    │
    ▼
Dict[evaluator_name, EvalSummary]
```

`predict_fn` 参数的设计意图：
- 不传：使用评测集中预填的 prediction（离线评测）
- 传入：在线调用模型生成 prediction（在线评测）

eval-pipeline/
├── app/
│   ├── config.py           # 配置中心（API key、评测参数）

**重点阅读文件：**
- `app/evaluators/base.py` — 评估器协议设计
- `app/evaluators/rag_metrics.py` — RAG 指标计算
- `app/runner.py` — 评测流程编排

## 12. 你需要重点观察什么

1. **exact_match 和 llm_judge 的差异**：同一条数据，两个评估器的评分差多大？哪个更合理？
2. **RAG 指标的信号**：Recall 高但 Precision 低说明什么？MRR 低但 Recall 高说明什么？
3. **Agent 三维度的分布**：任务完成了但工具调用不对，说明什么问题？
4. **评测集的覆盖度**：有没有哪类场景缺少评测数据？

## 13. 常见误区与失败原因

1. **只用 exact_match**：对语义等价的回答误判为失败
2. **只用 llm_judge**：成本高、慢，且可能被模型自身偏好误导
3. **评测集太少**：5 条数据的评测结论不可靠
4. **不区分评测类型**：用 Prompt 评测的方法评测 RAG，忽略了检索质量
5. **不保留历史报告**：没有基线对比，无法判断改进方向

## 14. 实战练习题

### 练习 1

运行 `run_prompt_eval.py`，对比 exact_match 和 llm_judge 在 15 条数据上的评分差异。找出评分差距最大的 3 条，分析原因。

### 练习 2

修改 `app/evaluators/llm_judge.py` 中的 prompt，让评分标准更严格（例如要求数据精确匹配才能得 5 分）。重新运行评测，对比分数变化。

### 练习 3

为 RAG 评测集新增 3 条"故意检索错误"的数据（retrieved_contexts 和 ground_truth 完全不重叠），观察 rag_metrics 的表现。

### 练习 4

实现一个新的评估器 `SemanticSimilarityEvaluator`，使用 BGE embedding 计算 prediction 和 reference 的余弦相似度作为分数。将它集成到 runner 中。

## 15. 总结与下一步建议

这个项目的核心学习点：

1. **评测是工程问题**，不是"看看效果"就行的
2. **不同场景需要不同评估器**——Prompt 看语义，RAG 看检索，Agent 看行为
3. **评测集的质量决定评测的价值**——垃圾进垃圾出
4. **LLM-as-Judge 是强大但不完美的工具**——需要和确定性指标结合使用
