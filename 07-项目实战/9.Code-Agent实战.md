## 学习目标

完成这个项目后，你应该能够：

1. 理解 Code Agent 的修复循环（plan → execute → verify）
2. 掌握代码工具的设计和沙箱安全的实现
3. 理解 LangGraph 条件路由在修复循环中的应用
4. 能够为自己的场景设计类似的自动化 Agent

---

## 1. 这个项目是什么

这是一个基于 LangGraph 构建的 **Code Agent**，能够自动读取有 bug 的代码、运行测试发现问题、制定修复计划、修改代码、验证修复结果。如果修复失败，Agent 会重新规划并再次尝试。

## 2. 它解决什么问题

"这个项目有 4 个 bug，测试全部失败。请帮我修好。"

传统 Copilot 只能建议修改，你还是得手动操作。Code Agent 直接帮你完成完整的"发现 → 诊断 → 修复 → 验证"闭环。

## 3. 为什么要用 LangGraph

Code Agent 的执行流程包含：

- **条件分支**：verify_result 后根据测试结果走不同路径
- **循环**：execute_step 节点循环执行多个计划步骤
- **多轮循环**：verify 失败后返回 plan_approach 重新规划

这些都是 LangGraph 的 StateGraph + 条件边天然擅长的模式。

## 4. 项目目标与最终效果

运行 `python scripts/run_fix_bugs.py`，你会看到：

1. Agent 列出项目文件结构
2. 分析任务类型（fix_bug）和相关文件
3. 运行测试，发现 4 个失败
4. 制定修复计划
5. 逐步执行：读取代码 → 修改 → 写入
6. 运行测试验证
7. 如果有失败，重新规划修复
8. 最终输出修复总结

## 5. 沙箱安全设计

### 三层安全控制

**路径白名单**：`Sandbox.validate_path()` 确保所有文件操作在 `data/sample_project/` 内。使用 `Path.resolve()` 防止路径穿越。

**命令白名单**：`Sandbox.validate_command()` 只允许 python, pytest, pip 等安全命令。`rm`, `curl` 等危险命令被拒绝。

**输出截断**：`Sandbox.truncate_output()` 限制命令输出 2000 字符，防止大输出撑爆 LLM context。

## 6. 核心流程拆解

### 图拓扑

```text
START → analyze_task → plan_approach → execute_step ─┐
                                          ↑          │ (循环执行计划步骤)
                                          └──────────┘
                                                     │
                                              verify_result ─┬─ [通过] → summarize → END
                                                             └─ [失败] → plan_approach
```

### 节点详解

**analyze_task**：LLM 分析任务类型和相关文件。输出：task_type, relevant_files。同时读取相关代码到 state。

**plan_approach**：LLM 根据代码和测试结果制定修复计划。输出：diagnosis, plan（步骤列表）。

**execute_step**：LLM 选择工具执行当前步骤。支持 read_file, write_file, run_command, search_code, list_dir。通过条件边循环执行所有步骤。

**verify_result**：运行 pytest 验证。LLM 分析测试结果，判断是否全部通过。

**summarize**：生成修复报告。

### 条件路由

**execute_step 后**：
- 还有未执行的步骤 → 继续 execute_step
- 所有步骤完成 → verify_result

**verify_result 后**：
- 测试全通过 → summarize
- 测试失败 + 未超限 → plan_approach（重新规划）
- 达到上限 → summarize（强制结束）

## 7. Code Agent 的核心能力

1. **项目级理解**：不只看一个函数，而是理解文件间的依赖
2. **测试驱动**：以测试通过为目标，不是"看起来对了就行"
3. **多轮修复**：一次修不好就再来，自动收敛
4. **工具组合**：灵活组合 5 种工具完成复杂操作
5. **安全执行**：沙箱隔离，不影响项目外的文件

## 8. Code Agent 的核心难点

1. **代码修改精度**：Agent 倾向于重写整个文件而非最小修改
2. **错误定位**：测试失败的 traceback 需要准确解析
3. **多 bug 耦合**：修一个 bug 可能暴露或引入另一个
4. **上下文限制**：大项目无法全部放入 context
5. **编程语言差异**：不同语言的测试框架和错误格式不同

## 9. 项目代码结构

```text
agent-code-assistant/
├── app/
│   ├── config.py           # 配置（沙箱路径、安全限制、Agent 参数）
```

**重点阅读文件：**
- `app/sandbox.py` — 安全设计的核心
- `app/nodes.py` — Agent 的决策逻辑
- `app/graph.py` — 条件路由的配置

## 11. 你需要重点观察什么

1. **诊断准确性**：Agent 对 4 个 bug 的诊断是否准确？
2. **计划合理性**：修复计划的步骤是否合理有序？
3. **工具选择**：Agent 是否正确选择了工具（先 read 再 write）？
4. **循环收敛**：几轮修复后测试是否全部通过？
5. **修改最小性**：Agent 是否只修改了有 bug 的部分？

## 12. 常见误区与失败原因

1. **Agent 重写整个文件**：没有精确定位 bug，而是用"我认为正确的版本"覆盖 → prompt 需要强调"最小修改"
2. **修复引入新 bug**：改了 divide 的逻辑但破坏了正常除法 → verify 循环的价值
3. **测试输出太长**：pytest 的详细输出超过截断限制 → 关键信息被截掉
4. **诊断不准**：Agent 误判 bug 原因 → 需要更多代码上下文
5. **不运行测试就声称修好了**：Agent 跳过 verify → 强制流程不可跳过

## 13. 实战练习题

### 练习 1

运行 `run_fix_bugs.py`，记录 Agent 修复 4 个 bug 的完整轨迹。对于每个 bug，记录：
- 诊断是否准确
- 修复方案是否合理
- 几轮修复后通过

### 练习 2

在 `calculator.py` 中新增第 5 个 bug：`max_value` 函数对包含负数的列表返回错误结果（如错误地忽略负数）。添加对应测试用例。运行 Agent 观察修复过程。

### 练习 3

修改 `app/config.py`，将 `max_fix_rounds` 设为 1，强制 Agent 只能修复一次。对比与默认（3 轮）的修复成功率差异。

### 练习 4

为 Code Agent 添加一个新工具 `diff_file(path, original, modified)`，让 Agent 能看到修改前后的差异。思考这对修复质量有什么帮助。

## 14. 参考答案与思考方向

### 练习 1 思考方向

- Bug 1（除零）和 Bug 3（空列表）相对容易，通常 1 轮修复
- Bug 2（类型检查）需要 Agent 理解"字符串拼接不是加法"
- Bug 4（基数为零）需要 Agent 理解业务语义
- 如果所有 bug 都在第一轮修好，说明 Agent 的分析能力不错

### 练习 2 思考方向

- 可以让 `max_value` 错误地过滤掉负数：`if n > 0 and n > result`
- 测试用例：`assert max_value([-5, -3, -1]) == -1`
- Agent 需要理解"最大值"的数学定义，不只是找正数

### 练习 3 思考方向

- 1 轮限制下，Agent 必须在一次计划中覆盖所有 bug
- 如果分析不全面，可能只修了 2-3 个
- 这证明了多轮修复循环的价值

### 练习 4 思考方向

- diff 工具让 Agent 能验证"我改了什么"
- 可以用 Python 的 difflib 实现
- 在 verify 失败时，Agent 可以查看 diff 决定是回滚还是继续修改

## 15. 总结与下一步建议

这个项目的核心学习点：

1. **Code Agent 的核心是"读-改-验"循环**，不是一次性生成代码
2. **沙箱安全是硬性要求**——有代码执行能力的 Agent 必须有安全控制
3. **测试是最好的验证**——比 LLM 自己判断"改好了"可靠得多
4. **条件路由实现修复循环**——LangGraph 让循环逻辑声明式、可控
