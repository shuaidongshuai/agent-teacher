# 角色、目标、约束、输出格式

## 学习目标

这篇文档适合已经知道 Prompt 会影响结果，但还不知道“该从哪些维度把任务写完整”的学习者。

学完后你应该能够：

1. 用一套稳定框架写 Prompt，而不是每次临场发挥
2. 理解角色、目标、约束、输出格式分别在解决什么问题
3. 识别为什么有些 Prompt 看起来很长，却依然效果差
4. 把模糊需求改写成更容易执行的任务说明

## 1. 为什么需要这个框架

很多 Prompt 失败，不是缺少某个技巧，而是缺少结构。

常见情况是：

- 说了很多背景，但没说清目标
- 提了目标，但没有限制条件
- 写了限制，但没规定输出格式
- 让模型“像专家一样回答”，却没说明面向谁

“角色、目标、约束、输出格式”这四部分，正好对应四类常见问题。

---

## 2. 角色在解决什么问题

角色主要解决的是：

**模型应该以什么视角、什么风格、什么能力边界来完成任务。**

例如：

- 面向初学者的老师
- 严谨的代码审阅者
- 金融研究助理

角色的作用通常体现在：

- 术语深浅
- 表达风格
- 关注重点

但要注意，角色不是越夸张越好。像“世界顶级、全宇宙最强专家”这类说法，通常没有实质帮助。

---

## 3. 目标在解决什么问题

目标是 Prompt 里最核心的一部分。

它回答的是：

**你到底要模型产出什么结果。**

一个好的目标应该尽量具体，包括：

- 任务类型
- 完成标准
- 面向对象
- 范围边界

例如下面两句相比：

- “解释一下 RAG”
- “面向零基础产品经理，用 300 字解释什么是 RAG，并说明它和微调的区别”

第二句显然更容易得到可用结果。

---

## 4. 约束在解决什么问题

约束的价值在于防止模型“会做太多”。

常见约束包括：

- 只能依据提供资料回答
- 不要编造数字和引用
- 不使用过深术语
- 控制篇幅
- 不输出代码，只输出思路

如果没有约束，模型往往会朝“看起来完整”而不是“真正符合需求”的方向生成。

所以很多时候，约束不是在削弱模型，而是在减少失控空间。

---

## 5. 输出格式在解决什么问题

输出格式解决的是“结果怎么被消费”。

很多人只关注内容本身，但忽略了结果还要被：

- 人阅读
- 程序解析
- 系统评测
- 下游流程继续处理

常见格式包括：

- 要点列表
- 表格
- JSON
- 固定字段段落

一个没有格式要求的 Prompt，即使回答内容不错，也可能很难复用。

---

## 6. 一个完整例子

下面看一个从模糊到清晰的改写过程。

原始需求：

“帮我分析这份用户反馈。”

改写后：

“你是一名 SaaS 产品分析助理。请阅读下面的用户反馈，完成三件事：  
1. 提炼 3 个高频问题；  
2. 判断每个问题更接近功能缺失、使用门槛还是稳定性问题；  
3. 给出 3 条产品优化建议。  
要求只基于提供内容判断，不要编造用户背景。  
输出格式为 Markdown 表格，列为：问题、分类、证据、建议。”

这段 Prompt 更强，不是因为它更长，而是因为它补齐了：

- 角色
- 目标
- 约束
- 输出格式

## 7. 常见误区

### 7.1 误区一：角色最重要

不是。角色通常只是辅助项，目标和约束往往更关键。

### 7.2 误区二：目标写一句就够了

如果任务稍复杂，一句目标常常不足以防止歧义。

### 7.3 误区三：输出格式只是美化

不是。格式会直接影响可解析性、一致性和后续自动化能力。

---

## 8. 实战建议

### 8.1 先写目标，再补角色

如果连任务目标都不清楚，角色写得再多也没用。

### 8.2 约束要具体，可判断

“尽量准确”这种话太虚，最好换成“不要使用未提供的数据”这类可执行限制。

### 8.3 输出格式尽量提前约定

尤其是要对接程序时，不要事后再补格式要求。

### 8.4 避免冲突指令

例如同时要求“尽量详细”和“控制在 100 字以内”，模型就会左右为难。

## 9. 总结与下一步

这篇文档最重要的一句话是：

**一个好 Prompt 往往不是“更会说”，而是把角色、目标、约束和输出格式补完整。**
