## 1. 什么是结构化输出

结构化输出指的是：

**不是让模型自由发挥，而是让它按预定义格式返回结果。**

这个格式可以是：

- 固定字段段落
- Markdown 表格
- JSON
- 列表对象

它的目的不是让答案看起来更整齐，而是让结果更容易被复用。

---

## 2. 它解决什么问题

如果没有结构化输出，常见问题包括：

- 每次字段顺序不同
- 漏掉关键信息
- 很难被程序解析
- 很难自动评测
- 下游流程无法稳定衔接

所以结构化输出解决的是：

**模型结果如何从“能看”变成“能接入系统”。**

---

## 3. 常见结构形式

### 3.1 固定标题段落

例如要求输出成：

- 问题
- 原因
- 建议

适合人工阅读优先的场景。

### 3.2 Markdown 表格

适合做对比、汇总和简单展示。

### 3.3 JSON

适合程序解析和自动化处理。

例如：

```json
{
  "topic": "RAG",
  "difficulty": "beginner",
  "key_points": ["降低幻觉", "引入外部知识", "依赖检索质量"]
}
```

### 3.4 列表对象

适合多条记录抽取，例如事件列表、问题列表、实体列表。

---

## 4. 怎样写更稳的格式要求

### 4.1 明确字段名

不要只说“请用 JSON 输出”，最好写明需要哪些字段。

### 4.2 说明字段含义

否则模型可能猜错字段用途。

### 4.3 说明字段类型

例如字符串、数组、布尔值、整数。

### 4.4 说明缺失值如何处理

例如：

- 如果没有相关信息，返回 `null`
- 如果无法判断，返回 `"unknown"`

### 4.5 尽量减少无关自然语言

如果前后再混很多解释文字，程序解析就容易出问题。

---

## 5. 常见失败原因

### 5.1 字段要求太模糊

模型不知道哪些字段必须有，哪些可以省略。

### 5.2 同时追求自然表达和严格结构

这两个目标有时会互相冲突。

### 5.3 没有定义异常情况

真实数据不总是完整的，如果没定义缺失处理，模型会开始自由发挥。

### 5.4 输出结构太复杂

层级过深、规则过多时，稳定性可能下降。

---

## 6. 实战建议

### 6.1 先从简单结构开始

不要一开始就设计非常复杂的 schema。

### 6.2 把“必填字段”写清楚

这会显著减少字段丢失。

### 6.3 用失败样本做回归测试

看哪些输入最容易造成字段漂移或解析失败。

### 6.4 将结构要求和内容要求分开写

这样模型更容易同时理解“写什么”和“怎么写”。

明确字段，例如 `issue_type`、`severity`、`evidence`、`suggestion`，并说明字段类型与缺失处理方式。

“如果结果最终要进入程序链路，结构化输出通常比纯自然语言更稳。”

参考答案：

正确。
