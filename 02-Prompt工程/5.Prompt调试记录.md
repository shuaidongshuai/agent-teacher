## 1. 为什么 Prompt 调试很重要

很多人写 Prompt 时最常见的状态是：

- 这次结果不错
- 下次类似输入又坏了
- 改了一句之后似乎好了
- 但不知道为什么好了

这种方式的问题在于：

你得不到可迁移经验，也无法保证稳定性。

所以 Prompt 调试的目标不是“碰巧试出一个能用版本”，而是：

**知道问题为什么出现，以及这次修改到底在修什么。**

---

## 2. 常见失败类型

### 2.1 任务理解错误

模型从一开始就理解错了要做什么。

### 2.2 约束失效

Prompt 明明写了“不要编造”，模型还是补了很多没提供的信息。

### 2.3 输出格式漂移

有时是 JSON，有时又夹杂解释文字。

### 2.4 边界样本失败

典型输入没问题，但稍复杂一点就崩。

### 2.5 上下文不足或噪声过多

这类问题看起来像 Prompt 不好，实际上可能是信息给错了。

---

## 3. 一个实用调试流程

### 3.1 先固定测试样本

不要每次都换输入，否则你根本不知道变化来自哪里。

### 3.2 明确失败定义

例如：

- 分类错了
- 漏字段
- 超出字数
- 使用了未提供事实

### 3.3 一次只改一个主要变量

例如先只改约束，不同时改角色、示例、格式和参数。

### 3.4 记录修改前后差异

包括：

- 改了哪一段 Prompt
- 预期修复什么问题
- 实际结果是否改善

### 3.5 把失败样本沉淀成回归集

下次改 Prompt 时，再跑一遍这些难例。

## 4. 调试示例

假设任务是“从客服对话中提取退款原因”，但模型经常输出不在原文里的猜测。

第一版 Prompt：

“请阅读下面对话，提取用户退款原因。”

问题：

- 目标太简略
- 没限制只能引用原文信息
- 没规定输出字段

第二版 Prompt：

“请阅读下面客服对话，提取用户明确表达的退款原因。  
如果原文没有明确原因，不要猜测。  
输出 JSON，包含 `has_reason` 和 `reason` 两个字段。  
如果无法判断，返回 `has_reason=false`，`reason=null`。”

这次修改修复的是：

- 限制猜测
- 约定缺失处理
- 提高可解析性

这个例子说明，很多所谓“模型幻觉”问题，实际上可以先从任务表达和输出规则层面减轻。

## 5. 常见误区

### 5.1 误区一：结果不好就不断加字

有时问题不在“信息不够”，而在“结构不清”。

### 5.2 误区二：一次改很多地方更快

短期也许更快，长期会让你失去因果判断。

### 5.3 误区三：只看成功样本

真正有价值的是那些最容易失败的输入。

---

## 6. 实战建议

### 6.1 建一个失败案例库

哪怕只有十几个样本，也会比纯靠感觉调 Prompt 强很多。

### 6.2 先排除参数和上下文问题

有时输出不稳，并不是 Prompt 本身，而是采样太发散或输入材料不足。

### 6.3 把修改目标写成一句话

例如：

“这次修改只为了减少编造，不追求文风优化。”

### 6.4 关注稳定性，而不是单次惊艳结果

工程里更重要的是平均表现和可预期性。

## 7. 练习题

### 7.1 思考题

为什么 Prompt 调试时要尽量固定测试样本？

参考答案：

因为只有固定输入，才能更清楚判断输出变化到底是不是由 Prompt 修改带来的。

### 7.2 判断题

“如果一个 Prompt 在 2 个样本上表现很好，就说明它已经足够稳定。”

参考答案：

错误。它可能只是碰巧覆盖了简单样本，没有经过边界情况验证。
