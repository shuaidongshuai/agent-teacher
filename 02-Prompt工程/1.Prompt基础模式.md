## 1. 什么是 Prompt 工程

Prompt 工程可以简单理解成：

**为了让模型更稳定地完成任务，而进行的任务表达、上下文组织和输出约束设计。**

它不是单纯“写一句提示词”，而是回答这些问题：

- 模型现在到底要做什么
- 它需要知道哪些背景信息
- 它不能做什么
- 它应该输出成什么样

所以 Prompt 工程更像“给模型下任务”，而不是“和模型聊天”。

---

## 2. 它解决什么问题

在真实使用里，很多失败不是因为模型完全不会，而是因为任务没有被说清楚。

常见问题包括：

- 模型理解错任务
- 输出太散，不可用
- 回答漏条件
- 格式不统一
- 遇到边界情况就跑偏

Prompt 工程的目标不是让模型无所不能，而是：

**在它本来有能力完成的范围内，让结果更稳定、更接近需求。**

---

## 3. Prompt 为什么会影响结果

从底层上说，模型是根据当前上下文预测接下来的 token。

这意味着 Prompt 一变，模型看到的条件也会变。

变化可能来自：

- 任务目标更明确了
- 重点信息被提前强调
- 输出格式被限制住了
- 例子暗示了你想要的风格

所以 Prompt 工程不是魔法，它本质上是在改变模型的输入条件。

## 4. 几种基础模式

下面这些模式不是互斥的，而是经常组合使用。

### 4.1 直接指令模式

最基础的写法，就是直接告诉模型要做什么。

例如：

“请把下面这段文章总结成 3 条要点。”

适合：

- 简单任务
- 一次性提问
- 不要求特别稳定的场景

问题是：

如果任务稍微复杂，直接指令常常不够。

### 4.2 角色设定模式

通过给模型一个工作身份，帮助它进入特定任务视角。

例如：

“你是一名面向初学者的 Python 讲师，请解释递归。”

角色设定的价值不在于“演戏”，而在于：

- 影响语言风格
- 影响解释深度
- 影响优先级判断

### 4.3 分步执行模式

把复杂任务拆成更清晰的步骤。

例如：

1. 先识别用户问题属于哪一类
2. 再提取关键约束
3. 最后输出建议

这种模式能减少模型一下子处理太多要求时的混乱。

### 4.4 约束模式

明确告诉模型什么能做，什么不能做。

例如：

- 只能基于给定资料回答
- 不要编造未提供的数据
- 输出控制在 200 字以内

约束越清楚，越有助于减少跑偏。

### 4.5 格式约束模式

不是只要求“回答正确”，还要求“回答可用”。

例如：

- 用 JSON 输出
- 用表格输出
- 固定成“问题-分析-建议”三段

这对后续程序处理和人工审核都很重要。

## 5. 常见误区

### 5.1 误区一：Prompt 工程就是堆更多字

不是。更长不一定更好，关键是更清楚、更有结构。

### 5.2 误区二：Prompt 只适合聊天场景

不是。分类、抽取、改写、规划、工具调用前置判断，都需要 Prompt 设计。

### 5.3 误区三：找到一个好 Prompt 就一劳永逸

不现实。任务、模型版本、上下文长度、样本类型变化后，原来的 Prompt 可能失效。

---

## 6. 实战建议

### 6.1 先说任务，再补背景，再说格式

这是一个很稳的基础顺序。

### 6.2 优先减少歧义

如果一句话可能被理解成两种意思，就尽量改写。

### 6.3 一个 Prompt 只解决一个主要目标

不要把太多不同目标堆在一起，否则模型容易顾此失彼。

### 6.4 记录失败案例

真正提升 Prompt 能力，不是只看成功样本，而是研究它为什么失败。

## 7. 练习题

### 7.1 改写题

把下面这句需求改写得更适合模型执行：

“帮我处理一下这个内容。”

参考思路：

至少补清楚任务类型、目标、限制和输出形式。例如是总结、改写、提取还是分类。

### 7.2 判断题

“Prompt 工程的重点是找到一套神秘固定模板，之后所有任务都照抄。”

参考答案：

错误。Prompt 工程更重要的是理解任务结构，并根据场景调整表达方式。
