# RAG 系统学习讲义

这份文档是给“想系统学会 RAG”的学习者准备的。

你不需要一开始就会向量数据库、重排模型、复杂框架。先抓住核心问题：

**为什么大模型明明很强，却还是需要 RAG？**

答案很简单：

- 大模型参数里不一定有你需要的最新知识
- 即使见过相关知识，也不一定能稳定回忆出来
- 在企业应用里，真正重要的信息往往在你自己的文档里，而不是模型训练语料里

RAG 的核心作用，就是：

**先找到相关资料，再让模型基于资料回答。**

## 1. 什么是 RAG

RAG 全称是 `Retrieval-Augmented Generation`，中文通常叫“检索增强生成”。

它不是单独一个模型，而是一种应用架构。你可以把它理解成两段式流程：

1. `Retrieval`：先从资料库里检索相关内容
2. `Generation`：再把检索结果连同用户问题一起交给大模型生成答案

一个最基础的 RAG 系统，通常会处理这类问题：

- “根据公司制度文档，年假怎么计算？”
- “这个产品说明书里提到的电池容量是多少？”
- “请总结这 20 篇内部会议记录里关于预算的结论”

这些问题的共同点是：

- 答案依赖外部资料
- 外部资料可能很多
- 需要模型帮你读、筛、总结、组织语言

## 2. RAG 解决了什么问题

### 2.1 降低幻觉

如果没有外部资料，模型只能“靠记忆”回答。它很容易把看起来合理但实际不准确的内容说出来。

加入 RAG 后，模型会先看到真实参考资料，再回答，幻觉风险通常会下降。

注意：

**RAG 不是消灭幻觉，而是降低幻觉。**

如果检索错了，或者检索到了不完整内容，模型照样可能答错。

### 2.2 引入私有知识

企业最有价值的信息，常常不在公开互联网，而在：

- 内部 Wiki
- 产品文档
- 客服知识库
- 合同与制度文件
- 会议纪要

RAG 让模型能“临时读取”这些知识，而不需要重新训练一个模型。

### 2.3 知识可更新

模型训练完成后，参数知识是相对固定的。RAG 的资料库可以随时更新。

这意味着：

- 新增文档后，系统可快速纳入新知识
- 不需要为了新文档重新训练模型
- 更适合知识经常变化的业务场景

## 3. RAG 的完整流程

可以把一个 RAG 系统理解成下面 6 步：

1. 文档收集
2. 文档预处理
3. 文档切分
4. 向量化与索引
5. 检索与重排
6. 把上下文交给模型生成答案

### 3.1 文档收集

首先要明确你的知识从哪里来，例如：

- Markdown
- PDF
- Word
- 网页
- 数据库记录
- FAQ 表格

这一步看起来简单，实际非常关键。因为很多 RAG 问题不是模型问题，而是源数据问题。

常见原始数据问题：

- 文档过期
- 格式混乱
- 重复内容多
- 标题缺失
- 表格抽取错误

### 3.2 文档预处理

在真正做检索之前，通常会做一些清洗：

- 去除无意义页眉页脚
- 修正文档编码问题
- 标记标题层级
- 识别表格、代码块、列表
- 给文档补充元数据

这里的“元数据”很重要，比如：

- 文档标题
- 来源
- 作者
- 创建时间
- 版本号
- 部门
- 标签

后面你会发现，很多检索优化不是靠“换更强模型”，而是靠“把元数据用好”。

### 3.3 文档切分

大模型一次不能无限读取所有文档，所以要把长文档切成更小的片段，这些片段通常叫 `chunk`。

切分的目的是：

- 让检索粒度更细
- 提高召回精度
- 控制上下文长度与成本

但切分不是越小越好。

如果切得太小：

- 信息可能不完整
- 上下文断裂
- 检索命中后也看不懂

如果切得太大：

- 检索会不够精准
- 无关信息太多
- 上下文窗口浪费严重

一个朴素但常见的做法是：

- 按段落切分
- 控制每块在一定 token 范围内
- 保留少量重叠区域

这个重叠区域通常叫 `overlap`，作用是减少被切断的信息损失。

### 3.4 向量化与索引

切分后的文本需要转换成可检索的表示。最常见的方法是生成 `embedding`，也就是向量表示。

直觉上，你可以这样理解：

- 语义相近的文本，向量位置通常更接近
- 用户问题也会被转成向量
- 系统通过比较距离，找出最相关的文本块

这一步通常包括：

- 为每个 chunk 生成 embedding
- 存入向量数据库或索引系统
- 保存原文与元数据

常见存储方式包括：

- 向量数据库
- 本地向量索引
- 混合检索系统

### 3.5 检索与重排

用户提问后，RAG 系统不会直接回答，而是先检索。

最常见的检索方式有两类：

- 关键词检索：更适合精确词项、专有名词、编号
- 向量检索：更适合语义相似和自然语言表达

真实系统里，很多效果更好的方案是：

**混合检索 = 关键词检索 + 向量检索**

拿到一批候选结果后，通常还会做 `rerank`，也就是重排。它的作用是：

- 从“粗召回”的候选里挑出最相关的少量内容
- 把真正跟问题最相关的片段排到前面

你可以把流程理解成：

- 检索负责“先多找一些”
- 重排负责“再精挑细选”

### 3.6 生成答案

系统把下面几部分一起给模型：

- 用户问题
- 检索到的相关文本块
- 回答规则
- 输出格式要求

然后模型根据这些上下文生成最终答案。

一个很关键的设计是：

**要让模型知道：它只能依据给定资料回答。**

否则模型会在资料不够时自动脑补。

## 4. 你可以怎样理解 RAG 的本质

RAG 的本质不是“往 Prompt 里塞资料”。

更准确地说，它是一个信息供给系统，负责回答下面两个问题：

1. 用户当前这个问题，最需要哪些信息？
2. 这些信息怎样组织后最适合模型理解？

所以一个 RAG 项目的效果，往往取决于三件事：

- 找得准不准
- 给得全不全
- 组织得好不好

这三件事分别对应：

- 检索质量
- 上下文质量
- 提示与生成质量

## 5. RAG 的几个关键模块

## 5.1 Chunk 设计

Chunk 是 RAG 的基本单位。很多初学者一上来就关注模型，其实 chunk 质量经常更影响效果。

好的 chunk 通常具备这些特点：

- 语义相对完整
- 不要混入太多无关内容
- 能保留标题和上下文线索
- 尽量可独立理解

举个简单例子：

如果一篇“员工报销制度”文档里有以下结构：

- 第一章：适用范围
- 第二章：交通报销标准
- 第三章：住宿报销标准

那么比起机械地每 500 个字切一刀，更好的方式通常是：

- 按标题与语义结构切
- 每个 chunk 保留章节信息

这样用户问“住宿报销上限是多少”时，系统更容易找到真正相关内容。

## 5.2 Embedding

Embedding 模型的作用，是把文本映射成向量。

你只需要先掌握两个直觉：

- 它不是在理解“真理”，而是在表达“语义相似性”
- 它的质量会直接影响召回质量

常见误区：

- 以为 embedding 模型越大越一定更好
- 忽略语言类型和领域适配
- 文本预处理很差，却怪检索模型不行

## 5.3 Retriever

Retriever 是负责“找资料”的部分。

常见策略：

- Top-K 检索：先找最相似的 K 个 chunk
- Metadata Filter：先按部门、时间、文档类型过滤
- Multi-query：把一个问题改写成多个查询再检索
- Hybrid Search：关键词 + 向量一起查

如果你的数据里有很多：

- 型号
- 版本号
- 日期
- 编号
- 固定术语

那么不要只依赖向量检索，关键词检索通常很重要。

## 5.4 Reranker

重排模型通常会对“问题-文档块”做更精细的相关性判断。

它比初筛检索更贵，但更准。

如果你发现系统“能找到相关文档，但最终给模型的片段排序不好”，那就很可能需要重排。

## 5.5 Prompt 设计

很多人以为 RAG 的重点只在检索，其实生成阶段的 Prompt 也很重要。

典型规则包括：

- 仅根据提供资料回答
- 如果资料不足，明确说不知道
- 引用来源或片段编号
- 优先回答用户问题，不要泛泛而谈

一个常见问题是：

检索结果其实已经够了，但 Prompt 没有约束模型，导致它“半引用半脑补”。

## 6. RAG 为什么会失败

学习 RAG 时，很重要的一点不是只看成功案例，而是学会分析失败原因。

一个答案答错，常见原因往往在下面几层：

### 6.1 没检索到

原因可能是：

- chunk 切分不合理
- embedding 表达效果一般
- query 写法和文档表达差异太大
- 过滤条件过严

### 6.2 检索到了，但排序不好

原因可能是：

- Top-K 过小，漏掉关键信息
- Top-K 过大，引入大量噪音
- 没有 rerank
- 关键词命中但语义不相关

### 6.3 检索内容不完整

原因可能是：

- 一个答案分散在多个 chunk 中
- chunk 切断了关键条件
- 只检索正文，漏了表格或附录

### 6.4 模型看到了资料，但还是答错

原因可能是：

- Prompt 约束不清
- 上下文过长，关键信息被淹没
- 多段资料之间互相冲突
- 模型没有被要求引用证据

### 6.5 数据源本身有问题

原因可能是：

- 原始资料过期
- 多个版本冲突
- OCR 抽取错误
- 文档标题和内容不匹配

这也是为什么做 RAG 时一定要记住一句话：

**RAG 首先是数据工程问题，其次才是模型工程问题。**

## 7. RAG 的常见优化方向

### 7.1 优化切分策略

你可以尝试：

- 固定长度切分
- 按段落切分
- 按标题层级切分
- 语义切分

然后比较哪种方式更适合你的文档类型。

### 7.2 优化检索查询

用户的问题不一定适合直接拿去检索。

可以做的增强包括：

- 查询改写
- 查询扩展
- 把复杂问题拆成多个子问题
- 把口语化问题改成文档风格表达

### 7.3 使用混合检索

如果你的知识库既有自然语言说明，又有大量编号和术语，混合检索通常很值得尝试。

### 7.4 增加重排

如果初步召回没问题，但最终送给模型的文档经常不是最相关的，就可以考虑重排。

### 7.5 做引用和可追溯

一个实用优化不是让答案“更像人说的话”，而是让答案“更容易验证”。

例如：

- 给出引用文档名
- 标明段落编号
- 返回证据片段

这样用户更容易信任系统，也更容易排查错误。

### 7.6 区分“检索型问题”和“生成型问题”

不是所有问题都适合直接套一个 RAG。

例如：

- “请严格依据制度说明差旅标准”是强检索型
- “请帮我写一封更有礼貌的催款邮件”是偏生成型

如果把生成型任务也强行塞很多检索材料，反而会让输出变差。

## 8. 一个最小可行 RAG 系统长什么样

你可以把 MVP 想成下面这样：

1. 准备 10 到 50 篇 Markdown 或 PDF 文档
2. 把文档转成纯文本
3. 按段落或固定长度切分
4. 为每个 chunk 生成 embedding
5. 用户提问时做相似度检索
6. 取最相关的几个 chunk 放进 Prompt
7. 让模型根据资料回答

如果你第一次学习 RAG，建议先做到这一步，不要一开始就追求：

- 多路召回
- 复杂工作流
- 自动查询改写
- 多 Agent
- 长链路编排

先把最小系统跑通，你会对每个模块为什么存在有更强的感知。

## 9. 学习 RAG 时最该关注的指标

很多初学者只看一个指标：

- “回答看起来像不像对的”

这不够。更建议你分层看：

### 9.1 检索层

- 有没有召回到正确 chunk
- 正确 chunk 排在第几位
- 噪音内容有多少

### 9.2 生成层

- 是否真正基于资料回答
- 是否遗漏关键条件
- 是否存在脑补
- 是否引用了错误片段

### 9.3 系统层

- 延迟是否可接受
- 成本是否过高
- 新文档能否快速接入
- 出错时是否方便排查

如果你只看最终答案，很难知道问题出在检索还是生成。

## 10. RAG 与微调的区别

这是一个特别常见的问题。

### RAG 更适合：

- 知识经常变化
- 需要接入私有文档
- 希望答案可追溯
- 不想频繁训练模型

### 微调更适合：

- 希望模型稳定学习某种输出风格
- 希望模型掌握固定任务格式
- 希望降低特定任务的提示复杂度

一个直观理解是：

- RAG 更像“开卷考试”
- 微调更像“把某种能力训练进肌肉记忆”

很多应用里，这两者不是互斥关系，而是可以结合。

## 11. 初学者常见误区

### 误区 1：以为 RAG 就是“上传文件然后问问题”

那只是表面形式。真正决定效果的是：

- 文档质量
- 切分策略
- 检索策略
- 上下文组织方式

### 误区 2：回答差就立刻换更强模型

很多时候问题不在生成模型，而在：

- 没召回
- 召回不准
- 资料脏
- Prompt 不清

### 误区 3：只做 Demo，不做评测

如果没有一组固定测试问题，你很难知道系统是真的变好了，还是只是“碰巧这次答对了”。

### 误区 4：把所有资料一股脑塞进上下文

上下文不是越多越好。太多资料会带来：

- 成本上升
- 关键证据被淹没
- 模型更容易跑偏

## 12. 推荐学习顺序

如果你是第一次系统学 RAG，建议按这个顺序：

### 第一步：先理解整体流程

先搞清楚：

- 文档从哪里来
- 为什么要切分
- 为什么要 embedding
- 为什么要检索
- 为什么还要 Prompt

这时不要急着纠结某个库的写法。

### 第二步：亲手做一个最小 Demo

哪怕只有 20 篇文档，也值得做。

你需要亲眼看到：

- 检索命中了什么
- 为什么答错
- 调整 chunk 后有什么变化

### 第三步：开始做失败分析

把错误分成几类：

- 没找到
- 找错了
- 找到了但答错了
- 答案不完整

这一步会让你真正理解 RAG。

### 第四步：再学习优化策略

等你已经看过失败案例，再去学：

- 混合检索
- 重排
- 查询改写
- 元数据过滤

你会更明白它们分别解决什么问题。

## 13. 一个帮助记忆的类比

你可以把 RAG 想成一次“带资料回答问题”的过程：

- 知识库像书架
- 检索像找书和翻页
- 重排像把最相关的几页先摊开
- Prompt 像老师告诉你“只能根据材料作答”
- 大模型像负责阅读、理解、组织答案的人

如果书找错了，回答很难对。
如果书找对了但摊开的页不对，回答也会偏。
如果材料对了，但答题要求不清，仍然可能失分。

## 14. 自学练习题

你可以用下面这些问题检验自己是否真的学懂了。

### 概念题

1. RAG 和直接问大模型的本质差别是什么？
2. 为什么切分太大和切分太小都不好？
3. 为什么检索到了相关内容，模型还是可能答错？
4. 混合检索为什么常常比单一向量检索效果更稳？
5. 为什么说 RAG 首先是数据工程问题？

### 概念题标准答案

#### 1. RAG 和直接问大模型的本质差别是什么？

标准答案：

直接问大模型，主要依赖的是模型参数中已经学到的知识和当前输入里的提示信息；而 RAG 会在回答前先去外部知识库检索相关资料，再基于这些资料生成答案。  
所以本质差别在于：

- 直接问模型，更依赖模型“记忆”
- RAG，更依赖系统“先找资料再回答”

进一步说，RAG 让回答过程从“闭卷作答”变成“开卷作答”，因此更适合知识经常变化、需要私有知识、需要可追溯证据的场景。

#### 2. 为什么切分太大和切分太小都不好？

标准答案：

切分太大时，一个 chunk 里会混入很多无关内容，检索不够精准，而且会浪费上下文窗口；切分太小时，原本应该连在一起的信息会被拆散，导致检索到的片段不完整，模型即使拿到了片段也可能理解不全。  
因此 chunk 设计要在“语义完整”和“检索精度”之间平衡。

你可以这样记：

- 太大：找得不准
- 太小：信息不全

#### 3. 为什么检索到了相关内容，模型还是可能答错？

标准答案：

因为“检索到相关内容”不等于“模型一定正确使用了这些内容”。模型仍然可能答错，常见原因包括：

- 检索结果排序不好，真正关键的证据没有排在前面
- 检索到的内容不完整，少了限定条件
- 上下文太长，关键信息被淹没
- Prompt 没有限制模型只能依据资料回答
- 多个片段之间存在冲突，模型选错了依据

所以一个 RAG 系统的错误，可能发生在检索后半段，也可能发生在生成阶段，而不是只看“有没有召回相关文档”。

#### 4. 混合检索为什么常常比单一向量检索效果更稳？

标准答案：

因为关键词检索和向量检索各有优势，混合检索能互补。

- 关键词检索擅长匹配精确词项，比如型号、编号、专有名词、日期
- 向量检索擅长理解语义相似，适合自然语言表达不一致的情况

如果只用向量检索，可能会漏掉一些必须精确匹配的关键信息；如果只用关键词检索，又可能错过语义上相关但措辞不同的内容。混合检索更稳，是因为它同时照顾了“精确匹配”和“语义匹配”两种需求。

#### 5. 为什么说 RAG 首先是数据工程问题？

标准答案：

因为 RAG 的效果很大程度上取决于数据链路是否健康，而不只是模型强不强。  
如果原始文档过期、抽取错误、切分不合理、元数据缺失、索引混乱，那么后面的检索和生成都会受到影响。

也就是说，RAG 的前半段在解决：

- 数据从哪里来
- 数据是否干净
- 数据如何组织
- 数据能否被正确检索

只有这些问题处理得足够好，模型后面的生成才有可靠基础。所以在真实项目中，很多 RAG 问题最终都不是靠“换一个更强模型”解决的，而是靠清洗数据、优化 chunk、改进索引和调整检索策略解决的。

### 实战题

1. 用 10 篇自己的笔记搭一个最小 RAG 系统
2. 人工准备 20 个测试问题，检查能否召回正确片段
3. 比较两种切分策略的表现
4. 观察加入引用后，答案可信度是否提升

## 16. 一句话总结

RAG 的本质，是让模型在回答前先拿到更相关、更可信的外部信息。

真正决定效果的，不只是模型强不强，而是：

- 你的资料干不干净
- 你的切分合不合理
- 你的检索准不准确
- 你的上下文组织得好不好

当你开始能从“检索链路”而不是只从“模型表现”去分析问题时，说明你真的开始掌握 RAG 了。
