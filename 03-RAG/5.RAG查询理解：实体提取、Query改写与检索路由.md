## 1. 这篇文档讲什么

很多人做 RAG 时，把注意力放在知识库内容上，却忽略了另一件同样重要的事：

**系统是否真的理解了用户在问什么。**

用户的一句自然语言问题，常常并不适合直接拿去检索，因为里面可能有：

- 省略
- 代词
- 简称
- 别名
- 时间范围
- 条件约束

查询理解要做的，就是把“用户说的话”转成“系统更适合检索的话”。

---

## 2. 为什么用户原始问题不能直接拿去检索

看几个典型例子：

- “这个政策现在还能用吗？”
- “P8 出差去深圳住酒店标准是多少？”
- “企业版那个新套餐支持私有化吗？”

这些问题里都存在检索难点：

- “这个政策”到底指哪份制度
- “P8”是不是员工等级实体
- “深圳”是不是地区过滤条件
- “企业版那个新套餐”是不是产品别名

如果系统不先做理解，直接把原句拿去向量检索，结果可能“方向差不多”，但并不稳定。

---

## 3. 什么是查询理解

查询理解可以理解成线上检索前的一层语义加工。它的目标通常包括：

1. 识别关键实体
2. 补全隐含条件
3. 把口语表达改写成更规范的查询
4. 判断应该走哪种检索策略

它不是为了把问题“改得更好看”，而是为了让检索模块拿到更适合工作的输入。

---

## 4. 实体提取在 RAG 里有什么用

### 4.1 什么叫实体

在企业场景里，实体可能包括：

- 产品名
- 部门名
- 员工级别
- 城市
- 日期
- 版本号
- 客户类型
- 制度名称

### 4.2 实体提取的价值

实体提取常见的价值有三类：

#### 一是做过滤

例如识别出“深圳”“P8”“2025 版”之后，就可以在检索时加元数据过滤。

#### 二是做别名归一

例如用户说“企微”“企业微信”“WeCom”，系统可以统一映射到同一实体。

#### 三是做上下文补全

如果系统知道“P8”是员工职级，“差旅标准”是制度主题，那么后续 query 改写和路由就会更稳。

---

## 5. 为什么要做 Query 改写

很多用户问题天然更适合“交流”，但不一定更适合“检索”。

Query 改写常见目的包括：

- 把口语问题改成更规范的表达
- 展开简称和别名
- 把模糊代词替换成明确实体
- 拆成主问题和子问题
- 生成多个检索角度的 query

例如：

原问题：

“这个月华东还能按老政策走吗？”

可能改写为：

- “华东地区当前生效政策是什么”
- “华东地区旧政策是否仍然有效”
- “华东地区政策生效时间与版本变更”

这里的目标不是让模型写得更漂亮，而是让检索覆盖更多正确候选。

---

## 6. 检索路由是什么

不是所有 query 都该走同一条检索链路。

检索路由的意思是：

**根据问题类型，把 query 分发到更适合的检索方式。**

例如：

- 明确问编号、产品名、制度名时，更适合关键词或精确匹配
- 自然语言问句、更偏语义理解时，更适合向量检索
- 高价值问题可以走混合检索加重排
- 某些实体类问题还可以先查结构化数据库，再补文档检索

路由的价值在于，不让所有问题都走“同一把锤子”。

---

## 7. 一个常见的查询理解链路

可以把线上流程简化成下面几步：

1. 接收用户原始 query
2. 识别实体、时间、版本、地域、产品等关键信息
3. 做别名归一和必要补全
4. 生成一个或多个适合检索的 query
5. 判断走关键词检索、向量检索还是混合检索
6. 检索后进入重排序
7. 再把高质量候选交给生成模型回答

这个流程的关键思想是：

**线上检索不是一句 query 打到底，而是一次轻量的信息理解与调度。**

## 8. 企业场景中的典型问题

### 8.1 简称和别名太多

同一个东西可能有：

- 正式名
- 内部简称
- 旧名字
- 销售叫法
- 用户口语叫法

如果不做归一，检索命中率会明显下降。

### 8.2 问题依赖上下文

用户会说：

- “这个”
- “上一版”
- “刚才那个产品”

这类 query 必须结合对话上下文做补全，否则几乎无法检索。

### 8.3 条件很多但表达不完整

例如：

- 地区
- 时间
- 版本
- 客户类型
- 员工级别

这些条件如果没有被识别出来，召回结果就容易“相关但不对”。

## 9. 常见误区

### 9.1 误区一：用户怎么问，就怎么检索

这在简单 FAQ 里可能还能工作，但在企业复杂场景里往往不够。

### 9.2 误区二：Query 改写越多越好

不是。改写太多会带来噪声、增加成本，还可能把原问题含义改偏。

### 9.3 误区三：实体提取只对结构化搜索有用

实际上，实体提取对文档检索也很重要，因为它能帮助做过滤、归一和排序增强。

### 9.4 误区四：查询理解可以完全替代高质量知识库

不能。查询理解只能优化“怎么问”，不能替代离线数据治理和高质量索引。

## 10. 练习题与思考方向

### 练习 1

为什么“这个政策现在还能用吗”不适合直接拿去检索？

参考方向：

- “这个政策”缺少明确实体
- “现在”包含时间语义
- “还能用吗”更像版本有效性判断

### 练习 2

实体提取和 Query 改写分别更像在解决什么问题？

思考方向：

- 实体提取更像识别关键条件
- Query 改写更像重写检索表达

### 练习 3

什么情况下应该把 query 路由到关键词检索而不是只走向量检索？

参考方向：

- 编号、产品名、专有名词明确
- 用户在找精确字段
- 术语必须精确命中

## 11. 总结与下一步建议

RAG 查询理解的核心不是“让问题变复杂”，而是：

**把用户自然表达，转成更适合检索系统工作的表达。**

如果你把实体提取、Query 改写和检索路由补上，很多原本“似乎召回不准”的问题，会变成一个更可定位、可优化的工程问题。
