## 1. 适用人群

这篇讲义适合已经学过 LLM 与 Prompt 基础，准备系统进入知识库问答、企业检索和 RAG 工程的人。

## 2. 学习目标

读完后，你应该能够：

1. 用自己的话解释 RAG 是什么
2. 理解 RAG 的离线链路和在线链路
3. 理解 RAG 为什么会失败，以及问题可能出在哪一层
4. 初步具备搭建最小 RAG 系统的能力

## 3. 什么是 RAG

RAG 全称是 `Retrieval-Augmented Generation`，中文通常叫“检索增强生成”。

它不是单独一个模型，而是一种应用架构。你可以把它理解成两段式流程：

1. `Retrieval`：先从资料库里检索相关内容
2. `Generation`：再把检索结果连同用户问题一起交给大模型生成答案

它适合处理这类问题：

- “根据公司制度文档，年假怎么计算？”
- “这个产品说明书里提到的电池容量是多少？”
- “请总结这 20 篇内部会议记录里关于预算的结论”

## 4. RAG 解决了什么问题

### 4.1 降低幻觉

如果没有外部资料，模型只能靠参数记忆和当前上下文回答。  
RAG 的作用是先把真实证据送到模型面前，再让它回答。

### 4.2 引入私有知识

企业最重要的信息往往不在公开互联网，而在：

- 内部 Wiki
- 产品文档
- 客服知识库
- 合同与制度文件

RAG 让模型可以“临时读取”这些知识，而不需要重新训练一个模型。

### 4.3 知识可更新

模型训练完成后，参数知识相对固定；RAG 的资料库可以持续更新。  
这让它更适合知识经常变化的业务场景。

## 5. RAG 的完整流程

可以把一个 RAG 系统理解成下面 6 步：

1. 文档收集
2. 文档预处理
3. 文档切分
4. 向量化与索引
5. 检索与重排
6. 把上下文交给模型生成答案

### 5.1 文档收集

首先要明确知识从哪里来，例如：

- Markdown
- PDF
- Word
- 网页
- 数据库记录

很多 RAG 问题不是模型问题，而是源数据问题。

### 5.2 文档预处理

在真正做检索之前，通常要做：

- 去除无意义页眉页脚
- 修正文档编码问题
- 标记标题层级
- 识别表格、代码块、列表
- 给文档补充元数据

### 5.3 文档切分

长文档不能无限塞进上下文，所以要切成更小的 `chunk`。

切分的目标不是越小越好，而是：

- 粒度足够细
- 信息仍然完整
- 成本和精度平衡

### 5.4 向量化与索引

切分后的文本通常会被转成 embedding，并和原文、元数据一起存入索引系统。

### 5.5 检索与重排

用户提问后，系统先做检索，而不是直接回答。  
真实系统里更常见的方案是：

**混合检索 = 关键词检索 + 向量检索**

之后再用重排模型对候选结果排序。

### 5.6 生成答案

最终模型不是“凭空回答”，而是基于检索证据组织答案。  
因此最终答案质量，依赖于上游证据是否相关、完整、排序合理。

## 6. 你可以怎样理解 RAG 的本质

一个非常实用的理解方式是：

**RAG 不是问答技巧，而是信息供给系统。**

很多看起来像“模型不聪明”的问题，本质上其实是：

- 没找对资料
- 找出来的资料不完整
- 排序不对
- 查询没被系统正确理解

## 7. RAG 的几个关键模块

建议把 RAG 拆成下面 5 个模块理解：

1. 数据层：文档质量、清洗、结构恢复
2. 索引层：切分、向量化、元数据组织
3. 检索层：关键词、向量、混合召回
4. 排序层：rerank、候选筛选
5. 生成层：上下文组织、答案生成、引用出处

## 8. RAG 为什么会失败

最常见的失败原因有：

1. 数据源本身就脏
2. 切分把关键信息切断了
3. 查询没有被正确理解
4. 召回候选里缺了关键证据
5. 重排顺序不合理
6. 给模型的上下文太长或太乱

## 9. RAG 的常见优化方向

真实优化通常围绕下面几个方向：

1. 数据清洗和结构恢复
2. 切分策略调整
3. Embedding 模型选型
4. 混合检索
5. 重排模型
6. Query 改写与检索路由
7. 答案模板与引用策略

## 10. 一个最小可行 RAG 系统长什么样

一个最小版本通常包括：

1. 一批清洗过的文档
2. 基础切分逻辑
3. 一个 embedding 模型
4. 一个向量索引
5. 一个把检索结果送给模型的问答接口

不要一开始就追求最复杂框架。  
真正重要的是：链路可解释、问题可定位。

## 11. 学习 RAG 时最该关注的指标

建议优先关注：

1. 是否召回了该找的资料
2. 候选排序是否合理
3. 最终答案是否忠于证据
4. 是否能给出稳定出处

## 12. RAG 与微调的区别

一个很实用的区分是：

- RAG 解决“怎么把外部知识送进去”
- 微调更偏“怎么改变模型行为模式”

如果问题是知识更新快、知识在外部文档里，通常先考虑 RAG。  
如果问题是输出风格、领域任务模式、行为一致性，则可能考虑微调。

## 13. 初学者常见误区

- 把 RAG 当成“向量库 + Prompt”
- 只盯着生成层，不看上游证据
- 看到效果差就只想换更强模型
- 忽视数据清洗和元数据设计

## 14. 自学练习题

### 练习 1

为什么说很多 RAG 问题不是模型问题，而是数据或检索问题？

思考方向：

- 模型看到的证据是否正确
- 关键资料是否根本没有被送到模型面前

### 练习 2

切分为什么既不能太小，也不能太大？

思考方向：

- 太小会丢上下文
- 太大则会稀释相关信息并浪费窗口

### 练习 3

如果用户问“去年研发投入变化和利润趋势是否一致”，你觉得这是更像召回问题、排序问题还是查询理解问题？

思考方向：

- 是否涉及时间范围
- 是否涉及多指标和比较关系

## 15. 总结与下一步建议

这篇讲义最重要的一句话是：

**RAG 的本质，不是让模型“更聪明”，而是让系统更稳定地给模型提供正确证据。**
