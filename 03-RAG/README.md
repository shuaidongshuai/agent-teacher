## 1. 这一章在整个学习路线里的位置

如果说 `01-LLM基础` 解决的是“模型是什么”，`02-Prompt工程` 解决的是“怎么把任务说清楚”，那 `03-RAG` 解决的就是：

**当答案依赖外部资料时，怎样把正确的信息在正确的时候送到模型面前。**

很多看起来像“模型不聪明”的问题，本质上其实是：

- 没有把相关资料找出来
- 找出来的资料不完整
- 找出来的资料排序不对
- 查询本身没有被系统正确理解

所以 RAG 不是只和“检索”有关，它本质上是一整套信息供给系统。

## 4. 企业实战增强主题

如果你已经能跑通一个基础 demo，接下来最值得补的不是更复杂的框架，而是下面 4 个工程主题：

### 4.1 离线数据治理

企业文档通常不干净、不统一、版本混乱。RAG 效果差，很多时候不是模型弱，而是知识底座脏。

### 4.2 Embedding 选型

不同模型在语言、领域、延迟、成本、私有化部署约束下差异很大，不能只看单一排行榜。

### 4.3 重排序

召回负责“先把候选找出来”，重排序负责“从候选里挑最相关的给模型”，这一步对最终答案质量影响非常直接。

### 4.4 查询理解

企业问题经常包含简称、省略、别名、时间范围和业务实体，不做实体提取和 Query 改写，检索系统很容易“没找错方向，但还是没找对答案”。

## 5. 学习建议与下一步

学习这一章时，建议你始终带着一个判断框架：

- 这是数据问题，还是检索问题？
- 这是召回不够，还是排序不对？
- 这是文档切分问题，还是查询理解问题？
- 这是模型能力不够，还是给模型的信息不对？

如果你能把这些问题逐渐分开看，RAG 学习就会从“调参碰运气”变成“按链路定位问题”。
