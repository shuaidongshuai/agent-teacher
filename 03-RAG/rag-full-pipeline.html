RAG 全流程图

这不是一条“用户提问后去向量库查一下”的简单链路，而是一个从离线知识构建、在线查询理解、召回与重排、答案生成，到评测回流与持续优化的完整系统。 这页的目标是让你同时看清教学逻辑和生产逻辑。

离线数据构建 在线查询理解 检索与重排 生成与评测回流
Stage 1

离线数据构建

目标是把原始资料变成结构化、可索引、可追踪、可更新的知识单元。这一层决定系统上限。

Stage 2

在线查询理解

目标是把“用户原话”变成“检索系统更容易工作的输入”，减少口语、省略、简称和歧义带来的损失。

Stage 3

检索与重排

目标不是“找到一些相关内容”，而是把最该给模型看的证据稳定放到前面，并控制噪声与上下文预算。

Stage 4

生成、评测与回流

最后不只是回答用户，还要控制引用、拒答、日志、评测与反馈闭环，否则系统很难持续优化。

生产级控制面：不直接回答用户，但决定系统是否敢上线

权限与版本

文档权限过滤、版本选择、废弃内容屏蔽、来源可追踪，是企业 RAG 必须补的底层能力。

观测与指标

至少看召回率、重排命中、引用覆盖、答案正确率、延迟、成本、失败类型与人工接管率。

成本与时延治理

控制 query 改写次数、召回规模、重排深度、上下文长度和模型选择，否则系统会又慢又贵。

数据接入与版本管理

目标是把分散在不同系统里的原始资料稳定接入，并明确每份知识的来源、归属、权限、版本与更新时间。 企业里最常见的问题不是“没有资料”，而是资料进来之后无法追踪和治理。

        生产注意点

          超完整单图海报

          这是一张适合截图、讲课和全局复习的压缩总图。它把离线知识构建、在线问答执行、生产治理闭环三条链路收拢到一页里。

          Column A · Offline

          离线知识构建

          决定系统上限。目标是把原始资料变成可检索、可追踪、可更新、可过滤的知识对象。

          1. 数据接入 接入 PDF、Word、网页、数据库、FAQ、工单、Wiki，并保留来源、权限、版本和更新时间。
          2. 解析与清洗 做 OCR、去噪、去重复、结构恢复，解决页眉页脚、表格、标题层级和扫描误差问题。
          3. 切分与元数据补充 把文档切成 chunk，并补上标题、主题、部门、时间、实体、权限、版本等元数据。
          4. Embedding 与多索引落库 生成向量，写入向量数据库，并建立 BM25、标签索引、过滤索引、结构化索引。
          5. 离线质检与发布 用样本 query 验证召回效果、索引正确性和知识覆盖后，再灰度发布索引版本。
          Column B · Online

          在线问答执行

          决定用户每次提问时系统怎么工作。目标是把用户问题转成正确证据，再把证据稳定交给大模型。

          6. 用户查询进入 拿到原始 query，并结合多轮上下文识别当前真实意图。
          7. Query 理解与改写 做实体抽取、条件补全、别名识别、时间范围识别，并生成一个或多个检索 query。
          8. 检索路由与初步召回 决定查哪个知识库、走什么索引，再从向量库、BM25、标签过滤里召回候选结果。
          9. 重排、去重与证据筛选 对召回候选做 rerank，过滤旧版本、重复块、关键词相关但不真正答题的噪声内容。
          10. 上下文组装与调用 LLM 把高价值证据组织进 prompt，上下文长度可控，然后调用大模型基于证据生成答案。
          11. 答案校验与结果返回 检查引用、幻觉、越权、答非所问和证据不足，必要时拒答、降级或转人工。
          Column C · Governance

          生产治理闭环

          决定系统能不能稳定跑、敢不敢上线、是否会持续变好。没有这条链路，系统通常只能停留在 Demo。

          12. 增量更新与删除同步 知识库变化后及时更新索引，并同步删除废弃、失效和越权内容。
          13. 权限过滤与版本治理 在线检索时按用户身份过滤，并优先保留最新、有效、可访问的知识版本。
          14. 缓存、性能与成本控制 对高频 query、热文档、Embedding、重排结果和答案做缓存，控制时延和费用。
          15. 降级、兜底与人工接管 当向量库、重排、模型异常时，支持 BM25 降级、FAQ 兜底、拒答和人工接管。
          16. 监控、评测与样本沉淀 记录召回、重排、答案、引用、延迟、成本、失败类型，并把线上问题沉淀成评测集。
          17. AB 实验与策略优化 灰度验证新索引、新 Embedding、新 rerank、新 Prompt 与新模型，再决定全量发布。

          一张图里最关键的 3 个判断

          • 如果“找不到”，先查离线知识构建和 Query 理解，而不是先怪大模型。
          • 如果“找到了但答不好”，重点查重排、上下文组装和答案约束。
          • 如果“系统能跑但不稳定”，重点查权限、版本、缓存、降级、评测和回流机制。

          一句话总纲

          生产级 RAG = `离线知识准备` + `在线检索问答` + `治理闭环持续优化`。 三条链路缺一条，系统都很难稳定。

          问题排查路径

          教学里最有价值的不是“背定义”，而是知道不同故障该先查哪层。下面给出一个生产里常用的初始排查顺序。

          召回差

          系统找不到该有的内容，优先怀疑离线链路和查询理解，不要一上来怪生成模型。

          1. 先查文档是否真的入库、是否版本正确
          2. 再查切分、标题和元数据是否破坏了语义
          3. 再查 query 改写、实体提取和路由是否偏了
          4. 最后再看 Embedding 与召回策略

          答案差

          找到了内容但答不好，通常不只是生成问题，往往是给模型的证据排序和组装就出了偏差。

          1. 先看前几条证据是否真的最相关
          2. 再看是否把旧版本、重复块或噪声送进了上下文
          3. 再看 prompt 是否约束了引用、拒答和答案边界
          4. 最后再评估生成模型能力

          延迟高

          RAG 慢通常是链路太长，不是某一个模块单独慢。要按步骤拆开看。

          1. 看 query 改写是否过多轮
          2. 看召回候选是不是拉得太大
          3. 看重排深度和模型是不是过重
          4. 看上下文是否过长、生成模型是否过大

          成本高

          成本问题往往和时延问题一体两面，常见根因是“每层都想做得更保险，最后所有层都过重”。

          1. 减少不必要的 query 改写和多路召回
          2. 控制重排数量和上下文 token
          3. 按任务复杂度分层选模型
          4. 把高频失败样本做成离线评测，避免线上反复试错

          执行链路流程图

          这张图更贴近“系统到底怎么跑起来”。它把你提到的关键步骤串成一条更线性的执行链，适合讲实现、讲项目、讲生产落地。

          离线准备链路：把原始数据变成可检索知识

          离线链路回答的是：知识从哪里来，怎样清洗，怎样切分，怎样进入向量数据库与其他索引系统。

          1. 原始数据接入 接入 PDF、Word、网页、数据库、FAQ、工单、Wiki，并保留来源、版本、权限信息。
          2. 数据清洗与解析 去除噪声，恢复标题层级、表格结构、正文段落和附件关系。
          3. 文本切分与元数据补充 把文档切成 chunk，并补标题、时间、部门、主题、实体、权限等元数据。
          4. 向量化与索引建立 生成 Embedding，写入向量数据库，同时建立 BM25、标签、过滤索引。
          5. 离线质检与发布 用样本 query 验证召回效果，确认可用后再发布索引版本到线上。
          关键理解：如果这条链路做不好，后面的 Query 改写、召回、重排和生成都会被拖累。生产里很多“模型回答不准”的问题，本质上其实是离线知识准备阶段埋下的。

          在线问答链路：从用户查询到大模型回答

          在线链路回答的是：用户一提问，系统如何理解问题、改写 query、检索召回、重排证据，并最终调用大模型生成答案。

          6. 用户查询进入系统 拿到用户原始问题，并结合多轮上下文判断这次到底在问什么。
          7. Query 理解与改写 抽取实体、时间、版本、过滤条件，把口语问题改写成更适合检索的 query，必要时生成多路 query。
          8. 检索与初步召回 从向量数据库、BM25 索引、标签过滤中召回 Top K 候选 chunk。
          9. 重排与证据筛选 对候选结果做 rerank，去重、去旧版本、去噪声，把最相关证据排到前面。
          10. 上下文组装 把高价值 chunk 组织成 prompt 上下文，控制 token 长度、引用结构和证据覆盖。
          11. 调用大模型生成答案 把用户问题和证据上下文一起交给 LLM，要求基于证据回答并给出处。
          12. 答案校验与安全检查 检查是否幻觉、是否越权、是否缺引用、是否答非所问，必要时拒答或回退。
          13. 返回结果给用户 输出最终答案、引用来源、边界说明，必要时说明证据不足或建议继续追问。
          14. 记录日志与用户反馈 记录 query、改写、召回、重排、答案和用户反馈，为后续评测和优化提供样本。
          15. 回流优化 把失败案例回流到离线数据清洗、切分、检索、重排、Prompt 和模型选择中。
          关键理解：线上链路不是“查一下向量库然后问一下模型”这么简单，而是一个包含 Query 改写、检索策略、召回规模、重排深度、上下文组装和答案校验的多阶段系统。

          生产治理链路：真正决定系统能不能长期稳定运行的流程

          生产级 RAG 不只是“能答出来”，还必须保证知识新鲜、权限正确、性能可控、问题可定位、效果可持续优化。

          16. 增量更新与删除同步 文档新增、修改、下线后，离线链路要及时增量更新索引，并同步删除废弃内容。
          17. 权限过滤与版本治理 在线检索时按用户身份、部门、角色过滤文档，并优先使用最新有效版本。
          18. 缓存与性能优化 对高频 query、热文档、Embedding、重排结果和答案做分层缓存，控制时延和成本。
          19. 降级与兜底策略 向量库异常、重排超时、模型失败时，要支持 BM25 降级、FAQ 兜底、拒答或人工接管。
          20. 线上监控与告警 持续监控召回率、空召回率、答案正确率、引用覆盖率、延迟、成本和错误类型。
          21. 失败样本沉淀 把答错、漏答、慢答、越权、误引等问题沉淀成结构化案例，而不是只靠人工记忆。
          22. 离线评测与回归测试 定期用评测集检查数据更新、检索策略、重排模型、Prompt 和大模型切换后的效果变化。
          23. AB 实验与灰度发布 新 Embedding、新重排模型、新 Prompt 或新索引不要直接全量发布，要先灰度验证。
          24. 人工反馈回流 把用户追问、人工修正、客服反馈回流到知识库和评测集中，持续提升效果。
          25. 策略持续优化 根据反馈决定优先改数据、改 Query 理解、改检索、改重排、改 Prompt，还是换模型。
          关键理解：生产级 RAG 至少包含三条并行逻辑。第一条是离线知识构建，第二条是在线问答执行，第三条是治理与运维闭环。没有第三条，系统通常只能做 Demo，很难长期稳定上线。

          使用建议：讲课时先按 4 个大阶段顺着讲，再挑 2 到 3 个最常见故障倒着查。这样学习者不会只记住“RAG 有哪些模块”，而是会真正建立“问题落在哪层、应该先改哪里”的工程判断。
