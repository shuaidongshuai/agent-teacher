## 适用人群

这份文档适合已经基本学完本资料库，希望通过“被提问”的方式检验掌握程度的学习者。

## 学习目标

学完并做完这份文档后，你应该能够：

1. 用自己的话解释 LLM、Prompt、RAG、Agent 的核心概念
2. 面对常见面试题时，给出结构化而不是零散的回答
3. 在项目题里说明设计思路、失败原因和优化方法
4. 识别自己目前最薄弱的知识模块

## 怎么使用这份模拟面试题

推荐你按下面方式使用：

1. 先只看题目，口头回答或自己写回答。
2. 再对照参考答案，看自己漏了哪些关键点。
3. 把不会的题目回链到前面对应章节继续补。
4. 第二轮练习时，尽量把答案讲得更短、更清楚、更像真实面试。

这份资料的重点不是背标准答案，而是训练你在压力下仍然能把问题讲明白。

## 第一部分：LLM 基础题

### 1. 什么是 Token？为什么它很重要？

**参考答案：**

Token 可以理解为模型处理文本时使用的最小计数单位，但它不等于“一个汉字”或者“一个单词”。模型的输入、输出、上下文窗口、费用统计，通常都是按 Token 来计算的。

它重要的原因有三个：

- 它决定一段内容能不能被完整放进上下文
- 它直接影响推理成本和响应延迟
- 它会影响模型对长文本任务的处理效果

面试里更好的回答方式不是只说定义，而是顺带说明：如果上下文超长，就要考虑截断、摘要、分段、RAG 等策略。

### 2. 什么是上下文窗口？窗口越大越好吗？

**参考答案：**

上下文窗口是模型在一次推理中能看到的总内容范围，包括系统提示词、用户输入、历史对话、工具结果和模型输出。

窗口更大通常意味着：

- 能处理更长文档
- 能保留更多对话历史
- 能放入更多检索资料

但窗口不是越大越好，因为更大的上下文通常意味着更高成本、更高延迟，而且上下文变长后，模型对关键信息的利用未必线性变好。很多问题不是靠“塞更多内容”解决，而是靠“放更对的内容”解决。

### 3. 预训练、微调、对齐分别在解决什么问题？

**参考答案：**

可以这样理解：

- 预训练解决“模型先学会语言和世界知识”
- 微调解决“模型在某类任务或风格上更稳定”
- 对齐解决“模型更符合人类偏好、规则和使用预期”

如果你想答得更完整，可以补一句：真实系统里，很多效果问题并不一定要靠重新训练解决，往往先从 Prompt、检索、工具和工作流设计入手更实际。

### 4. Transformer 为什么重要？

**参考答案：**

Transformer 重要，是因为它提供了一种更适合大规模并行训练、并且能处理长距离依赖的建模方式。它让模型能通过注意力机制，在处理一个位置时参考序列中其他位置的信息。

面试时不需要把公式全背出来，但最好能说清：

- 它和传统循环结构相比更容易并行
- 注意力机制是核心思想
- 它是现代大模型的重要基础架构

### 5. 温度参数是做什么的？温度越低越好吗？

**参考答案：**

温度控制的是采样时的随机性。温度低，输出更稳定、更保守；温度高，输出更发散、更有创造性。

它不是越低越好。像事实问答、结构化抽取、工具参数生成，往往更适合低温；创意写作、头脑风暴、广告文案，则可以适当提高温度。真正好的回答是“参数要跟任务目标匹配”。

## 第二部分：Prompt 工程题

### 6. 一个好的 Prompt，通常应该包含哪些部分？

**参考答案：**

一个好 Prompt 不一定很长，但通常会把任务边界说清楚。常见组成包括：

- 角色或身份
- 任务目标
- 输入背景
- 约束条件
- 输出格式
- 示例

如果要更像面试回答，可以补一句：Prompt 工程的本质不是堆字数，而是减少歧义、稳定输出、提高可控性。

### 7. Few-shot 示例为什么有用？

**参考答案：**

Few-shot 的作用是“用例子告诉模型你想要什么”。很多时候，模型不是不懂任务，而是不确定你想要的风格、结构和判断标准。示例能帮助模型快速对齐输出方式。

它尤其适合：

- 分类标准不够显式的任务
- 输出格式要求严格的任务
- 你希望模型模仿某种写法的任务

但示例也不是越多越好，太多会占上下文，还可能把模型带偏。

### 8. 为什么明明写了 Prompt，模型结果还是不稳定？

**参考答案：**

常见原因有：

- 任务本身定义不清
- 输入信息不完整
- 输出标准不明确
- Prompt 与任务复杂度不匹配
- 温度等参数设置不合适
- 工具结果或外部上下文本身有噪声

更成熟的回答是：Prompt 不稳定，很多时候不是“模型不行”，而是整个系统没有把约束、上下文、示例和评测闭环做好。

### 9. 如果让你调一个效果不好的 Prompt，你会怎么做？

**参考答案：**

可以按一个很实用的顺序回答：

1. 先明确失败现象，别笼统说“效果不好”
2. 收集失败样本，判断是理解错、格式错还是事实错
3. 缩小变量，一次只改一项
4. 补充约束、示例或输出模板
5. 建立小规模测试集做对比验证

面试官通常喜欢听到“基于样本调试”和“控制变量”，因为这说明你不是拍脑袋调 Prompt。

## 第三部分：RAG 题

### 10. 什么是 RAG？它解决什么问题？

**参考答案：**

RAG 是检索增强生成。它的核心思路是：先从外部知识源中检索相关内容，再把检索结果提供给模型生成答案。

它主要解决两个问题：

- 模型知识不是实时更新的
- 纯靠参数记忆回答专业问题，容易幻觉

所以 RAG 的价值不是“让模型更聪明”，而是“让模型回答时更有依据”。

### 11. RAG 的核心流程是什么？

**参考答案：**

一个典型 RAG 流程可以概括为：

1. 文档采集与清洗
2. 文档切分
3. 向量化或建立索引
4. 查询检索
5. 重排与筛选
6. 构造上下文
7. 生成答案

如果面试官继续追问，重点往往会落在两个地方：切分策略为什么这样设计，以及为什么明明检索到了内容，答案还是不好。

### 12. 文档切分为什么重要？

**参考答案：**

切分决定了知识库最小检索单元的质量。切得太碎，语义不完整；切得太大，召回不精准，还容易把无关信息一起塞进上下文。

所以切分的本质，是在“信息完整性”和“检索精度”之间找平衡。实际工程里，常会结合：

- 固定长度切分
- 按标题或段落切分
- overlap 重叠
- 面向业务语义的定制切分

### 13. 为什么有时候检索正确，回答还是错？

**参考答案：**

这是 RAG 里非常常见的问题。原因可能包括：

- 检索片段虽然相关，但不够完整
- 检索结果排序不合理，关键证据没排前面
- 上下文太长，关键信息被淹没
- 模型没有严格依据材料回答
- 提问本身含糊，导致召回方向偏了

所以 RAG 不能只看“有没有召回到”，还要看“召回质量、上下文组织、生成约束”。

### 14. 召回和重排分别在做什么？

**参考答案：**

召回负责“尽量别漏掉可能相关的内容”，重排负责“把最相关、最值得给模型看的内容排到前面”。

你可以把它理解成：

- 召回追求广度
- 重排追求精度

这也是为什么很多系统单靠向量检索效果有限，因为“找得到”不等于“排得好”。

## 第四部分：Agent 与工具调用题

### 15. Agent 和普通聊天机器人有什么区别？

**参考答案：**

普通聊天机器人主要是在用户提问后生成回答，而 Agent 更强调围绕目标持续推进任务。它往往需要：

- 规划下一步
- 调用工具
- 根据执行结果调整行为
- 在多步过程中保持状态

一句话说，聊天机器人偏“回答问题”，Agent 偏“完成任务”。

### 16. 什么情况下不应该上 Agent？

**参考答案：**

如果任务流程固定、步骤明确、判断空间很小，通常没必要上 Agent，工作流可能更简单、更稳定、更便宜。

例如：

- 固定字段抽取
- 固定模板生成
- 已经明确的审批流程

这是面试里很加分的一点，因为它说明你不是“看到大模型就想上 Agent”，而是会做方案边界判断。

### 17. 什么是工具调用？它的标准闭环是什么？

**参考答案：**

工具调用是指模型先输出结构化调用意图，由外部程序真正执行工具，再把结果回注给模型生成最终回答。

标准闭环通常包括：

1. 识别是否需要工具
2. 生成工具名和参数
3. 程序执行工具
4. 回注工具结果
5. 模型生成最终回答

关键点不只是“能调用”，而是参数生成是否稳定、异常处理是否完善、结果回注是否清晰。

### 18. Agent 为什么容易失败？

**参考答案：**

Agent 容易失败，往往不是单一点问题，而是链路长导致误差累积。常见原因包括：

- 目标理解偏差
- 规划不合理
- 工具选错或参数错
- 记忆污染
- 外部环境不稳定
- 缺少中间校验和兜底

成熟一点的回答可以加一句：Agent 工程的重点不是“让模型更会想”，而是“把失败限制在可观测、可恢复、可回退的范围内”。

### 19. 单 Agent 和工作流、多 Agent 分别适合什么场景？

**参考答案：**

可以这样区分：

- 工作流：适合确定性强、步骤固定的流程
- 单 Agent：适合中等复杂度、需要动态决策的任务
- 多 Agent：适合任务天然可以分工，且收益大于协作成本的场景

面试时不要只说概念，还要补充一个判断原则：系统越复杂，协同成本、调试难度和不稳定性通常也越高。

### 20. 如果工具调用经常报错，你会怎么排查？

**参考答案：**

一个比较完整的排查顺序是：

1. 先看是模型参数生成错，还是工具本身执行错
2. 检查参数 schema 是否清晰
3. 记录调用日志，包括参数、耗时、错误码、原始返回
4. 对高频错误做重试、降级和兜底
5. 回看 Prompt 是否明确了调用条件和参数约束

面试时如果能主动提到“日志”和“可观测性”，通常会显得更像做过真实系统。

## 第五部分：项目与评测题

### 21. 如果让你做一个企业知识库问答系统，你会怎么设计？

**参考答案：**

一个基础版本通常包括：

- 文档接入与清洗
- 切分与索引
- 检索与重排
- 上下文构造
- 回答生成
- 引用来源展示
- 评测与反馈闭环

更好的回答方式是按“先最小可用，再逐步增强”来讲。比如先做单轮问答，再补多轮上下文、权限控制、缓存、反馈收集。

### 22. 你会怎么评估一个 RAG 或 Agent 系统好不好？

**参考答案：**

不能只看“感觉还行”，至少要分层评估：

- 检索层：召回率、重排效果、命中质量
- 生成层：正确性、完整性、引用一致性
- 系统层：延迟、成本、稳定性、失败率
- 用户层：满意度、任务完成率、人工接管率

这是很典型的面试题，重点在于说明你会把问题拆层，而不是只盯最终回答。

### 23. 如果用户反馈“答案听起来很像对的，但其实是错的”，你会怎么处理？

**参考答案：**

这本质上是在处理幻觉或错误置信输出。可以从三个层面入手：

- 数据层：检查是否检索不到、检索错、知识过期
- 生成层：加强引用约束、要求基于材料回答
- 产品层：展示来源、降低过度自信表达、必要时提示不确定

如果能再补一句“要先收集错误案例做归因分析”，会比直接拍结论更好。

### 24. 项目上线后，最容易被忽略的问题是什么？

**参考答案：**

很多人只关注 Demo 跑通，但上线后真正容易出问题的是：

- 数据更新机制
- 监控与日志
- 成本控制
- 权限和安全
- 异常兜底
- 用户反馈闭环

所以“能演示”不等于“能上线”，工程系统最终拼的是稳定性和运营能力。

### 25. 如果让你用一句话总结这个项目学习的主线，你会怎么说？

**参考答案：**

可以这样回答：

这套学习的主线，是从理解模型能力边界开始，逐步掌握 Prompt、RAG、工具调用和 Agent，最后把这些能力组织成一个能在真实场景里稳定工作的系统。

这个问题看似简单，其实是在考你有没有建立完整知识地图。

## 常见失分点

学生在回答这类题目时，最容易出现下面几种问题：

- 只会背定义，不会讲为什么重要
- 只会讲概念，不会联系工程实践
- 只会说“可以优化”，但说不出怎么优化
- 把 RAG、工作流、Agent 的边界讲混
- 回答没有结构，想到哪里说到哪里

如果你发现自己经常卡在这些点上，不一定是你不够聪明，更多时候只是还缺少“把知识讲出来”的训练。

## 实战建议

你可以把这份文档当成一个小型训练营来用：

1. 第一轮，只要求自己能答出大意。
2. 第二轮，要求每道题都能在 1 到 2 分钟内结构化回答。
3. 第三轮，找朋友或让 AI 随机追问，把答案讲得更接近真实面试。

如果你想继续增强，可以自己补两类题：

- 反问题：为什么不用别的方案？
- 失败题：这个系统最可能在哪一步翻车？

这两类题最能拉开“学过”和“真的理解了”的差距。

## 总结

模拟面试的价值，不只是检查你记住了多少，更是逼你把零散知识连成系统。真正能通过面试或独立做项目的人，通常不是知道最多的人，而是能把问题拆开、讲清、落地的人。

下一步建议是：先完整做一轮这 25 道题，再把自己最容易卡壳的 5 道题对应回前面章节复习，然后进行第二轮口头输出训练。
