## 面试题与参考答案

### 1. 什么是 RAG？它解决什么问题？

**参考答案：**

RAG 是检索增强生成。它的核心思路是：先从外部知识源中检索相关内容，再把检索结果提供给模型生成答案。

它主要解决两个问题：

- 模型知识不是实时更新的
- 纯靠参数记忆回答专业问题，容易幻觉

### 2. RAG 的核心流程是什么？

**参考答案：**

一个典型 RAG 流程可以概括为：

1. 文档采集与清洗
2. 文档切分
3. 向量化或建立索引
4. 查询检索
5. 重排与筛选
6. 构造上下文
7. 生成答案

### 3. 如果面试官让你简洁讲一遍 RAG 全流程，你会怎么回答？

**参考答案：**

我一般会把 RAG 全流程分成离线和在线两段来说。

离线阶段，先做数据接入和清洗，把 PDF、Word、网页这些原始资料解析出来；然后做切分，补充标题、时间、来源、权限这些元数据；接着做 Embedding 和索引构建，把内容写进向量数据库，必要时再配 BM25 或标签索引。

在线阶段，用户提问后，系统先做 Query 理解和改写，比如抽实体、补条件、识别时间范围；然后去做检索召回，拿到一批候选文档；再通过重排把最相关的证据放到前面；接着把这些证据组装进上下文，最后调用大模型生成答案。

如果是生产级 RAG，我还会补一句：除了主流程，还要有权限过滤、版本治理、缓存、降级兜底、日志评测和反馈回流，不然它只能算 Demo，不能算稳定上线系统。

**3 分钟版：**

RAG 本质上是一个“知识供给系统”，不是简单的“查一下向量库再问模型”。  

第一段是离线知识构建。先把 PDF、Word、网页、数据库这些原始数据接进来，然后做解析和清洗，去掉噪声、恢复标题层级和表格结构。接着做切分，因为 chunk 的质量会直接影响后面的召回效果。切分后还要补元数据，比如标题、时间、来源、部门、权限、版本这些。最后做 Embedding，写入向量数据库，必要时再配 BM25 或标签索引。  

第二段是在线问答执行。用户提问后，不是直接拿原句去检索，而是先做 Query 理解和改写，比如抽实体、识别时间范围、补全隐含条件。有了更适合检索的 query 之后，再去做召回，拿到一批候选文档。召回之后通常还要加一层重排，因为召回更关注覆盖，重排更关注精度。然后把筛出来的高价值证据组装进 prompt 上下文，最后调用大模型生成答案，并尽量要求带引用、基于证据回答。  

第三段是生产治理闭环。真正上线时，还要考虑权限过滤、版本治理、增量更新、删除同步、缓存、降级兜底、日志监控、评测回归和用户反馈回流。因为生产级 RAG 的问题往往不只是“答得对不对”，还包括“有没有越权”“是不是最新版本”“延迟和成本能不能接受”“出了问题能不能定位”。  

所以如果一句话总结，我会说：`生产级 RAG = 离线知识构建 + 在线检索问答 + 治理与优化闭环`。

### 4. 文档切分为什么重要？

**参考答案：**

切分决定了知识库最小检索单元的质量。切得太碎，语义不完整；切得太大，召回不精准，还容易把无关信息一起塞进上下文。

所以切分的本质，是在“信息完整性”和“检索精度”之间找平衡。

### 5. 为什么有时候检索正确，回答还是错？

**参考答案：**

这是 RAG 里非常常见的问题。原因可能包括：

- 检索片段虽然相关，但不够完整
- 检索结果排序不合理，关键证据没排前面
- 上下文太长，关键信息被淹没
- 模型没有严格依据材料回答
- 提问本身含糊，导致召回方向偏了

### 6. 召回和重排分别在做什么？

**参考答案：**

召回负责“尽量别漏掉可能相关的内容”，重排负责“把最相关、最值得给模型看的内容排到前面”。

你可以把它理解成：

- 召回追求广度
- 重排追求精度
