## 1. 适用人群

这篇文档适合已经开始做项目，且经常遇到“知道它有问题，但不知道先改哪里”的学习者和开发者。

## 2. 学习目标

学完后，你应该能够：

1. 把一次失败从表面现象拆到具体链路
2. 识别 Prompt、RAG、工具、Agent 状态等不同层的问题
3. 建立“先定位，再改动，再验证”的优化闭环
4. 避免在错误层上反复发力

## 3. 为什么很多优化效率很低

很多 AI 项目的优化效率低，不是因为团队不努力，而是因为优化动作和真实问题没有对齐。

常见现象是：

- 看到结果差，就立刻改 Prompt
- 看到召回差，就立刻换 Embedding 模型
- 看到 Agent 不稳，就立刻换框架

这些动作有时会有效，但更多时候只是“碰运气”。  
因为没有先确认问题究竟出在哪一层。

## 4. 一次失败应该怎样拆

建议把一次失败至少拆成下面 4 层：

### 4.1 输入层

检查：

- 用户问题是否本身模糊
- 是否缺少必要上下文
- 是否超出系统能力边界

### 4.2 理解与决策层

检查：

- Prompt 是否表达清楚
- Query 改写是否偏了
- Agent 是否选错了下一步

### 4.3 执行层

检查：

- 工具有没有调用成功
- 参数是否合法
- 检索链路是否拿到正确资料

### 4.4 输出层

检查：

- 模型是否正确使用了证据
- 输出是否合规
- 最终答复是否对用户可用

## 5. 从现象到根因的排查路径

下面是一条很实用的排查顺序。

### 第一步：先记录现象

不要上来就下结论。  
先明确：

- 错在什么地方
- 哪类问题最常出现
- 是偶发还是高频

### 第二步：看中间链路

例如：

- 检索结果是什么
- 工具返回了什么
- Agent 轨迹走了哪些步骤

### 第三步：判断是哪一层的问题

常见分类：

- Prompt / 指令问题
- 数据 / 检索问题
- 工具 / 参数问题
- 状态 / 编排问题
- 模型能力边界问题

### 第四步：只做最小必要改动

一次只改一个关键因素，避免多变量同时变化导致无法归因。

### 第五步：回到评测集验证

新改动必须回到固定样本集上验证，不要只靠手测几个顺手问题。

## 6. 常见问题到改进动作的映射

下面给一个常用映射框架。

### 6.1 现象：模型经常答非所问

优先排查：

- Prompt 目标不清
- 用户输入本身过于模糊
- 上下文关键约束被稀释

常见改进：

- 重写任务说明
- 增强约束表达
- 在必要时先做澄清而不是直接回答

### 6.2 现象：RAG 明明有文档，但答案仍然不对

优先排查：

- 正确文档排序是否靠后
- 上下文拼接是否太杂
- 生成层是否没有被约束为“基于证据回答”

常见改进：

- 增加重排
- 调整上下文构建
- 增强证据引用约束

### 6.3 现象：Agent 经常走弯路

优先排查：

- 决策空间是否过大
- 工具描述是否不清
- 是否缺少中间检查点

常见改进：

- 减少工具自由度
- 强化工具 schema 和描述
- 加入阶段性校验或终止条件

### 6.4 现象：线上一出错就很难查

优先排查：

- 有没有运行轨迹
- 有没有中间结果日志
- 有没有错误分类

常见改进：

- 增加 trace
- 增加关键指标
- 建立失败样本回流机制

## 7. 如何避免无效优化

### 7.1 不要一次改太多

一次同时换模型、改 Prompt、换检索策略，最后很难知道是哪一项起作用。

### 7.2 不要只看平均效果

平均分提升，不代表关键高风险样本也变好了。

### 7.3 不要只优化“看起来最显眼”的层

用户看到的是最终答案，但根因可能在更前面的链路。

## 8. 一个可复用的复盘模板

每次失败后，可以按下面 5 个问题复盘：

1. 现象是什么？
2. 证据是什么？
3. 问题最可能出在哪一层？
4. 本次最小改动是什么？
5. 改动后在固定样本集上结果怎样？

如果你能长期按这个模板积累，项目会越来越从“玄学调试”转向“可解释优化”。

## 9. 总结与下一步建议

优化方法论最重要的价值，不是教你更多技巧，而是防止你把时间浪费在错误方向上。

一句话总结：

**高质量优化的前提，不是更努力地改，而是更准确地定位。**

读完这一章后，建议直接回到你自己的项目：

1. 选 1 个失败样本
2. 按本文的 4 层拆解方法复盘
3. 做 1 次最小改动
4. 用固定样本集重新验证
