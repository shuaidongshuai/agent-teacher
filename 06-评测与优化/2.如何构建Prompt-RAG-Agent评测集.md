## 1. 适用人群

这篇文档适合已经意识到评测重要，但还不知道“第一批样本该怎么做”的学习者和开发者。

## 2. 学习目标

学完后，你应该能够：

1. 理解评测集为什么是版本对比的基础
2. 区分 Prompt、RAG、Agent 三类任务需要的样本差异
3. 为自己的项目构建一套最小可用评测集
4. 避免只收集“容易成功”的样本

## 3. 为什么评测集是第一步

没有评测集，很多优化都无法稳定比较。  
因为你每次测试的题目不同、难度不同、上下文不同，结论就会漂移。

评测集的价值在于：

- 固定比较口径
- 保留失败案例
- 支持版本回归测试
- 帮助团队讨论“到底哪里变好了”

所以评测集不是文档附件，而是优化工作的基线资产。

## 4. 评测集应该包含什么

一套好用的评测集，通常不只是“问题列表”，而应该至少包含下面几类信息：

### 4.1 输入

例如：

- 用户问题
- 上下文信息
- 任务目标
- 可用工具或知识范围

### 4.2 期望输出或判定标准

不是所有任务都能写成唯一标准答案，但至少要定义：

- 什么叫成功
- 什么叫部分成功
- 什么叫失败

### 4.3 样本标签

例如：

- 难度
- 业务场景
- 风险等级
- 是否依赖外部数据
- 是否属于历史高频失败问题

### 4.4 中间链路标注

这对 RAG 和 Agent 特别重要。  
例如：

- 正确文档 ID
- 应该调用的工具
- 合理的执行步数范围
- 允许的失败恢复路径

## 5. Prompt 评测集怎么构建

Prompt 任务通常更适合围绕“任务类型”来组织样本。

### 5.1 先按任务类型分组

例如：

- 摘要
- 改写
- 分类
- 抽取
- 结构化输出

### 5.2 再覆盖不同难度

不要只收最容易的输入，还要补：

- 歧义输入
- 冗长输入
- 信息缺失输入
- 多约束输入

### 5.3 明确判定方式

Prompt 评测里常见的判定维度包括：

- 是否理解任务
- 是否遵守格式
- 是否遗漏关键信息
- 是否出现明显幻觉

## 6. RAG 评测集怎么构建

RAG 评测集比 Prompt 更复杂，因为它不只看答案，还要看检索链路。

### 6.1 先准备问题和标准知识来源

每个样本最好至少标注：

- 用户问题
- 正确答案依据来自哪些文档
- 哪些文档是强相关，哪些只是弱相关

### 6.2 覆盖不同检索难题

建议重点补齐下面几类样本：

- 同义表达
- 别名和简称
- 时间范围问题
- 多跳问题
- 文档分散在多个片段中的问题

### 6.3 不要只看答案对不对

RAG 评测至少要拆成 3 层：

1. 召回对不对
2. 排序对不对
3. 最终生成对不对

因为：

- 召回错了，后面再强也很难救
- 召回对了但排位不对，也可能导致答案变差
- 候选都对，但上下文拼装差，最终答案也可能错

## 7. Agent 评测集怎么构建

Agent 评测最难，因为它涉及多步任务和动态决策。

### 7.1 先定义“任务完成”的标准

Agent 任务不能只看“写了一段像样的话”，而要看：

- 目标是否完成
- 是否调了正确工具
- 是否遗漏关键步骤
- 是否在合理步数内收敛

### 7.2 样本要覆盖不同失败模式

建议至少覆盖：

- 用户目标不清
- 工具调用失败
- 多工具依赖
- 中间状态更新
- 需要人工兜底的场景

### 7.3 允许多条正确路径

Agent 不像结构化抽取那样只存在唯一过程。  
同一个任务可能有多条合理路径，所以标注时应更多关注：

- 是否完成目标
- 是否越权或乱调用
- 是否出现明显低效或危险行为

## 8. 样本标注的实用建议

### 8.1 先追求可用，不追求完美

很多团队迟迟不开始评测，是因为总想等一套很完整的标注规范。  
更实用的做法是：

- 先做 20 到 50 条核心样本
- 先建立最基础字段
- 先能跑版本对比

### 8.2 优先收高价值样本

高价值样本通常包括：

- 线上高频问题
- 高风险业务问题
- 历史失败案例
- 业务方最关心的问题

### 8.3 定期淘汰和补充

评测集不是一劳永逸的。  
随着产品迭代，旧样本可能失去代表性，新问题需要持续补入。

## 9. 小团队如何低成本开始

如果你现在只是个人项目或小团队，可以从下面这套最小方案开始：

1. 准备 30 条核心问题
2. 按简单 / 中等 / 困难分层
3. 每条问题记录成功标准
4. 对 RAG 问题补充正确文档来源
5. 对 Agent 问题补充期望工具和关键步骤
6. 每次改 Prompt、模型或流程后跑一遍

这已经足够让你的项目从“主观试用”提升到“有基本对比能力”。

## 10. 总结与下一步建议

评测集建设的核心不是追求形式完整，而是建立一套能持续复用、能帮助定位问题的样本资产。

一句话概括：

**没有评测集，优化就是临场发挥；有了评测集，优化才有基线。**
