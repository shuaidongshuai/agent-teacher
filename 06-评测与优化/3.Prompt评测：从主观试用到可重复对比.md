## 3. Prompt 为什么需要单独评测

很多人会把 Prompt 当成“试几次就能调出来的东西”。  
这种做法在简单 Demo 上可能有效，但问题是：

- 结果不可复现
- 优化经验不可沉淀
- 一旦换场景就容易失效

Prompt 之所以值得单独评测，是因为它直接影响：

- 模型对任务的理解
- 输出格式的稳定性
- 多约束条件下的执行一致性
- 工具或下游系统是否能正确消费结果

## 4. Prompt 评测重点看什么

Prompt 评测的核心，不是看“语言是否更漂亮”，而是看它是否更稳地完成目标任务。

可以重点看下面 4 个问题：

### 4.1 任务是否理解正确

模型有没有抓住真正目标，而不是只抓住表面词。

### 4.2 约束是否被执行

例如：

- 字数限制
- 风格限制
- 不能编造
- 必须输出指定格式

### 4.3 不同输入分布下是否稳定

一个 Prompt 在标准输入上表现很好，不代表在模糊输入、脏输入、长输入下也稳定。

### 4.4 是否适合下游消费

如果输出要给程序解析，格式稳定性往往比文字优美更重要。

## 5. 常见评测维度

下面这些维度在 Prompt 任务中很常用。

### 5.1 正确性

模型是否完成了任务，没有明显误解或胡编。

### 5.2 完整性

是否遗漏了关键字段、关键步骤或重要信息。

### 5.3 格式合规性

是否按要求输出 JSON、表格、列表或固定结构。

### 5.4 一致性

类似输入下，模型输出是否稳定，而不是同一任务一会儿这样做、一会儿那样做。

### 5.5 安全性

是否容易被提示注入、角色混乱、越权输出等问题影响。

## 6. 如何设计一轮 Prompt 对比实验

一个实用的最小实验，通常包括下面几步：

### 6.1 固定模型和参数

如果你一边改 Prompt，一边换模型、换温度，那最后很难知道效果差异到底来自哪里。

### 6.2 固定评测样本

使用同一批样本对比 Prompt A 和 Prompt B。

### 6.3 固定判定维度

例如：

- 任务理解正确率
- 格式通过率
- 关键信息覆盖率

### 6.4 记录典型失败样本

比平均分更有价值的，往往是那些反复出问题的样本。  
因为它们更能暴露 Prompt 结构上的缺陷。

### 6.5 不只比较总分

还要看：

- 哪类题变好了
- 哪类题变坏了
- 改动主要影响了什么维度

## 7. 案例：为什么一个 Prompt 在演示时很好但上线后不稳

这通常不是因为模型“突然变笨了”，而是因为演示环境过于理想。

常见原因包括：

- 演示问题都比较规范
- 演示者默认知道很多隐含上下文
- 输出格式要求不严格
- 失败样本没有被纳入验证

所以一个更成熟的判断方式是：

不要问“这个 Prompt 看起来顺不顺”，而要问：

- 它在高频真实样本上是否更稳
- 它是否降低了失败率
- 它是否让下游流程更容易处理

## 8. 常见误区

### 8.1 误区一：Prompt 越长越稳

不一定。  
长 Prompt 可能补充了约束，也可能引入冲突、冗余和重点稀释。

### 8.2 误区二：只要回答更像人，就说明更好

自然不等于正确，流畅也不等于可执行。

### 8.3 误区三：看到个别漂亮案例就认为优化成功

真正有用的是整体分布变化，而不是一两个演示样本。

## 9. 总结与下一步建议

Prompt 评测的本质，是把“写 Prompt”从经验活变成可比较、可复盘的工程动作。

一句话总结：

**Prompt 优化不是比谁更会写漂亮指令，而是比谁更能稳定完成目标任务。**
