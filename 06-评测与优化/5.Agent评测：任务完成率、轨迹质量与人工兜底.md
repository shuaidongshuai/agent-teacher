## 1. 适用人群

这篇文档适合已经进入 Agent 设计或项目实战阶段，希望知道“Agent 到底怎样算好”的学习者。

## 2. 学习目标

学完后，你应该能够：

1. 理解 Agent 评测为什么比 Prompt 和 RAG 更复杂
2. 区分任务完成率、轨迹质量和系统稳定性
3. 知道为什么人工兜底不是失败补丁，而是系统设计的一部分
4. 为自己的 Agent 项目设计一套最小评测框架

## 3. 为什么 Agent 评测最难

因为 Agent 不是单步输出系统，而是多步任务推进系统。  
这意味着它可能在很多地方出问题：

- 目标理解错
- 规划错
- 工具选错
- 参数传错
- 状态更新错
- 失败恢复错

最终你看到的只是“任务没完成”，但真正的问题往往埋在中间轨迹里。

## 4. Agent 评测不该只看最终回复

如果你只看最终回复，很容易错过很多风险。

例如：

- 结果看起来对了，但中间调用了不该调的工具
- 任务完成了，但花了过多步骤，成本极高
- 成功率不低，但一旦出错就没有任何兜底能力

所以 Agent 评测至少要回答 3 个问题：

1. 结果是否完成目标
2. 过程是否合理可控
3. 出错时是否能安全结束或转交

## 5. Agent 评测的关键维度

### 5.1 任务完成率

这是最直观的指标。  
但要注意先定义清楚“完成”是什么意思。

例如：

- 查到信息并输出结论
- 成功创建一个日程
- 按要求完成多步资料整理

### 5.2 轨迹合理性

即使最终成功，也要看过程是否健康，例如：

- 是否走了明显多余的步骤
- 是否反复调用同一工具
- 是否遗漏关键确认动作

### 5.3 工具使用正确率

重点看：

- 选错工具的频率
- 参数错误率
- 工具结果是否被正确使用

### 5.4 状态稳定性

重点看：

- 中间结果有没有被正确继承
- 是否出现前后矛盾
- 是否出现“像失忆一样重复问同样问题”

### 5.5 失败恢复能力

重点看：

- 工具失败后是否会无限重试
- 是否会错误升级
- 是否能转向降级路径或人工接管

## 6. 轨迹质量为什么重要

Agent 的很多风险，并不会立刻反映在最终答案上。

例如：

- 某个问题虽然成功了，但调用了 8 次工具才成功
- 某个任务虽然完成了，但中间做了不必要的数据写入
- 某个样本虽然表面成功，但逻辑路径已经非常脆弱

如果不看轨迹，你会以为系统健康；  
但一旦流量变大、任务变复杂，问题就会放大。

所以对 Agent 来说，轨迹本身就是评测对象。

## 7. 人工兜底也属于评测范围

很多人把人工兜底看成“系统失败了才找人补”。  
但在真实业务里，更成熟的理解是：

**人工兜底本身就是系统稳定性设计的一部分。**

要评测的不只是“有没有人工兜底”，还包括：

- 什么情况下触发人工兜底
- 触发是否足够及时
- 移交信息是否完整
- 人工处理后是否能回流为新样本

如果一个 Agent 一出错就沉默、乱答或者无限循环，它即使平时成功率还行，也不算成熟系统。

## 8. 一套最小可行的 Agent 评测框架

如果你是自己做项目，可以从下面这套最小框架开始：

### 8.1 样本层

准备 20 到 30 条任务样本，覆盖：

- 简单任务
- 多工具任务
- 工具失败任务
- 目标模糊任务

### 8.2 判定层

为每条样本记录：

- 是否完成任务
- 是否调用了正确工具
- 是否在合理步数内完成
- 是否出现危险行为

### 8.3 轨迹层

每次运行保存：

- 规划内容
- 工具调用记录
- 中间状态变化
- 错误与重试记录

### 8.4 兜底层

记录：

- 是否触发人工兜底
- 触发原因
- 人工最终如何解决

这已经足够让你对一个 Agent 项目建立基本工程判断。

## 9. 总结与下一步建议

Agent 评测真正难的地方，不是指标名字多，而是它必须同时覆盖结果、过程和失败路径。

一句话总结：

**一个成熟的 Agent，不只是“经常成功”，还应该“失败得可控、可解释、可接管”。**
