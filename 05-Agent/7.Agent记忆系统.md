## 1. 为什么 Agent 需要记忆

没有记忆的 Agent 有三个致命问题：

### 1.1 对话断裂

用户第 3 轮说"我之前提到的那个项目"，Agent 完全不知道在说什么。每轮对话都是独立的，没有上下文连续性。

### 1.2 Context window 爆炸

最直觉的解法是把所有历史消息都塞进 context。但 20 轮对话后，光历史就占了 8000+ token。模型的注意力被稀释，回答质量反而下降。

### 1.3 无法积累

用户上周告诉 Agent "我喜欢 Python"，今天再问"推荐一门语言"，Agent 还是不知道用户的偏好。

**记忆系统的核心价值：让 Agent 在有限的 context window 内，保持最大的信息利用率。**

## 2. 三层记忆模型

借鉴认知科学的三层记忆模型：

| 层级 | 类比 | 实现 | 特点 |
|------|------|------|------|
| 短期记忆 | 工作台上的便签 | LLM 压缩摘要 | 当前会话，自动压缩 |
| 长期记忆 | 文件柜 | FAISS 向量索引 + JSON | 跨会话持久化，语义检索 |
| 工作记忆 | 黑板上的任务清单 | 结构化 Dict | 当前任务状态 |

### 为什么需要三层？

单层记忆不够用：
- 只有短期记忆 → 关掉就忘
- 只有长期记忆 → 检索噪音大，缺少当前对话的连贯性
- 只有工作记忆 → 无法记住用户偏好

三层协作才能同时满足"当前对话流畅"和"跨会话记住"。

## 3. 短期记忆压缩策略

### 3.1 问题：消息列表越来越长

每轮对话加 2 条消息（用户 + 助手）。10 轮就是 20 条，20 轮就是 40 条。

### 3.2 方案：LLM 摘要压缩

当消息数超过阈值（如 20 条），触发压缩：

```text
[msg_1, msg_2, ..., msg_14] → LLM 压缩 → 摘要文本（200字）
[msg_15, ..., msg_20] → 保留原文

最终 context = 摘要 + 最近 6 条消息
```

### 3.3 压缩 prompt 的设计要点

1. 保留关键信息：用户身份、偏好、讨论过的主题和结论
2. 去除闲聊和重复
3. 使用第三人称描述（"用户提到他在北京工作"）
4. 限制摘要长度（200 字以内）

### 3.4 与 LangGraph MemorySaver 的区别

`MemorySaver` 是全量保存——每条消息都存储。适合消息量不大的场景。

LLM 压缩是有损保存——用摘要替代旧消息。适合长对话场景，但可能丢失细节。

## 4. 长期记忆的存储与检索

### 4.1 存储结构

每条记忆 = `{content, category, timestamp, source_turn}`

category 分类：
- `personal_info`：姓名、职业、所在城市
- `preference`：喜好、偏好
- `fact`：讨论过的事实性信息
- `task`：任务相关信息

### 4.2 存储流程

```text
每轮对话结束 → LLM 提取值得记忆的信息 → 编码为向量 → 存入 FAISS + JSON
```

提取的关键：不是什么都记，而是 LLM 判断"这条信息值得长期保留吗"。

### 4.3 检索流程

```text
新一轮对话开始 → 用户输入 → BGE embedding 编码 → FAISS 语义检索 → top-5 相关记忆 → 注入 system prompt
```

用 BGE 的检索前缀确保 query 和 document 在同一语义空间。

### 4.4 与 regex 提取的区别

`agent-chat-langgraph` 项目用正则表达式提取 facts（如匹配"我叫xxx"、"我住在xxx"）。

LLM 提取 + FAISS 检索的优势：
- 不需要预定义 pattern
- 能理解隐含信息（"最近加班太多" → fact: 用户工作繁忙）
- 语义检索比精确匹配更灵活

劣势：
- 成本更高（每轮多 1 次 LLM 调用）
- 提取质量依赖 LLM 能力

## 5. 工作记忆的结构设计

### 5.1 什么是工作记忆

工作记忆是"当前任务的草稿纸"。它不是历史信息，而是当前推理过程中的关键状态。

### 5.2 结构

```python
{
    "current_goal": "帮用户优化 RAG 召回率",
    "key_facts": ["用户在做金融投研助手", "当前召回率约60%", "已尝试调整chunk size"],
    "pending_questions": ["用户使用什么 embedding 模型？"],
    "reasoning_steps": ["建议先检查 embedding 质量", "再考虑混合检索"]
}
```

### 5.3 注入方式

工作记忆以文本形式注入 system prompt。LLM 看到当前任务状态后，能更好地保持聚焦，不会"忘了在讨论什么"。

## 6. 记忆与 LangGraph 的集成

### 6.1 图拓扑

```text
START → load_memory → chat → extract_and_store → compress_if_needed → END
```

### 6.2 各节点的记忆职责

| 节点 | 记忆操作 |
|------|---------|
| `load_memory` | 从长期记忆检索相关条目，注入 state |
| `chat` | 使用三层记忆构建 system prompt，调用 LLM |
| `extract_and_store` | 提取新记忆存入长期记忆，记录到短期记忆 |
| `compress_if_needed` | 检查短期记忆是否需要压缩，更新工作记忆 |

### 6.3 闭包模式

记忆对象（ShortTermMemory、LongTermMemory、WorkingMemory）通过 `make_nodes` 闭包注入，不存储在 state 中。这样 state 保持轻量级，记忆对象可以有复杂的内部状态。

## 7. 记忆系统的评测

### 7.1 记忆准确率

```
记忆准确率 = Agent 正确回忆的信息数 / 用户告知的总信息数
```

测试方式：先告诉 Agent 多条信息，然后逐一询问"我之前说xxx是什么"。

### 7.2 记忆遗忘率

```
遗忘率 = 压缩后丢失的关键信息数 / 压缩前的关键信息数
```

测试方式：进行 30+ 轮对话触发多次压缩，检查早期信息是否还能召回。

### 7.3 检索相关性

用 RAG 评测的指标（Recall@K、MRR）评估长期记忆的语义检索质量。

## 8. 与人类记忆的类比

| 人类记忆 | Agent 记忆 | 相似点 |
|---------|-----------|--------|
| 感觉记忆 | 当前输入 | 极短暂，未经处理 |
| 短期/工作记忆 | 短期记忆 + 工作记忆 | 容量有限，需要主动维护 |
| 长期记忆 | 长期记忆 | 容量大，需要线索触发提取 |
| 遗忘曲线 | 压缩丢失 | 不重要的信息逐渐消失 |
| 记忆巩固 | 提取+存储 | 重要信息从短期转入长期 |

类比的价值：帮助理解设计决策。但不要过度类比——Agent 的记忆是工程系统，不需要也不应该完全模拟人类认知。

## 9. 常见误区

### 误区 1：什么都记

不加区分地记住所有信息，长期记忆膨胀，检索噪音增大。应该让 LLM 判断"值不值得记"。

### 误区 2：只用全量保存

把所有消息存在 context 里，不做压缩。20 轮后 context 爆满，回答质量下降。

### 误区 3：忽略检索质量

长期记忆存了很多，但检索时返回不相关的内容，反而误导 LLM。记忆系统的价值取决于检索质量。

### 误区 4：不考虑成本

三层记忆每轮可能增加 3 次 LLM 调用（提取 + 压缩 + 工作记忆更新）。对于简单聊天场景，可能不值得。
