## 1. 这篇文档讲什么

前面的讲义分别覆盖了 Agent 的架构、生产难点、选型、稳定性和评测。但有一个问题一直没有专门回答：

**当 Agent 的核心能力需要检索外部知识时，该怎么设计？**

传统 RAG 是一条固定管道，Agent 是一个会"思考和决策"的系统。两者结合，就产生了 Agentic RAG——让 Agent 自主控制检索过程。

这不是简单的"Agent + RAG"拼接，而是一种新的系统设计范式。

## 2. 为什么传统 RAG 不够

传统 RAG 的标准流程：

```text
用户问题 → 查询向量化 → 检索 top-k → （可选: 重排序） → 拼接上下文 → LLM 生成
```

这个管道的特点是**固定的**：无论什么问题，都走同样的路径，取同样的 top-k。

当问题变复杂时，问题就出现了：

### 2.1 问题1：一次检索不够

"分析公司的盈利能力变化趋势，并解释原因，同时对比研发投入是否在增长。"

这个问题涉及至少两个章节（盈利能力 + 研发投入）。一次检索可能只命中一个，导致回答不完整。

### 2.2 问题2：不是所有问题都需要检索

"你好" "谢谢" "请帮我总结一下刚才的内容"

这些问题根本不需要检索，但固定管道还是会去检索，浪费资源还可能引入噪声。

### 2.3 问题3：检索 query 需要改造

"去年赚了多少钱" → 用户意图可能是"营业收入"或"净利润"，直接用原文检索效果差。

"那个新能源板块怎么样" → 指代不明，需要结合上下文理解。

### 2.4 问题4：检索结果好坏无人判断

固定管道不检查检索结果是否有用。如果 top-k 全是不相关的内容，模型还是会基于这些内容生成答案——这就是 RAG 幻觉的常见来源。

### 核心矛盾

**固定管道假设"一次检索就够"且"检索结果一定有用"，但现实场景经常不满足这两个假设。**

## 3. 什么是 Agentic RAG

Agentic RAG = **Agent 自主控制 RAG 管道中的关键决策**。

具体来说，Agent 在以下决策点发挥作用：

| 决策点 | 传统 RAG | Agentic RAG |
|--------|---------|-------------|
| 是否需要检索 | 总是检索 | Agent 判断（有些问题不需要） |
| 用什么 query 检索 | 用户原文 / 简单改写 | Agent 基于分析制定检索策略 |
| 检索结果够不够 | 不判断 | Agent 评估，不够就补充检索 |
| 换什么角度补充 | 不支持 | Agent 分析缺什么，针对性改写 query |
| 什么时候停止检索 | 检索一次就停 | Agent 判断信息充分后才停 |

### RAG 演进脉络

```text
Naive RAG          → Advanced RAG           → Agentic RAG
固定切片+向量检索     混合检索+重排序+查询改写     Agent 自主控制检索决策
管道固定             管道增强但仍固定            管道动态，由 Agent 驱动
```

Agentic RAG 不是否定前两者，而是在它们基础上增加了**自适应决策能力**。

## 4. Agentic RAG 的几种典型模式

### 4.1 自适应检索（Adaptive Retrieval）

Agent 先判断是否需要检索：

```text
用户问题 → Agent 判断 ─┬─ 需要检索 → RAG 管道 → 生成
                       └─ 不需要 → 直接生成
```

适用场景：对话式应用中，并非每句话都涉及知识检索。

典型实现：Self-RAG，Agent 在生成过程中动态决定是否需要检索补充。

### 4.2 多轮检索（Iterative Retrieval）

Agent 检索后评估结果，不满意就换角度继续检索：

```text
问题 → 检索 → 评估 ─┬─ 够了 → 生成
                     └─ 不够 → 改写 query → 检索 → 评估 → ...
```

适用场景：复杂多维问题（如跨多个章节的金融分析）。

关键设计：必须有最大轮数限制，防止无限循环。

### 4.3 检索策略路由（Retrieval Routing）

Agent 根据问题类型选择不同的检索方式：

```text
问题 → Agent 路由 ─┬─ 精确查找 → BM25
                   ├─ 语义理解 → 向量检索
                   ├─ 结构化查询 → SQL/图查询
                   └─ 混合需求 → 混合检索
```

适用场景：多种数据源和检索方式并存的企业系统。

### 4.4 多源检索（Multi-source Retrieval）

Agent 决定从哪个知识库检索：

```text
问题 → Agent 路由 ─┬─ 年报数据 → 年报知识库
                   ├─ 研报观点 → 研报知识库
                   ├─ 实时数据 → Web 搜索
                   └─ 多个来源 → 并行检索 → 合并
```

适用场景：需要综合多个信息源的场景（如投研分析需要年报+研报+新闻）。

## 5. 一个典型的 Agentic RAG 链路

以多轮检索模式为例（也是项目 `agent-agentic-rag` 的实现方式）：

```text
START
  │
  ▼
analyze_query（LLM 分析问题，制定初始策略）
  │
  ▼
retrieve（混合检索 + 重排序）
  │
  ▼
evaluate_results（LLM 评估：信息够不够？）
  │
  ├─ 信息充分 ────────────────────── generate_answer → END
  │
  ├─ 信息不足，有改进方向 ─────────── refine_query → retrieve → ...
  │
  └─ 达到最大轮数 ─────────────────── generate_answer（用已有信息尽力回答）→ END
```

**关键节点的职责：**

- `analyze_query`：不是简单的 query 改写，而是分析问题涉及几个信息点、复杂度如何
- `evaluate_results`：这是 Agentic RAG 的核心决策点。Agent 需要判断"我手上的信息够不够回答这个问题"
- `refine_query`：不是随便换个说法，而是根据"已有什么、缺什么"有针对性地生成新 query

## 6. 关键设计决策

### 6.1 何时让 Agent 决定 vs 何时用固定规则

Agent 的每个决策都消耗一次 LLM 调用（成本+延迟）。不是所有决策都需要 LLM：

| 决策 | 建议方式 |
|------|---------|
| 是否需要检索 | Agent（需要理解意图） |
| 检索 top-k 取多少 | 固定规则（不值得 LLM 判断） |
| 检索结果是否相关 | Agent（需要理解内容） |
| 最多检索几轮 | 固定规则（硬性安全限制） |
| 用什么 query 检索 | Agent（需要分析缺失信息） |

**原则：用 LLM 做需要"理解"的决策，用规则做可以预定义的约束。**

### 6.2 如何控制检索轮数

最大轮数必须设上限（通常 2-4 轮）。原因：

1. 每多一轮，延迟增加（用户等不起）
2. 边际收益递减（第 3 轮检索很少能补充第 1-2 轮没找到的信息）
3. 上下文膨胀（累积太多上下文反而降低生成质量）

### 6.3 如何评估检索结果质量

这是 Agentic RAG 最难的部分。LLM 评估检索结果会犯两类错误：

- **假阳性**：认为"够了"但其实缺关键信息 → 回答不完整
- **假阴性**：认为"不够"但其实够了 → 浪费轮数和成本

优化方向：
1. 在 evaluate prompt 中明确列出需要回答的信息点，逐一检查
2. 要求 LLM 同时输出"覆盖了什么"和"缺少什么"，增加判断透明度
3. 兜底策略：达到最大轮数后，用已有信息尽力回答（而不是放弃）

### 6.4 成本与延迟权衡

一次 Agentic RAG 调用的 LLM 使用量 = analyze(1) + evaluate(N) + refine(N-1) + generate(1) = 2N+1 次。

如果最大 3 轮检索，就是 7 次 LLM 调用。对比固定管道 RAG 的 1 次 LLM 调用，成本和延迟增加 7 倍。

**不是所有问题都需要 Agentic RAG。** 对于简单的单维度问题，固定管道就够了。Agentic RAG 的价值在复杂、多维、需要综合分析的场景。

## 7. 与纯 Agent 和纯 RAG 的对比

| 维度 | 纯 RAG | Agentic RAG | 纯 Agent |
|------|--------|-------------|---------|
| 检索 | 总是检索，固定流程 | Agent 控制检索时机和策略 | 检索是众多工具之一 |
| 决策 | 无 | 检索相关决策 | 任意任务决策 |
| 工具 | 无 | 检索就是工具 | 多种工具 |
| 知识 | 外部知识库 | 外部知识库 | 可能不需要外部知识 |
| 成本 | 低（1次 LLM） | 中（多次 LLM） | 高（多次 LLM + 多工具） |
| 延迟 | 低 | 中 | 高 |
| 适用 | 简单问答 | 复杂知识问答 | 复杂任务执行 |

## 8. 企业场景中的实际考量

### 适合 Agentic RAG 的场景

- 问题复杂度差异大（有简单也有复杂，需要自适应）
- 知识库涵盖多个领域/维度（需要跨域检索）
- 对回答完整性要求高（宁可多检索也不能漏信息）
- 用户愿意等更长时间换取更准确的答案

### 不适合 Agentic RAG 的场景

- 所有问题都是同一类型、同一复杂度
- 对延迟敏感（如实时客服）
- 知识库小且集中（一次检索就能覆盖）
- 成本预算有限

### 实际建议

很多企业的最佳策略是**混合方案**：

1. 简单问题走固定管道 RAG（快且便宜）
2. 复杂问题路由到 Agentic RAG（准且完整）
3. 由一个轻量级路由器判断问题复杂度

## 9. 常见误区

### 误区1：Agent 不停检索但不知道何时停

表现：每轮评估都说"不够"，直到达到最大轮数。

原因：evaluate prompt 的标准太严格，或者缺乏"信息点逐项核对"的机制。

修复：在 analyze_query 阶段就列出信息点清单，evaluate 阶段逐项检查。

### 误区2：把 Agentic RAG 当成万能方案

如果基础 RAG（数据质量、切片策略、检索算法）都没做好，加 Agent 层只会放大问题——Agent 会在烂数据上做出错误决策。

**Agentic RAG 的前提是底层 RAG 基本可用。**

### 误区3：忽略基础 RAG 质量

Agent 的检索决策再聪明，如果向量模型选错了、切片粒度不对、重排序没做——检索结果质量差，Agent 也无能为力。

### 误区4：所有问题都走 Agentic RAG

简单问题走 Agentic RAG 是浪费。应该有一个前置的复杂度判断（可以很轻量），只有复杂问题才走 Agent 链路。

## 11. 总结与下一步建议

Agentic RAG 的核心价值：

1. **让 RAG 管道从"固定"变"自适应"**，Agent 根据问题动态调整检索策略
2. **但不是万能的**，前提是底层 RAG 基本可用，且问题确实需要自适应
3. **安全控制是必须的**，没有最大轮数限制的 Agentic RAG 迟早失控

下一步建议：

- 跑一遍 `07-项目实战/agent-agentic-rag/` 项目，观察 Agent 的完整推理轨迹
- 回到 `06-评测与优化/5.Agent评测.md`，思考如何评测 Agentic RAG 的检索决策质量
- 思考你自己的业务场景中，是否有适合 Agentic RAG 的应用点

一句话总结：

**Agentic RAG 的本质不是"更多轮检索"，而是"让 Agent 自主判断何时检索、检索什么、何时停止"。**
