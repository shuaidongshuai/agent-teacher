## 1. 什么是 Agent

很多人第一次接触 Agent，会把它理解成“更聪明的聊天机器人”。这个理解不完全错，但太浅。

一个更准确的定义是：

**Agent 是一个围绕目标持续进行感知、决策、调用工具、执行动作并根据结果继续推进任务的系统。**

这个定义里最重要的不是“会聊天”，而是下面几件事：

- 它有目标
- 它会决定下一步做什么
- 它可以使用外部工具
- 它会根据执行结果调整后续行为
- 它往往不是一步完成，而是多步推进

所以 Agent 的重点不是“回答一个问题”，而是“完成一个任务”。

比如：

- “解释一下什么是 Transformer”更像普通问答
- “帮我搜集竞品信息、整理表格、给出对比结论”更像 Agent 任务

## 2. 为什么 Agent 会变成热门方向

因为单次 Prompt 已经不够解决很多真实问题。

现实任务往往具有这些特点：

- 需要多步操作
- 需要读外部信息
- 需要调用工具
- 需要中间判断
- 需要根据新结果改变计划

这时，单轮对话就不够了。你需要一个能“边做边想”的系统。

Agent 热门的背后，实际上是大家想让 LLM 从“语言接口”升级成“任务执行接口”。

也就是说，人们不只想让模型回答：

- “这个问题怎么理解？”

更想让它完成：

- “请帮我把这件事做完”

## 3. Agent 和聊天机器人、工作流、RAG 的区别

理解边界，比背定义更重要。

### 3.1 Agent 和聊天机器人

聊天机器人通常以“响应用户输入”为中心，它的任务多半是解释、总结、改写、回答。

Agent 则以“推进任务完成”为中心，它不只生成语言，还要决定：

- 要不要查资料
- 要不要调工具
- 要不要拆分任务
- 下一步先做什么

一句话理解：

- 聊天机器人：偏回答
- Agent：偏执行

### 3.2 Agent 和工作流

工作流通常是人提前设计好的固定流程，比如：

1. 用户上传文件
2. 系统抽取文本
3. 调模型总结
4. 输出报告

这类流程的每一步、顺序、分支，通常是预先写死的。

Agent 的区别在于：

- 它不一定知道一开始的完整路径
- 它会在执行过程中动态决定下一步
- 它的决策空间更大

所以你可以这样理解：

- Workflow 更像“按剧本执行”
- Agent 更像“边看情况边行动”

### 3.3 Agent 和 RAG

RAG 解决的是“知识怎么拿到”的问题，Agent 解决的是“任务怎么推进”的问题。

RAG 可以是 Agent 的一个模块，但不等于 Agent。

例如：

- 一个知识库问答系统，可能只是 RAG，不一定是 Agent
- 一个研究助理 Agent，可能会把 RAG 当作其中一个能力来用

所以二者关系更像：

- RAG 是能力组件
- Agent 是任务执行系统

## 4. Agent 的核心闭环

你可以把一个成熟 Agent 看成一个循环系统，而不是一次性调用模型。

最核心的闭环通常包含 5 个部分：

1. 感知
2. 规划
3. 执行
4. 记忆
5. 反思

### 4.1 感知

感知是 Agent 获取当前世界状态的过程，来源可以包括：

- 用户输入
- 历史对话
- 工具返回结果
- 知识库资料
- 环境状态

如果感知阶段拿到的信息不完整，后面的规划和执行都会偏。

### 4.2 规划

规划是 Agent 判断“为了完成目标，接下来应该做什么”。

规划可能是：

- 轻量规划：决定下一步先调哪个工具
- 中等规划：把任务拆成几步
- 重度规划：先产出完整计划，再逐步执行

注意：

规划不是越复杂越好。计划太重会导致：

- 过度思考
- 浪费 token
- 遇到新信息时计划迅速失效

所以很多实用系统采用的是：

**局部规划 + 边执行边修正**

### 4.3 执行

执行是 Agent 真正对外部世界产生动作的部分，比如：

- 调搜索
- 调数据库
- 调日历
- 调代码执行器
- 调浏览器
- 生成报告

这一步的关键不在“能不能调用工具”，而在：

- 工具选得对不对
- 参数传得对不对
- 调用失败后怎么处理

### 4.4 记忆

没有记忆，Agent 很容易变成“每一步都像失忆”。

记忆通常可以分成：

- 短期记忆：当前任务上下文
- 长期记忆：用户偏好、历史经验、长期状态
- 工作记忆：中间过程、草稿、待办状态

记忆并不是存得越多越好。

真正重要的是：

- 存哪些
- 什么时候取
- 取出来后怎么用

### 4.5 反思

反思是 Agent 评估自己当前结果是否可靠，并决定是否修正的过程。

反思可能发生在：

- 执行失败后
- 结果明显不完整时
- 工具结果互相冲突时
- 发现自己偏离目标时

但要注意，反思也有成本。过多反思会造成：

- 延迟变高
- token 消耗增加
- 系统来回打转

所以工程上常做的不是“无限反思”，而是“在关键节点触发反思”。

## 5. Agent 的几种典型架构

### 5.1 ReAct 型

这是非常经典的一类 Agent 思路，可以粗略理解为：

- 先思考
- 再行动
- 再观察结果
- 再继续下一步

它的优点是直观，适合多步决策场景；缺点是轨迹长、成本高，而且一旦早期思路偏了，后面会一路带偏。

### 5.2 Planner-Executor 型

这种结构会把“规划”和“执行”相对分开。

常见形式是：

- 先由 Planner 生成任务计划
- 再由 Executor 按计划执行
- 必要时再回到 Planner 调整

它更适合中等复杂度任务，因为可以让系统少一点临场瞎试，多一点结构化推进。

### 5.3 Router 型

这类 Agent 的核心不是复杂推理，而是“把任务分发给合适能力”。

比如它需要判断：

- 这个问题应该走 RAG
- 还是走搜索
- 还是走数据库查询
- 还是走普通生成

Router 型系统在业务里很常见，因为它稳定性通常比“完全自由行动的 Agent”更高。

### 5.4 Multi-Agent 型

多 Agent 并不是“角色扮演越多越高级”，而是把不同职责拆开。

例如：

- 一个 Agent 负责规划
- 一个 Agent 负责检索
- 一个 Agent 负责写报告
- 一个 Agent 负责审查

它的好处是职责清晰，但坏处也很明显：

- 协调成本更高
- 延迟更高
- 错误传递链更长

所以很多情况下，一个设计良好的单 Agent 比一个花哨的多 Agent 更实用。

## 6. Agent 的关键能力模块

### 6.1 目标理解

Agent 的第一步不是行动，而是理解任务本身。

它至少要搞清楚：

- 目标是什么
- 成功标准是什么
- 有哪些限制
- 哪些输出是必须的

如果目标理解错了，后面动作再多也只是“努力跑偏”。

### 6.2 任务拆解

复杂任务通常不能一口气完成，需要拆成若干子任务。

好的拆解不是把任务拆得越细越厉害，而是让每一步都：

- 有清晰目标
- 能独立执行
- 有明确产出
- 能支持下一步

### 6.3 工具选择

Agent 不是见工具就调。它要学会判断：

- 什么时候需要工具
- 哪个工具最合适
- 是先检索再总结，还是先搜索再验证

这一步很像“行动策略选择”。

### 6.4 状态管理

一旦任务变成多步骤，状态管理就变得很关键。

Agent 需要知道：

- 当前做到哪一步
- 哪一步已完成
- 哪一步失败了
- 哪些中间结果可复用

很多看似“模型变笨”的问题，本质上是状态丢了。

### 6.5 停止条件

这点特别容易被忽略。

一个 Agent 系统不仅要知道怎么开始，还要知道什么时候该停。

停止条件常见包括：

- 已满足任务目标
- 达到最大步骤数
- 多次重试后仍失败
- 缺少必要信息

没有停止条件，Agent 很容易进入无意义循环。

## 7. 为什么 Agent 难做

Agent 难，不是因为“LLM 不够强”这么简单，而是因为它把多个脆弱环节串到了一起。

一个 Agent 系统，至少同时依赖：

- 任务理解
- 规划能力
- 工具调用
- 上下文管理
- 错误恢复
- 停止判断

这些环节每一个都可能出问题，而且会相互放大。

RAG 答错时，通常还是“答错一次”。  
Agent 答错时，可能是“错误地连续执行 5 步”。

所以 Agent 的挑战不是单点准确率，而是：

**长链路下的稳定性。**

## 8. Agent 常见失败模式

### 8.1 目标漂移

Agent 一开始理解了任务，但执行几步后慢慢偏离了原目标。

原因常见于：

- 中间结果把注意力带偏
- 没有持续对齐目标
- 局部最优覆盖了全局目标

### 8.2 工具滥用

有些 Agent 一遇到问题就疯狂调工具，导致：

- 成本高
- 速度慢
- 信息噪音增多
- 错误链路变长

这通常意味着它没有学会“什么时候不该调用工具”。

### 8.3 无效循环

比如：

- 查询失败
- 改写一下再查
- 还是失败
- 再改写再查

如果系统没有明确停止条件和失败升级策略，就会陷入循环。

### 8.4 伪规划

有些 Agent 看起来写出了很完整的计划，但计划并不真的指导执行。

表现为：

- 计划很漂亮
- 实际执行时完全不按计划
- 中间状态也不回写

这种系统表面像在“思考”，实际只是生成了好看的文字。

### 8.5 记忆污染

如果长期记忆和短期记忆混在一起，或者旧信息没清理，就可能让 Agent 被过期信息误导。

记忆不是越多越好，错误记忆会比没有记忆更危险。

### 8.6 工具结果盲信

Agent 很容易把工具返回结果当成绝对正确，但真实世界中的工具结果可能：

- 不完整
- 超时
- 空结果
- 字段错误
- 来源不可信

所以成熟 Agent 不只是“会调用工具”，还要“会质疑工具结果”。

## 9. Agent 的评测与观察指标

Agent 不能只看最终答案像不像对。

更好的评测方式是分层观察：

### 9.1 任务完成率

- 任务最终是否完成
- 是否满足用户约束
- 是否在合理步骤内完成

### 9.2 轨迹质量

- 每一步动作是否合理
- 是否使用了合适工具
- 是否有无效往返
- 是否偏离目标

### 9.3 工具调用质量

- 工具选择是否正确
- 参数是否正确
- 失败是否被处理

### 9.4 成本与延迟

- 调用了多少次模型
- 调用了多少次工具
- 总耗时是否可接受

### 9.5 可解释性

- 能不能回放执行轨迹
- 能不能解释为什么做出某个动作
- 出错时是否方便定位问题

Agent 的一个核心工程要求是：

**可观测。**

如果你看不到它每一步做了什么，就很难把它从“神秘系统”变成“可调系统”。

## 10. 什么时候适合用 Agent

Agent 适合的，不是“所有复杂问题”，而是具有下面特征的任务：

- 任务需要多步推进
- 路径不完全固定
- 需要动态决策
- 需要调用多个工具
- 中间结果会影响后续步骤

典型场景包括：

- 研究助理
- 销售线索筛选
- 多系统数据汇总
- 自动化运营分析
- 编码助手

这些任务的共同点是：  
不是一步回答，而是要持续推进。

---

## 11. 什么时候不适合用 Agent

这部分和“什么时候适合”同样重要。

如果任务本身满足下面条件，通常优先考虑普通工作流，而不是 Agent：

- 步骤固定
- 分支有限
- 成功标准明确
- 不需要太多动态决策

比如：

- 固定格式信息抽取
- 标准文档总结
- 分类打标
- 表单转结构化数据

这些任务往往用：

- Prompt
- 规则
- Workflow

就能更稳、更便宜、更可控。

一个非常重要的工程原则是：

**能用工作流解决，就先不要上自由度太高的 Agent。**

## 12. 一个最小可行 Agent 怎么搭

如果你想真的学会 Agent，不建议一开始就做多 Agent。

更好的路径是先搭一个最小单 Agent：

1. 接收任务目标
2. 判断是否需要工具
3. 选择一个工具执行
4. 读取结果
5. 判断下一步
6. 达到目标后停止

这个最小系统至少需要：

- 一个主控 Prompt
- 一个工具列表
- 一个状态对象
- 一个循环控制器
- 一个停止条件

建议第一版只支持：

- 1 到 2 个工具
- 最多 3 到 5 步
- 明确输出格式

你学 Agent 的关键，不在于一开始做得多复杂，而在于你能不能观察：

- 它为什么选了这个动作
- 它为什么会失败
- 它在哪一步开始偏

## 13. Agent 学习顺序建议

如果你想深入学，建议按下面顺序推进：

### 第一步：先学边界

先把下面几个概念区分清楚：

- 聊天机器人
- Workflow
- RAG
- Agent

这一步是为了避免你把所有 LLM 应用都统称为 Agent。

### 第二步：做一个单 Agent Demo

先只做：

- 一个目标
- 一个循环
- 两个工具
- 一个停止条件

这样你能学到 Agent 最核心的执行机制。

### 第三步：记录执行轨迹

不要只看最终结果，要记录：

- 每一步想做什么
- 为什么这样做
- 调了什么工具
- 返回了什么

这一步会让你真正进入工程视角。

### 第四步：开始分析失败

重点问自己：

- 是目标理解错了
- 还是规划错了
- 还是工具选错了
- 还是状态丢了
- 还是没有及时停止

### 第五步：再学多 Agent 和复杂编排

在单 Agent 不稳定之前，不建议直接上多 Agent。否则你会把原本已经难调的问题进一步放大。

## 14. 深入思考题与标准答案

### 思考题

1. 为什么说 Agent 的核心不是“会聊天”，而是“会推进任务”？
2. 为什么很多业务场景其实更适合 Workflow，而不是 Agent？
3. 为什么 Agent 的难点在长链路稳定性，而不是单次回答质量？
4. 为什么记忆对 Agent 既重要又危险？
5. 为什么多 Agent 不一定比单 Agent 更高级？

### 标准答案

#### 1. 为什么说 Agent 的核心不是“会聊天”，而是“会推进任务”？

标准答案：

因为聊天主要是在做语言响应，而 Agent 的本质是围绕目标持续决策和执行。一个 Agent 不只是生成一句回答，而是要判断下一步做什么、是否调用工具、如何根据结果继续推进，所以它关注的是任务闭环而不是单次表达质量。

#### 2. 为什么很多业务场景其实更适合 Workflow，而不是 Agent？

标准答案：

因为很多业务流程本身是固定的、可预定义的，没必要引入过高自由度。Workflow 的优势在于稳定、可控、便宜、好排查，而 Agent 更适合路径不固定、需要动态判断的任务。如果一个问题能被明确写成步骤，通常 Workflow 会比 Agent 更合适。

#### 3. 为什么 Agent 的难点在长链路稳定性，而不是单次回答质量？

标准答案：

因为 Agent 会连续执行多个动作，前一步的错误会传递到后一步，甚至被后续步骤进一步放大。单次回答只需要看这一轮是否合理，而 Agent 需要在多轮规划、工具调用、状态更新和停止判断中持续保持正确，这对系统稳定性要求高得多。

#### 4. 为什么记忆对 Agent 既重要又危险？

标准答案：

记忆重要，是因为没有记忆就无法持续跟踪任务状态、用户偏好和中间结果；记忆危险，是因为错误、过期或不相关的信息一旦被写入，就可能在后续步骤中不断误导 Agent。所以记忆系统的关键不是“多存”，而是“存对、取对、用对”。

#### 5. 为什么多 Agent 不一定比单 Agent 更高级？

标准答案：

因为多 Agent 只是职责拆分的一种方式，不代表一定更有效。它确实能让任务分工更清晰，但也会带来协调成本、通信开销、错误传递和调试难度。对于很多中小型任务，一个设计良好的单 Agent 往往更稳定、更便宜，也更容易落地。

---

## 15. 你学完后下一步该做什么

建议你接下来不要只停留在概念理解，而是做三件事：

1. 先搭一个最小单 Agent
2. 记录每一步轨迹
3. 拿失败案例来分析，而不是只展示成功案例

如果你真的想深入 Agent，这里有一个特别重要的转变：

**从“模型会不会回答”转向“系统会不会稳定完成任务”。**

当你开始用这个视角看问题时，你就已经在用工程师而不是体验者的方式理解 Agent 了。

---

## 16. 一句话总结

Agent 的本质，不是更会说话，而是能围绕目标在不确定环境中持续决策和执行。

真正难的，不是让它做出一个聪明动作，而是让它在多步链路里持续做出足够可靠的动作。
