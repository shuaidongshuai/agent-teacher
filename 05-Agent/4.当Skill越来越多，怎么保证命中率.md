---

## 1. 这篇文档要解决什么问题

很多人一开始会觉得：

**Skill 越多，系统能力就越强。**

但真实情况经常相反。

当 Skill 从几个变成几十个之后，系统开始出现这些问题：

- 该调用的时候没调用
- 明明有更合适的 Skill，却选了别的
- 多个 Skill 功能差不多，模型分不清
- 有些 Skill 长期高频命中，有些 Skill 几乎永远打不到

所以这篇文档真正要回答的是：

**当 Skill 越来越多时，怎样让系统更容易选对。**

---

## 2. 先理解一个核心前提：渐进加载

Anthropic 在 Agent Skills 里的一个核心思想是：

**渐进加载（Progressive Disclosure）**

意思不是一开始把所有 Skill 内容都塞进上下文，而是分层加载：

1. 启动时，模型先看到每个 Skill 的 `name` 和 `description`
2. 只有当模型判断某个 Skill 和当前任务相关时，才会继续加载这个 Skill 的完整内容
3. Skill 里的额外参考文件、脚本和资源，再按需继续读取

所以，Skill 命中率高不高，第一步往往不取决于 Skill 主体写得多详细，而取决于：

**模型能不能先从名称和描述里判断“这个 Skill 值得进一步打开”。**

这也是为什么：

- 描述写不清，Skill 很容易根本进不了候选
- Skill 太多时，平铺结构会越来越不稳

---

## 3. 为什么 Skill 一多，命中率反而会下降

原因其实很简单：

**候选越多，判断越难。**

尤其在下面几种情况下，命中率会明显下降：

### 3.1 描述都很像

例如很多 Skill 都写成：

- 处理文档
- 查询信息
- 生成内容

这种写法功能看起来没错，但触发场景太模糊，模型很难区分。

### 3.2 全部平铺

如果几十个 Skill 全部直接摆在一层，让模型一次性选择，误选率通常会升高。

### 3.3 没有边界

如果只写“这个 Skill 能做什么”，不写“这个 Skill 什么时候不要用”，模型就容易误触发。

### 3.4 Skill 太多，但还在用人工记忆式匹配

当 Skill 数量很大时，不能再假设模型只靠一眼扫 description 就能稳定命中全部能力。

---

## 4. 提高命中率的 4 个实用方法

### 4.1 提高描述区分度

不要只写“这个 Skill 是什么”，而要重点写：

**什么时候应该用它。**

例如：

- 不好：`负责生成 PPT`
- 更好：`当用户要求制作汇报材料、融资路演、季度总结或演讲文稿时使用`

核心思想是：

**description 里应该主要写触发场景，而不只是功能介绍。**

### 4.2 做分层路由

不要把所有 Skill 平铺在一层。

更稳的方式是先分大类，再分小类。

例如：

- 第一层：研发、运营、市场、财务
- 第二层：代码生成、代码评审、测试生成

这样做的本质是：

**先缩小候选范围，再做更细的选择。**

### 4.3 增加负样本描述

不仅要告诉模型什么时候该用，还要告诉模型什么时候不用。

例如：

- `仅用于生成 SQL，不适用于数据库设计和性能优化`
- `仅用于前端代码生成，不用于后端代码开发`

这类描述的价值很大，因为它在帮模型划边界。

很多误触发，不是因为模型不知道这个 Skill 能做什么，而是因为它不知道：

**这个 Skill 不该做什么。**

### 4.4 Skill 很多时，做召回 + 重排

当 Skill 数量很少时，模型直接选通常还可以。

但当 Skill 非常多时，更稳的做法通常是：

1. 先用 embedding、关键词或检索策略召回一批候选 Skill
2. 再让大模型在 Top K 里判断哪个最合适

例如：

- 先召回 Top 10
- 再让模型做最终选择

本质上就是把：

**Skill 选择问题，先当成一个检索问题，再当成一个判断问题。**

## 5. 常见误区

### 5.1 误区一：Skill 越多越好

Skill 数量增加，只代表能力覆盖可能更广，不代表命中率一定更高。

### 5.2 误区二：description 只写功能就够了

如果只写功能，不写触发场景，模型很难在第一步判断它该不该打开这个 Skill。

### 5.3 误区三：有正例就够了

只写“什么时候用”，不写“什么时候不用”，边界会越来越模糊。

### 5.4 误区四：Skill 多了还继续全量平铺

Skill 很少时还能勉强工作，Skill 一多就会明显恶化。

---

## 6. 练习题与思考方向

### 练习 1

把你手上的 3 个 Skill 描述改写一遍，不要只写功能，而要改成“触发场景”写法。

思考方向：

- 用户会怎么提需求
- 哪些关键词会触发这个 Skill

### 练习 2

找两个很容易混淆的 Skill，为它们分别补一条“不适用场景”描述。

思考方向：

- 两者最容易在哪类任务上误触发
- 哪句话最能把边界拉开

### 练习 3

如果你现在有 100 个 Skill，你会怎么分层？

参考方向：

- 先按业务域分
- 再按任务类型分
- 最后按具体动作分

---

## 7. 总结与下一步建议

Skill 多了以后，命中率问题本质上不是“模型记不住更多名字”，而是：

**模型能不能先快速缩小候选范围，再稳定选对。**

所以最实用的 4 个方法是：

1. 描述重点写触发场景
2. Skill 做分层路由
3. 增加“不适用场景”描述
4. Skill 很多时使用召回 + 重排
