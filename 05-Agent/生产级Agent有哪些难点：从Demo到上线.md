## 1. 这篇文档讲什么

很多 Agent demo 之所以看起来顺利，是因为它们默认了很多理想条件：

- 用户问题表达清楚
- 工具稳定可用
- 数据权限简单
- 任务规模不大
- 出错可以人工忽略

但上线之后，这些假设往往都不成立。

所以生产级 Agent 的难，不是“模型不会思考”，而是：

**任务复杂度、系统复杂度和业务约束同时出现时，怎样仍然让系统可控。**

---

## 2. 为什么 Demo 成功不等于上线成功

Demo 往往只验证了一件事：

“在一小批顺利样本上，这套链路能不能跑通。”

而上线要回答的是：

- 输入是否稳定多样
- 工具是否经常失败
- 权限是否复杂
- 结果是否可追踪
- 成本是否可接受
- 用户是否能容忍偶发错误

一句话说，demo 更像在看“能力上限”，上线更像在看“稳定下限”。

---

## 3. 生产级 Agent 的 6 类核心难点

### 3.1 目标理解不稳定

用户给出的目标常常不完整、不精确，甚至会不断变化。

常见表现：

- Agent 抓错重点
- 明明做了很多步骤，但不是用户真正想要的结果

### 3.2 工具调用链脆弱

真实系统里的工具会遇到：

- 超时
- 鉴权失败
- 限流
- 参数格式不匹配
- 返回结构变化

模型会调用工具，不代表它能稳定处理这些异常。

### 3.3 多步状态容易漂移

只要任务需要多步推进，就会遇到状态问题，例如：

- 上一步结果没有被正确继承
- 已完成和未完成任务混淆
- 中间草稿和最终结论混在一起

### 3.4 决策空间过大

Agent 的自由度越高，理论上越灵活，但实际也越容易：

- 走弯路
- 重复动作
- 工具乱试
- 长时间不收敛

### 3.5 失败后没有兜底

很多 demo 假设“出错就重试一下”，但线上需要更明确的失败策略：

- 什么时候自动重试
- 什么时候回退到简单路径
- 什么时候交给人工

### 3.6 很难定位问题

如果系统没有执行轨迹、关键日志和阶段指标，最后你只会看到：

- 用户觉得结果不对
- 但团队不知道到底是哪一层出了问题

---

## 4. 这些难点在真实业务里会怎么表现

在真实业务里，你通常会看到下面这些表象：

- “有时很好，有时很差”
- “同一个任务今天能跑，明天不能跑”
- “结果看起来努力了很多，但还是没完成”
- “偶尔会在某一步反复循环”
- “一出错就完全不知道该查哪里”

这些表象背后，往往不是一个单点问题，而是多层问题叠加。

## 6. 面对这些问题应当怎样拆解解决

可以先按下面 4 个层次排查：

### 6.1 先看任务边界

先确认：

- 这个任务是否真的适合 Agent
- 成功标准是否明确
- 是否可以拆成更稳的子流程

### 6.2 再看架构自由度

如果问题高度固定，优先考虑：

- 固定 Workflow
- Router
- 半结构化 Agent

不要一上来就给系统过大的自由度。

### 6.3 再看工具与状态

重点排查：

- 工具接口是否稳定
- 参数是否有 schema
- 状态是否可恢复
- 中间产物是否可追踪

### 6.4 最后看观测与兜底

如果没有以下能力，生产优化会很慢：

- 执行轨迹
- 错误分类
- 关键指标
- 人工接管机制

## 7. 练习题与思考方向

### 练习 1

为什么一个能稳定演示的 Agent，到了真实用户那里会显得“不稳定”？

参考方向：

- 输入分布变复杂
- 工具故障开始出现
- 用户目标更模糊
- 权限和异常处理更真实

### 练习 2

为什么 Agent 的失败常常是链式问题，而不是某一个点单独出错？

思考方向：

- 前一步错误会污染后续决策和状态

### 练习 3

如果团队说“这个 Agent 效果很玄学”，你会优先补什么能力？

参考方向：

- 执行日志
- 中间轨迹
- 失败分类
- 基础评测集

---

## 8. 总结与下一步建议

生产级 Agent 最大的难点，不是“它不会输出文字”，而是：

**它要在不确定输入、不稳定工具和复杂业务约束下，持续做出可控动作。**
