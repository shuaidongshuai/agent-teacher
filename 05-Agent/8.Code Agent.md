## 1. 什么是 Code Agent

Code Agent 是一种能够**自主操作代码文件**的 Agent。它不是被动地回答"这段代码有什么问题"，而是主动地：

1. 读取项目文件，理解代码结构
2. 运行测试，发现哪些失败
3. 分析失败原因，制定修复计划
4. 修改代码文件
5. 再次运行测试验证修复
6. 如果失败，重新规划修复

核心区别：**从"告诉你怎么改"变成"帮你改好"**。

## 2. 与 IDE Copilot 的区别

| 维度 | IDE Copilot | Code Agent |
|------|-------------|------------|
| 交互方式 | 补全/建议 | 自主执行 |
| 范围 | 单文件/单函数 | 跨文件项目级 |
| 能力 | 代码生成 | 读+写+执行+验证 |
| 反馈闭环 | 无（人工判断） | 有（自动运行测试） |
| 上下文 | 当前编辑器内容 | 整个项目 |

Copilot 是"辅助工具"，Code Agent 是"自主执行者"。两者互补，不是替代关系。

## 3. 代码工具设计原则

### 3.1 五种基本工具

| 工具 | 功能 | 输入 | 输出 |
|------|------|------|------|
| `read_file` | 读取文件 | path | content |
| `write_file` | 写入文件 | path, content | success |
| `list_dir` | 列出目录 | path | entries |
| `search_code` | 搜索代码 | pattern, path | matches |
| `run_command` | 执行命令 | cmd | output |

### 3.2 设计原则

**最小权限**：每个工具只做一件事，不要设计"读取并修改"这样的复合工具。

**统一返回格式**：所有工具返回 `{"success": "true/false", ...}`，方便 Agent 判断执行结果。

**路径安全**：所有文件操作必须在沙箱目录内，防止 Agent 误操作系统文件。

**输出截断**：命令输出限制长度（如 2000 字符），防止大输出撑爆 context。

## 4. 沙箱安全

Code Agent 有代码执行能力，安全是第一优先级。

### 4.1 路径白名单

只允许操作指定目录下的文件。路径验证时必须检查 `resolve()` 后的绝对路径，防止 `../../etc/passwd` 等路径穿越。

### 4.2 命令白名单

只允许执行预定义的安全命令（python, pytest, pip 等）。不允许 rm, curl, wget 等危险命令。

### 4.3 超时控制

每个命令有超时限制（如 30 秒），防止死循环或恶意代码。

### 4.4 输出截断

限制命令输出的最大字符数，防止巨量输出消耗 context window。

## 5. 修复循环：plan → execute → verify

### 5.1 完整流程

```text
analyze_task → plan_approach → execute_step(循环) → verify_result
                    ↑                                      │
                    └──────── [测试失败] ──────────────────┘
```

### 5.2 为什么需要循环

一次修复往往不够：
- 修了 Bug A 可能暴露 Bug B
- 修复方案本身可能引入新问题
- 第一轮诊断可能不准确

循环 + 测试验证 = 自动收敛。

### 5.3 循环控制

必须设上限（如最多 3 轮），否则 Agent 可能陷入"改了又改回去"的死循环。

## 6. 代码理解 vs 代码修改的 prompt 差异

### 6.1 代码理解的 prompt

要求 LLM 输出自然语言解释。重点是**准确性和结构化**（整体功能 → 核心逻辑 → 函数说明 → 潜在问题）。

### 6.2 代码修改的 prompt

要求 LLM 输出**工具调用指令**（JSON 格式）。重点是：
- 明确当前步骤和目标
- 提供已读取的代码上下文
- 提供已执行的操作历史
- 限制可用工具列表

两者的 prompt 结构完全不同，不能混用。

## 7. Code Agent 的评测

### 7.1 核心指标

**修复成功率**：测试全部通过的比例。
```
成功率 = 全部测试通过的任务数 / 总任务数
```

**修复效率**：平均需要几轮循环、多少次工具调用。

**代码质量**：修复后的代码是否引入新的代码质量问题（可以用 linter 检查）。

### 7.2 基准数据集

可以构建一组"有已知 bug 的代码 + 对应测试"，作为 Code Agent 的标准评测集。类似 SWE-bench 的思路，但规模更小。

## 8. 生产级 Code Agent 的挑战

### 8.1 大项目上下文

实际项目可能有数百个文件。Agent 不可能全部读取。需要智能的文件发现策略（搜索 → 相关文件 → 逐步深入）。

### 8.2 依赖管理

修复可能需要安装新包或修改依赖配置。这涉及更多安全风险。

### 8.3 版本控制

生产级 Code Agent 应该在独立分支操作，修改可以被 review 和回滚。

### 8.4 多语言支持

不同语言的测试工具、构建系统、包管理器都不同。通用 Code Agent 需要适配多种技术栈。

## 9. 常见误区

### 误区 1：不设沙箱

"反正是在本机跑的"——Agent 的 LLM 可能输出 `rm -rf /` 等危险命令。安全永远是第一位。

### 误区 2：一次修改整个文件

应该让 Agent 先定位问题行，再修改最小范围。"全量覆盖"容易引入新问题。

### 误区 3：不验证修复

"改了就算修好了"——必须运行测试验证。没有测试的修复是不可靠的。

### 误区 4：无限循环

不设最大轮数，Agent 反复修改-回退。必须有兜底：达到上限后用当前最优结果。

## 10. 练习题与思考方向

### 练习 1

运行 `agent-code-assistant/scripts/run_fix_bugs.py`，观察 Agent 修复 4 个 bug 的完整过程。记录每个 bug 的修复轮数和工具调用次数。

### 练习 2

在 `data/sample_project/calculator.py` 中添加一个新的 bug（如 multiply 的溢出处理），并在测试中添加对应用例。重新运行 Agent，观察它能否发现并修复新 bug。

### 练习 3

修改 `app/sandbox.py`，添加"命令审计日志"功能：记录所有执行的命令和结果到文件。思考：在生产环境中，这个日志有什么用？

### 练习 4

思考：如果要让 Code Agent 支持 JavaScript 项目（用 jest 做测试），需要修改哪些配置和 prompt？
